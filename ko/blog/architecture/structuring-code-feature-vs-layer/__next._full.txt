<h2 id="why-it-matters">왜 폴더 구조가 자전거 창고 논쟁이 아닌 아키텍처 결정인가</h2>
<p>사람들은 폴더 구조를 "단순한 정리"라고 일축합니다. 그것은 그것보다 훨씬 더 중요합니다. 코드를 그룹화하는 방식은 실제로 중요한 세 가지를 제어합니다:</p>
<ul>
  <li><strong>결합.</strong> 함께 변하는 코드가 멀리 떨어져 있으면, 모든 작은 업데이트가 폴더를 넘나드는 탐험이 됩니다. 함께 있으면, 관련 변경들이 한 곳에 모입니다.</li>
  <li><strong>발견 가능성.</strong> 코드베이스에 익숙하지 않은 개발자는 "청구(billing) 코드는 어디에 있나요?"라고 묻습니다. 그 질문의 답 — 한 폴더, 또는 다섯 개의 흩어진 폴더 — 이 온보딩 시간을 결정합니다.</li>
  <li><strong>영향 범위.</strong> 새 기능을 출시하거나 오래된 것을 안전하게 삭제하려면 얼마나 많은 파일을 열어야 하나요? 구조는 그 영향 범위를 제한하거나 무제한으로 확산시킵니다.</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">정신 모델</span><p>폴더 구조를 <strong>지도</strong>로 생각하세요. 좋은 지도는 실제 세계에서 관련된 방식으로 사물을 그룹화합니다 — 어떤 재료로 만들어졌는지가 아니라. 타입별로 파일을 그룹화하는 것(컨트롤러, 모델)은 지도 아이콘을 모양으로 정렬하는 것과 같습니다. 기능별로 그룹화하는 것은 이웃으로 정렬하는 것과 같습니다. 한 가지는 탐색에 도움이 되고; 다른 것은 기술적으로 맞지만 실용적으로는 쓸모없습니다.</p></div>
<p>Conway의 법칙 — 소프트웨어 시스템이 그것을 구축하는 팀의 소통 구조를 반영한다는 관찰 — 도 여기에 적용됩니다. 팀이 비즈니스 기능을 중심으로 구성되어 있다면(결제 스쿼드, 주문 스쿼드), 기능별 구조가 자연스럽게 느껴지고 올바른 경계를 강제합니다. 레이어별 구조는 여러분과 싸울 것입니다.</p>

<h2 id="by-layer">레이어별: 익숙한 기본값</h2>
<p>레이어별 구성은 기술적인 역할로 파일을 그룹화합니다. 일반적인 Node나 Spring Boot 프로젝트는 다음과 같이 됩니다:</p>

<figure class="blog-figure">
  <pre><code>src/
├─ controllers/
│  ├─ OrderController.ts
│  ├─ UserController.ts
│  └─ BillingController.ts
├─ services/
│  ├─ OrderService.ts
│  ├─ UserService.ts
│  └─ BillingService.ts
├─ repositories/
│  ├─ OrderRepository.ts
│  ├─ UserRepository.ts
│  └─ BillingRepository.ts
└─ models/
   ├─ Order.ts
   ├─ User.ts
   └─ Invoice.ts</code></pre>
  <figcaption>레이어별 구조: 모든 기술 레이어가 자체 최상위 폴더를 갖습니다. 기능을 추가하면 <em>네 개 모두</em>의 폴더에 파일을 추가해야 합니다.</figcaption>
</figure>

<p>모든 웹 프레임워크 튜토리얼이 여기서 시작합니다. Rails도, Spring Boot의 스타터도, Express 생성기도 그렇습니다. 그 친숙함이 주요 장점입니다.</p>
<p><strong>레이어별이 빛나는 곳:</strong></p>
<ul>
  <li>기능이 두세 개인 소규모 앱 — 구조가 너무 평평해서 거의 중요하지 않습니다.</li>
  <li>모든 사람이 모든 기능을 담당하는 팀; 레이어가 전문화의 단위입니다.</li>
  <li>초기 설정 속도가 장기적 명확성보다 중요한 프로토타입과 내부 도구.</li>
</ul>
<p><strong>레이어별이 아픈 곳:</strong></p>
<ul>
  <li><em>단일 기능이 폴더에 흩어집니다.</em> "환불" 기능을 추가하면 <code>controllers/</code>, <code>services/</code>, <code>repositories/</code>, <code>models/</code>를 수정해야 합니다. 개념적으로 하나인 것에 대해 네 개의 폴더입니다.</li>
  <li><em>기능 삭제가 고고학 프로젝트입니다.</em> <code>services/</code>의 어떤 파일이 청구와 주문에 속하는지 어떻게 알 수 있나요? 폴더 이름이 알려주지 않습니다.</li>
  <li><em>병합 충돌이 증가합니다.</em> 모든 기능 변경이 같은 레이어 수준 파일을 수정하여, 두 개발자가 병렬로 작업할 때 충돌을 일으킵니다.</li>
</ul>
<div class="blog-callout blog-callout--warn"><span class="blog-callout__label">확장성 절벽</span><p>레이어별은 10~15개 기능까지는 잘 작동합니다. 그 이후부터, 각 최상위 폴더가 <strong>잡동사니 서랍</strong>이 됩니다: 40개 파일짜리 <code>services/</code> 폴더, 명확한 소유권 없음, 그리고 검토자가 이전에 본 적 없는 파일을 수정하는 풀 리퀘스트.</p></div>

<h2 id="by-feature">기능별: 변경이 하나의 폴더에 삽니다</h2>
<p>기능별 구성은 서비스하는 제품 슬라이스별로 파일을 그룹화합니다. 기술 레이어(컨트롤러, 서비스, 모델)는 여전히 존재합니다 — 단지 최상위 수준이 아니라 각 기능 폴더 <em>안에</em> 있을 뿐입니다:</p>

<figure class="blog-figure">
  <pre><code>src/
├─ orders/
│  ├─ OrderController.ts
│  ├─ OrderService.ts
│  ├─ OrderRepository.ts
│  └─ Order.ts
├─ billing/
│  ├─ BillingController.ts
│  ├─ BillingService.ts
│  ├─ BillingRepository.ts
│  └─ Invoice.ts
├─ auth/
│  ├─ AuthController.ts
│  ├─ AuthService.ts
│  └─ User.ts
└─ shared/
   ├─ database.ts
   ├─ logger.ts
   └─ Money.ts</code></pre>
  <figcaption>기능별 구조: 각 제품 개념이 자체 폴더를 소유합니다. 환불 기능을 추가하면 폴더 하나가 추가됩니다 — 다른 것은 아무것도 이동하지 않습니다.</figcaption>
</figure>

<p><strong>기능별이 빛나는 곳:</strong></p>
<ul>
  <li><em>변경이 하나의 폴더에 삽니다.</em> 환불 흐름을 추가하면 <code>billing/</code> 안에 파일을 추가하는 것입니다 — 다른 폴더는 수정되지 않습니다. 영향 범위가 작고 국소적입니다.</li>
  <li><em>기능 삭제가 단순합니다.</em> 폴더를 제거하세요. 외부에서 아직 그것을 import한다면, IDE가 즉시 알려줄 것입니다.</li>
  <li><em>소유권이 명확합니다.</em> 결제 팀이 <code>billing/</code>을 소유하고; 신원 팀이 <code>auth/</code>를 소유합니다. 모노레포 도구와 코드 리뷰 규칙이 이것을 기계적으로 강제할 수 있습니다.</li>
  <li><em>온보딩이 더 빠릅니다.</em> 주문을 담당하는 신입 직원은 <code>orders/</code>만 읽습니다. 먼저 전체 코드베이스를 이해할 필요가 없습니다.</li>
</ul>
<p><strong>기능별이 주의가 필요한 곳:</strong></p>
<ul>
  <li><em>공유 코드 배치가 어색합니다.</em> <code>Money</code> 타입은 <code>orders/</code>와 <code>billing/</code> 모두에서 사용됩니다. <code>orders/</code> 안에 있어야 할까요? 그건 이상합니다. <code>shared/</code>에 들어가게 되는데 — 아무도 소유하지 않는다면 그것 자체가 두 번째 잡동사니 서랍이 될 수 있습니다.</li>
  <li><em>기능 간 의존성에 훈련이 필요합니다.</em> <code>orders/OrderService.ts</code>가 <code>billing/BillingService.ts</code>에서 직접 import한다면, 폴더 구조가 볼 수 없는 결합을 만든 것입니다. 팀들은 린팅 규칙(<code>eslint-plugin-import/no-restricted-paths</code>)이나 모듈 경계 도구(Nx, NestJS 모듈)를 사용하여 이것을 명시적으로 만듭니다.</li>
</ul>

<h2 id="screaming">소리치는 아키텍처: 최상위 수준이 무엇을 말하는가?</h2>
<p>Uncle Bob(Robert C. Martin)은 유명한 블로그 포스트에서 <strong>소리치는 아키텍처(Screaming Architecture)</strong>라는 용어를 만들었습니다: "당신의 애플리케이션 아키텍처는 무엇을 소리치는가? 최상위 디렉터리 구조와 가장 상위 패키지의 소스 파일을 볼 때; 그것들이 소리치는가: <em>의료 시스템</em>, 또는 <em>회계 시스템</em>, 또는 <em>재고 관리 시스템</em>? 아니면 소리치는가: <em>Rails</em>, 또는 <em>Spring/Hibernate</em>, 또는 <em>ASP.Net</em>?"</p>
<p>통찰이 우아합니다: 최상위 폴더가 <code>controllers/</code>, <code>models/</code>, <code>views/</code>라면, 코드베이스는 <em>MVC 프레임워크</em>를 소리치고 있습니다. 프레임워크는 도구입니다 — 독자가 처음 접하는 것이 되어서는 안 됩니다.</p>
<p>소리치는 아키텍처(또는 <strong>도메인별</strong>이라고도 불림)는 기능별에서 한 단계 더 나아갑니다. 기능이 최상위 수준에만 있는 것이 아니라 — 그 기능들의 이름이 기술적인 역할이 아닌 <em>비즈니스 언어</em>(도메인 주도 설계의 유비쿼터스 언어)에서 직접 나옵니다:</p>

<pre><code><span class="cmt">// "이커머스 플랫폼"을 소리칩니다</span>
src/
├─ catalog/          <span class="cmt">// 도메인 개념: 제품 카탈로그</span>
├─ ordering/         <span class="cmt">// 도메인 개념: 주문 배치 및 추적</span>
├─ fulfillment/      <span class="cmt">// 도메인 개념: 수령, 포장, 발송</span>
├─ payments/         <span class="cmt">// 도메인 개념: 청구, 환불, 인보이스</span>
└─ identity/         <span class="cmt">// 도메인 개념: 계정, 인증, 권한</span>

<span class="cmt">// 프레임워크를 소리치는 대안과 비교:</span>
src/
├─ controllers/      <span class="cmt">// "MVC"를 소리침</span>
├─ models/           <span class="cmt">// "ORM"을 소리침</span>
├─ views/            <span class="cmt">// "템플릿 엔진"을 소리침</span>
└─ services/         <span class="cmt">// "서비스 레이어 패턴"을 소리침</span></code></pre>

<p>각 도메인 폴더 내부의 레이아웃은 팀에게 달려 있습니다 — 그것은 괜찮습니다. 핵심 약속은 <em>최상위 수준</em>이 인프라가 아닌 비즈니스를 전달한다는 것입니다. 새 개발자(또는 CTO, 또는 비즈니스 쪽의 도메인 전문가)가 최상위 폴더를 읽고 어떻게 구축되었는지 모르면서도 시스템이 <em>무엇을 하는지</em> 이해할 수 있어야 합니다.</p>

  <li><em>기술 레이어는 여전히 각 폴더 내부에 존재합니다</em>, 따라서 MVC나 Clean Architecture 레이어로 생각하는 개발자들이 길을 잃지 않습니다. 기능 폴더 안을 보면 됩니다.</li>
  <li><em><code>shared/</code> 폴더</em>는 진정으로 가로로 잘라내는 관심사(<code>Money</code> 값 객체, 로거, 데이터베이스 연결)를 담습니다. 핵심 규칙: <code>shared/</code>에 무언가가 추가된다면, 의도적으로 추출되고 소유되어야 합니다 — 탈출구로 거기에 던져두는 것이 아닙니다.</li>
</ul>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">공유 폴더 이름 짓기</span><p>어떤 팀들은 <code>shared/</code>라고 부르고, 다른 팀들은 <code>common/</code>, <code>lib/</code>, 또는 <code>core/</code>라고 합니다. 이름은 훈련보다 덜 중요합니다: 미니 내부 라이브러리처럼 취급하세요. 파일이 특정 기능과 함께 계속 수정된다면, 그것은 shared에 속하지 않습니다 — 그 기능 안에 속합니다.</p></div>

<h2 id="at-scale">팀 규모에서: 강제된 경계와 모노레포</h2>
<p>코드베이스가 성장하고 여러 팀이 기여하면서, 폴더 규칙만으로는 충분하지 않습니다 — 개발자는 언제든 빠른 상대 import로 경계를 넘을 수 있습니다. 다음 단계는 경계를 <em>강제</em>하는 것입니다:</p>
<ul>
  <li><strong>NestJS 모듈</strong>은 모듈 간 의존성을 명시적으로 만듭니다: 다른 모듈이 <code>exports</code>하는 것에만 접근할 수 있습니다. 공개 API를 거치지 않고 모듈 경계를 넘는 import는 컴파일 오류입니다.</li>
  <li><strong>Nx 경계 규칙</strong>은 어떤 기능이나 라이브러리가 어떤 다른 것에 의존할 수 있는지 선언하고, CI에서 린트 오류로 강제할 수 있게 합니다.</li>
  <li><strong>모노레포</strong>는 이것을 가장 멀리 가져갑니다: 각 도메인(주문, 청구, 인증)은 자체 <code>package.json</code>, 자체 테스트, 명시적인 공개 export를 가진 자체 패키지가 됩니다. Turborepo와 Nx는 이를 관리하기 위한 JavaScript 생태계의 주요 도구입니다. 구조는 폴더 트리보다 패키지 그래프처럼 보입니다 — 하지만 기본 아이디어는 동일합니다: 비즈니스 기능으로 그룹화하고, 경계를 명시적으로 만들고, 기계적으로 강제합니다.</li>
</ul>
<p>모노레포로의 도약이 항상 필요한 것은 아닙니다. 많은 팀들이 단일 패키지와 잘 훈련된 하이브리드 구조로 수년간 행복하게 운영합니다. 팀 간 결합이 실제 고통이 될 때 모노레포 도구를 사용하세요 — 선제적으로 하지 마세요.</p>

<div class="blog-takeaways">
<h3 id="takeaways">핵심 요약</h3>
<ul>
  <li><strong>구조는 아키텍처입니다.</strong> 폴더 레이아웃은 결합, 발견 가능성, 그리고 모든 미래 변경의 영향 범위를 제어합니다.</li>
  <li><strong>레이어별</strong>은 친숙하고 소규모 앱에 적합하지만, 10~15개 기능을 넘으면 잡동사니 서랍이 됩니다.</li>
  <li><strong>기능별</strong>은 변경을 국소적으로 유지합니다. 새 기능은 하나의 폴더를 수정합니다. 기능을 삭제하면 하나의 폴더를 삭제합니다.</li>
  <li><strong>소리치는 아키텍처</strong>는 최상위 수준이 프레임워크가 아닌 <em>비즈니스</em>를 설명해야 한다고 합니다. 비기술적인 이해관계자가 폴더 이름을 읽고 시스템이 무엇을 하는지 이해할 수 있다면, 성공한 것입니다.</li>
  <li><strong>실용적인 하이브리드</strong>가 좋은 팀들이 도달하는 곳입니다: 최상위에 도메인/기능 이름, 각 폴더 내부에 작은 기술 레이어.</li>
  <li><strong>규모가 커질수록 경계를 강제하세요.</strong> 폴더 규칙은 제안입니다; 모듈 시스템, 린트 규칙, 모노레포 패키지가 그것을 보증으로 만듭니다.</li>
  <li><strong>일찍 재구성하세요</strong> — 15개 기능에서 오후 한 나절 짜리 폴더 리팩터가 50개에서의 수개월간의 혼란을 피합니다.</li>
</ul>
