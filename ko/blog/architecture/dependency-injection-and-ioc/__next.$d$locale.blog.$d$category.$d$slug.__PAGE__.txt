14:T846c,<p>어느 시점에서든 모든 개발자는 같은 벽에 부딪힙니다. 코드의 일부를 테스트하려는데, 그 코드가 내부에서 몰래 <code>new Database()</code>를 호출합니다. 무슨 짓을 해도 실제 데이터베이스 연결이 실행됩니다. 이메일 제공업체를 교체하고 싶은데, 이메일 SDK가 세 레이어 깊숙이 묻혀 있습니다. HTTP 요청과 예약 작업 모두에서 같은 로직을 실행하고 싶은데, 프레임워크가 너무 단단히 고정되어 있어 사실상 불가능합니다.</p>
<p>이것들은 하나의 근본 원인의 증상입니다: <strong>자신의 의존성을 직접 생성하는 코드</strong>. 그 해결책에는 공식적인 이름이 있습니다 — <em>Dependency Injection</em> — 하지만 그 밑에 있는 아이디어는 의외로 단순합니다. 코드가 자신의 협력자를 직접 찾지 않도록 하고, <em>외부에서 전달받는 것</em>입니다. 그것이 전부입니다. 나머지 모든 것 — IoC 컨테이너, DI 프레임워크, 주입 토큰 — 은 그 하나의 아이디어 위에 구축된 기계장치입니다.</p>
<p>이 가이드는 처음부터 풀어나갑니다: 문제, 원칙, 패턴, 그리고 테스트에서의 실질적인 보상. TypeScript는 웹이나 서버 배경을 가진 엔지니어들에게 명확하게 읽히기 때문에 전체적으로 사용합니다.</p>

<h2 id="the-problem">하드와이어드 의존성: 코드가 스스로 <code>new</code>를 호출할 때</h2>
<p>다음은 주문을 처리하는 서비스입니다. 처음 보면 완전히 합리적으로 보입니다:</p>
<pre><code><span class="cmt">// order-service.ts — "이전" 버전</span>
<span class="kw">class</span> OrderService {
  <span class="kw">async</span> placeOrder(cart: Cart): Promise&lt;Order&gt; {
    <span class="kw">const</span> db     = <span class="kw">new</span> PostgresClient(process.env.DATABASE_URL!)  <span class="cmt">// 하드와이어드 DB</span>
    <span class="kw">const</span> stripe = <span class="kw">new</span> Stripe(process.env.STRIPE_SECRET_KEY!)        <span class="cmt">// 하드와이어드 vendor</span>
    <span class="kw">const</span> mailer = <span class="kw">new</span> SendGridMailer(process.env.SENDGRID_KEY!)    <span class="cmt">// 하드와이어드 vendor</span>

    <span class="kw">const</span> order = Order.fromCart(cart)
    <span class="kw">await</span> stripe.charges.create({ amount: order.total, source: cart.token })
    <span class="kw">await</span> db.query(<span class="str">'INSERT INTO orders …'</span>, [order])
    <span class="kw">await</span> mailer.send({ to: cart.email, template: <span class="str">'order-confirmed'</span> })
    <span class="kw">return</span> order
  }
}</code></pre>
<p>이것으로 작업하려는 순간 세 가지 문제가 명확해집니다:</p>
<ul>
  <li><strong>격리된 상태로 테스트 불가능.</strong> 실제 자격증명, 실제 데이터베이스, 실제 Stripe 샌드박스 없이는 <code>placeOrder</code>를 유닛 테스트로 실행할 수 없습니다. 기본적으로 모든 테스트가 느리고 불안정한 통합 테스트입니다.</li>
  <li><strong>특정 구현에 단단히 결합되어 있음.</strong> 비즈니스가 Stripe에서 Adyen으로 이동하면, 이 파일을 열고 수술해야 합니다. 스테이징 환경에 더미 메일러가 필요하다면, 깔끔하게 연결할 방법이 없습니다.</li>
  <li><strong>숨겨진 의존성.</strong> 클래스 시그니처를 읽는 독자는 <code>placeOrder(cart)</code>를 보고 그 메서드가 세 개의 외부 서비스를 필요로 한다는 것을 전혀 알 수 없습니다. 모든 줄을 읽기 전까지 의존성이 보이지 않습니다.</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">주목해야 할 패턴</span><p>컴포지션 루트가 아닌 메서드나 생성자 본문 <em>안에서</em> <code>new SomeExternalThing()</code>을 볼 때마다 — 그것은 하드와이어드 의존성입니다. 클래스는 자신의 일을 하는 것과 그 일을 하는 데 필요한 도구를 찾고 구축하는 것, 두 가지 일을 담당하고 있습니다. 그 두 가지는 서로 반대 방향으로 당깁니다.</p></div>

<h2 id="inversion-of-control">Inversion of Control: 누가 생성을 담당하는가?</h2>
<p>코드로 문제를 해결하기 전에, 관련된 원칙에 이름을 붙이는 것이 도움이 됩니다. <strong>Inversion of Control</strong>(IoC)는 코드가 자신의 의존성을 얻는 책임을 지지 말아야 한다는 아이디어입니다. 대신, 외부의 무언가 — 프레임워크, 테스트 하네스, 컴포지션 루트 — 가 것들을 생성하고 전달하는 일을 담당합니다.</p>
<p>소위 <em>할리우드 원칙(Hollywood Principle)</em>을 들어보셨을 겁니다: <strong>"전화하지 마세요, 저희가 연락드리겠습니다."</strong> 전통적인 명령형 코드에서는 비즈니스 로직이 의존성을 직접 호출합니다 — 데이터베이스를 직접 가져옵니다. IoC에서는 의존성이 여러분에게 옵니다. 그 관계의 제어가 역전됩니다: 더 이상 협력자를 언제 어떻게 가져올지 결정하지 않습니다; 외부 세계가 결정하고 전달합니다.</p>
<p>IoC는 넓은 원칙이지, 특정 기술이 아닙니다. 프레임워크들은 다양한 방식으로 그것을 구현합니다:</p>
<ul>
  <li><strong>Dependency Injection (DI)</strong> — 생성자 인수나 메서드 매개변수로 협력자를 전달합니다. 가장 일반적이고 가장 명시적인 형태.</li>
  <li><strong>Service Locator</strong> — 코드가 런타임에 이름으로 의존성을 검색하는 전역 레지스트리. (나중에 왜 이것이 안티패턴인지 더 설명합니다.)</li>
  <li><strong>Template Method</strong> — 기본 클래스가 서브클래스가 오버라이드하는 훅 메서드를 호출하므로, 기반이 흐름을 제어하고 서브클래스가 내용을 채웁니다.</li>
  <li><strong>이벤트 / 옵저버</strong> — 컴포넌트가 이벤트를 발행하고; 프레임워크가 구독자에게 라우팅합니다. 양쪽 모두 상대방의 존재를 알지 못합니다.</li>
</ul>
<p>이 중에서 <strong>Dependency Injection이 가장 투명하고 가장 테스트하기 쉽습니다</strong>. 또한 IoC 자체와 가장 자주 혼동되는데, 그래서 두 용어를 분리할 가치가 있습니다: IoC는 <em>원칙</em>이고, DI는 그 원칙의 하나의 <em>구현</em>입니다.</p>

<h2 id="dependency-injection">Dependency Injection: 의존성을 전달하기</h2>
<p>DI의 메커니즘은 일단 보고 나면 당혹스러울 만큼 단순합니다. 클래스 내부에서 협력자를 구성하는 대신, 그것들을 매개변수로 선언하고 호출자가 제공하도록 합니다.</p>
<p>세 가지 방식이 있습니다 — 하지만 대부분의 상황에서는 하나가 명확한 승자입니다:</p>
<h3 id="constructor-injection">생성자 주입 (권장)</h3>
<p>의존성이 생성자에서 선언됩니다. 이것이 표준 선택입니다: 모든 의존성이 클래스 시그니처에 보이고, 구성 시점에 필요하며, <code>readonly</code>로 만들 수 있어 나중에 아무것도 재할당하지 않습니다.</p>
<pre><code><span class="cmt">// order-service.ts — 생성자 주입을 사용한 "이후" 버전</span>
<span class="kw">interface</span> PaymentGateway { charge(amount: Money, token: string): Promise&lt;Receipt&gt; }
<span class="kw">interface</span> OrderRepository { save(order: Order): Promise&lt;void&gt; }
<span class="kw">interface</span> Mailer          { send(to: string, template: string): Promise&lt;void&gt; }

<span class="kw">class</span> OrderService {
  <span class="kw">constructor</span>(
    <span class="kw">private readonly</span> payments: PaymentGateway,
    <span class="kw">private readonly</span> orders:   OrderRepository,
    <span class="kw">private readonly</span> mailer:   Mailer,
  ) {}

  <span class="kw">async</span> placeOrder(cart: Cart): Promise&lt;Order&gt; {
    <span class="kw">const</span> order = Order.fromCart(cart)                   <span class="cmt">// 순수한 비즈니스 규칙</span>
    <span class="kw">await</span> <span class="kw">this</span>.payments.charge(order.total, cart.token)
    <span class="kw">await</span> <span class="kw">this</span>.orders.save(order)
    <span class="kw">await</span> <span class="kw">this</span>.mailer.send(cart.email, <span class="str">'order-confirmed'</span>)
    <span class="kw">return</span> order
  }
}</code></pre>
<p>그 생성자 시그니처를 소리 내어 읽어보세요: "<code>OrderService</code>를 만들려면 결제 게이트웨이, 주문 레포지토리, 그리고 메일러가 필요합니다." 이것은 계약입니다. 단 하나의 메서드 본문도 읽지 않고 평범한 코드로 작성되어 보입니다.</p>
<h3 id="setter-injection">세터 / 메서드 주입 (가끔 사용)</h3>
<p>의존성이 생성 후 메서드를 통해 설정됩니다. 선택적 협력자에 유용하거나 두 단계로 객체를 구성하는 프레임워크(예: 일부 테스트 프레임워크, 일부 IoC 컨테이너)에 유용합니다. 드물게 사용하세요 — 객체가 부분적으로 구성된 상태로 존재할 수 있어 미묘한 버그의 원천이 됩니다.</p>
<pre><code><span class="cmt">// 세터 주입 — 구성 시점 배선이 불가능할 때만 사용</span>
<span class="kw">class</span> ReportGenerator {
  <span class="kw">private</span> formatter: ReportFormatter = <span class="kw">new</span> DefaultFormatter()

  setFormatter(f: ReportFormatter) { <span class="kw">this</span>.formatter = f }  <span class="cmt">// 선택적 오버라이드</span>

  generate(data: ReportData): string {
    <span class="kw">return this</span>.formatter.format(data)
  }
}</code></pre>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">단순한 경험 법칙</span><p>기본적으로 <strong>생성자 주입</strong>을 사용하세요. 클래스가 <em>항상</em> 필요로 하는 모든 의존성은 생성자에 속합니다. 의존성이 진정으로 선택적이거나 프레임워크가 강제할 때만 세터 주입을 사용하세요. 의존성 그래프에 대해 생각하지 않으려는 탈출구로 세터 주입을 절대 사용하지 마세요.</p></div>

<h2 id="composition-root">컴포지션 루트: 모든 것을 연결하는 한 곳</h2>
<p>생성자를 통해 의존성을 주입하면 새로운 질문이 생깁니다: <em>실제로 누가 <code>new</code>를 호출하고 조각들을 연결하는가?</em> 답은 <strong>컴포지션 루트(composition root)</strong>입니다: 애플리케이션의 진입점에 있는 단일 장소로 모든 실제 객체가 구성되고 서로에게 주입됩니다.</p>
<p>그것을 전체 코드베이스에서 <code>new ConcreteImplementation()</code>이 자유롭게 나타날 수 있는 유일한 장소로 생각하세요. 다른 모든 것은 인터페이스 측면에서 이야기합니다. 컴포지션 루트는 "추상 시스템"과 "실제 세계" 사이의 이음새입니다.</p>
<pre><code><span class="cmt">// composition-root.ts — 실제 것들이 연결되는 단 하나의 장소</span>
<span class="kw">import</span> { PostgresOrderRepository } <span class="kw">from</span> <span class="str">'./adapters/postgres-order-repository'</span>
<span class="kw">import</span> { StripePaymentGateway }    <span class="kw">from</span> <span class="str">'./adapters/stripe-payment-gateway'</span>
<span class="kw">import</span> { SendGridMailer }          <span class="kw">from</span> <span class="str">'./adapters/sendgrid-mailer'</span>
<span class="kw">import</span> { OrderService }            <span class="kw">from</span> <span class="str">'./domain/order-service'</span>
<span class="kw">import</span> { OrderController }         <span class="kw">from</span> <span class="str">'./http/order-controller'</span>

<span class="kw">export function</span> buildApp() {
  <span class="cmt">// 1. 먼저 리프 레벨 의존성 구성 (자신의 의존성 없음)</span>
  <span class="kw">const</span> db      = <span class="kw">new</span> PostgresOrderRepository(process.env.DATABASE_URL!)
  <span class="kw">const</span> stripe  = <span class="kw">new</span> StripePaymentGateway(process.env.STRIPE_SECRET_KEY!)
  <span class="kw">const</span> mailer  = <span class="kw">new</span> SendGridMailer(process.env.SENDGRID_KEY!)

  <span class="cmt">// 2. 서비스 구성, 의존성 주입</span>
  <span class="kw">const</span> orderSvc = <span class="kw">new</span> OrderService(stripe, db, mailer)

  <span class="cmt">// 3. 딜리버리 레이어 구성, 서비스 주입</span>
  <span class="kw">const</span> orderCtrl = <span class="kw">new</span> OrderController(orderSvc)

  <span class="kw">return</span> orderCtrl
}</code></pre>
<p>컴포지션 루트가 거의 레시피처럼 읽히는 것을 주목하세요: "이 재료들을 만들고, 서비스로 결합하고, 서비스를 컨트롤러에 넣어라." 전체 코드베이스에서 <em>모든 것이 어떻게 연결되는지</em> 이해하기 위해 가는 한 곳이 있습니다. 이것은 새 엔지니어를 온보딩할 때나 재료를 교체해야 할 때 엄청난 자산입니다.</p>
<p>테스트에서는 <em>테스트 특화 컴포지션 루트</em>를 작성합니다 — 실제 어댑터 대신 가짜를 연결하는 작은 동등물입니다:</p>
<pre><code><span class="cmt">// 테스트 파일에서 — 같은 서비스, 가짜 협력자들</span>
<span class="kw">const</span> fakePayments = <span class="kw">new</span> InMemoryPaymentGateway()
<span class="kw">const</span> fakeOrders   = <span class="kw">new</span> InMemoryOrderRepository()
<span class="kw">const</span> fakeMailer   = <span class="kw">new</span> NoOpMailer()

<span class="kw">const</span> svc = <span class="kw">new</span> OrderService(fakePayments, fakeOrders, fakeMailer)
<span class="cmt">// 밀리초 안에 실행; 네트워크도 데이터베이스도 없이</span></code></pre>

<h2 id="by-hand-vs-container">수동 DI vs DI 컨테이너 — 무엇이 달라지고 언제 중요한가</h2>
<p>위의 모든 것은 <em>수동 DI</em>, 또는 <em>Pure DI</em>라고도 합니다. 배선 코드를 직접 작성합니다. 많은 프로젝트, 특히 소규모의 경우 이것이 올바른 선택입니다: 평범한 코드이고, 마법이 없으며, 즉시 디버깅 가능하고, 새 개발자가 하나의 파일을 읽으면 전체 배선을 이해할 수 있습니다.</p>
<p>애플리케이션이 성장하면서, 수동 접근법이 불편해지기 시작합니다. DI 컨테이너(때로는 IoC 컨테이너라고도 불림)는 의존성의 등록과 해결을 자동화하고, 그 위에 수명 관리를 추가합니다.</p>

<h2 id="testing">테스트의 보상: 가짜를 주입하고, 빠른 테스트를 얻어라</h2>
<p>DI를 배울 가치가 있는 이유 — 어떤 아키텍처적 우아함보다 더 — 는 테스트 스위트에 미치는 영향입니다. 모든 의존성이 생성자를 통해 도착할 때, 테스트를 작성하는 것은 실제 어댑터 대신 가짜를 사용하는 작은 컴포지션 루트를 작성하는 것입니다. 데이터베이스도, 네트워크도, vendor 샌드박스도 없습니다. 테스트는 밀리초 안에 실행되며 스테이징 Stripe 계정에 요청 제한이 있어서 실패하는 일이 없습니다.</p>
<pre><code><span class="cmt">// order-service.test.ts</span>
<span class="kw">class</span> InMemoryPaymentGateway <span class="kw">implements</span> PaymentGateway {
  receipts: Receipt[] = []
  <span class="kw">async</span> charge(amount: Money, token: string): Promise&lt;Receipt&gt; {
    <span class="kw">const</span> r = { id: <span class="str">'fake-receipt'</span>, amount }
    <span class="kw">this</span>.receipts.push(r)
    <span class="kw">return</span> r
  }
}

<span class="kw">class</span> InMemoryOrderRepository <span class="kw">implements</span> OrderRepository {
  store: Order[] = []
  <span class="kw">async</span> save(order: Order) { <span class="kw">this</span>.store.push(order) }
}

<span class="kw">class</span> SpyMailer <span class="kw">implements</span> Mailer {
  sent: string[] = []
  <span class="kw">async</span> send(to: string) { <span class="kw">this</span>.sent.push(to) }
}

<span class="cmt">// 테스트 자체는 순수하고, 빠르며, 결정론적입니다:</span>
<span class="kw">it</span>(<span class="str">'주문을 확인하고 고객에게 알립니다'</span>, <span class="kw">async</span> () =&gt; {
  <span class="kw">const</span> payments = <span class="kw">new</span> InMemoryPaymentGateway()
  <span class="kw">const</span> repo     = <span class="kw">new</span> InMemoryOrderRepository()
  <span class="kw">const</span> mailer   = <span class="kw">new</span> SpyMailer()

  <span class="kw">const</span> svc   = <span class="kw">new</span> OrderService(payments, repo, mailer)
  <span class="kw">const</span> order = <span class="kw">await</span> svc.placeOrder(testCart)

  expect(payments.receipts).toHaveLength(1)
  expect(repo.store).toContainEqual(order)
  expect(mailer.sent).toContain(testCart.email)
})</code></pre>
<p>이것은 <a href="/blog/architecture/ports-and-adapters">Ports &amp; Adapters</a>의 핵심에 있는 같은 통찰입니다 — 인메모리 가짜는 테스트 시간 어댑터이고, 실제 Stripe나 Postgres 구현은 프로덕션 어댑터입니다. DI는 교체를 가능하게 하는 메커니즘이고; 포트 &amp; 어댑터는 경계를 의도적이고 명시적으로 만드는 아키텍처 패턴입니다.</p>
<p>여기서 SOLID에 대해 잠깐 멈출 가치도 있습니다. "D"는 <strong>Dependency Inversion Principle</strong>을 의미합니다: 고수준 모듈이 저수준 모듈에 의존해서는 안 됩니다; 둘 다 추상화에 의존해야 합니다. 위의 인터페이스들(<code>PaymentGateway</code>, <code>OrderRepository</code>, <code>Mailer</code>)이 그 추상화입니다. <code>OrderService</code>는 고수준 모듈입니다. <code>StripePaymentGateway</code>와 <code>PostgresOrderRepository</code>는 저수준 모듈입니다. 둘 중 어느 것도 상대방에 대해 직접 알지 못합니다 — 그들은 인터페이스에서 만나고, 컴포지션 루트가 그들을 함께 연결합니다.</p>

<h2 id="pitfalls">안티패턴: DI가 잘못되는 방법들</h2>
<p>아이디어는 단순하지만, 그것을 약화시키는 몇 가지 잘 알려진 방법들이 있습니다.</p>
<h3 id="service-locator">Service Locator</h3>
<p>Service Locator는 코드가 런타임에 이름으로 의존성을 검색하기 위해 호출하는 전역 레지스트리입니다:</p>
<pre><code><span class="cmt">// 안티패턴: Service Locator — DI처럼 보이지만 아닙니다</span>
<span class="kw">class</span> OrderService {
  <span class="kw">async</span> placeOrder(cart: Cart) {
    <span class="kw">const</span> payments = ServiceLocator.get&lt;PaymentGateway&gt;(<span class="str">'PaymentGateway'</span>)
    <span class="kw">const</span> repo     = ServiceLocator.get&lt;OrderRepository&gt;(<span class="str">'OrderRepository'</span>)
    <span class="cmt">// … 계속</span>
  }
}</code></pre>
<p>이것은 <code>new Stripe()</code> 인라인보다 깔끔해 보이지만, 같은 문제를 숨기고 있습니다. 의존성은 여전히 시그니처에서 보이지 않습니다. 테스트는 실행 전에 전역 레지스트리를 구성해야 합니다. 그리고 클래스는 어느 시점에서든 <em>무엇이든</em> 요청할 수 있습니다 — DI를 가치 있게 만드는 명시성을 잃었습니다.</p>
<h3 id="over-injection">과도한 주입 (10인수 생성자)</h3>
<p>생성자에 7, 8, 10개의 매개변수가 쌓일 때, 이는 보통 DI가 잘못된 도구라는 신호가 아니라 클래스가 너무 많은 일을 하고 있다는 신호입니다. 해결책은 클래스를 각각 짧은 의존성 목록을 가진 더 작고 집중된 협력자들로 분해하는 것입니다. 생성자 길이는 유용한 건강 지표입니다.</p>
<h3 id="hidden-globals">숨겨진 전역 컨테이너</h3>
<p>컴포지션 루트에서만이 아닌 서비스 내부에서 DI 컨테이너에 직접 접근하는 것 — 은 다른 이름의 Service Locator 패턴입니다. 규칙은 엄격합니다: 컴포지션 루트만 컨테이너와 이야기합니다. 서비스는 주입된 인터페이스하고만 이야기합니다.</p>
<h3 id="magic-you-cant-trace">추적할 수 없는 마법</h3>
<p>일부 프레임워크는 데코레이터와 리플렉트 메타데이터를 통해 의존성을 배선하는데, 이는 무언가 잘못되었을 때 진정으로 따라가기 어렵습니다. 컨테이너 문서를 읽는 데 자신의 비즈니스 로직을 읽는 것보다 더 많은 시간을 보내고 있다면, 이것은 더 단순한 접근법을 선택하거나 — 최소한 주요 바인딩을 명시적으로 설명하는 잘 문서화된 컴포지션 루트를 추가해야 한다는 신호입니다.</p>

<h2 id="by-company-size">DI 적정 규모: 솔로 프로젝트에서 엔터프라이즈까지</h2>
<p>대부분의 아키텍처 아이디어와 마찬가지로, DI는 문제에 맞게 확장됩니다. 목표는 항상 같습니다 — 테스트 가능하고 변경 가능한 코드 — 하지만 그것을 달성하기 위한 올바른 기계장치는 팀과 코드베이스가 성장함에 따라 변합니다.</p>

<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">실용적인 조언</span><p>수동 DI와 컴포지션 루트로 시작하세요. 배선 파일이 부담이 되면 언제든 컨테이너를 도입할 수 있으며, 마이그레이션은 기계적입니다: 명시적인 <code>new</code> 호출을 등록으로 교체하는 것입니다. 도메인, 서비스, 또는 테스트에서는 아무것도 바꿀 필요가 없습니다 — 컴포지션 루트만 바꾸면 됩니다.</p></div>

<div class="blog-takeaways">
<h3 id="takeaways">핵심 요약</h3>
<ul>
  <li><strong>경직되고 테스트 불가능한 코드의 근본 원인</strong>은 클래스가 자신의 의존성에 대해 <code>new</code>를 호출한다는 것입니다. 해결책은 그것들을 전달하는 것입니다.</li>
  <li><strong>Inversion of Control</strong>은 원칙입니다 (다른 누군가가 의존성을 제공하는 것을 담당합니다). <strong>Dependency Injection</strong>은 그것을 구현하는 가장 명시적이고 테스트하기 쉬운 방법입니다.</li>
  <li><strong>생성자 주입</strong>이 기본 선택입니다. 의존성이 시그니처에 보이고, 구성 시점에 필요하며, 테스트에서 쉽게 교체할 수 있습니다.</li>
  <li><strong>컴포지션 루트</strong>는 코드베이스에서 실제 객체들이 연결되는 단 하나의 장소입니다. 다른 모든 것은 인터페이스를 통해 이야기합니다.</li>
  <li><strong>수동 DI</strong>는 소규모 코드베이스에서 더 단순하고 디버깅하기 쉽습니다. <strong>DI 컨테이너</strong>는 컴포지션 루트 자체가 유지보수 부담이 될 때 가치를 발휘합니다.</li>
  <li><strong>테스트의 보상</strong>은 즉각적입니다: 테스트에 가짜를 주입하고, 네트워크와 데이터베이스를 건너뛰고, 수천 개의 케이스를 초 단위로 실행합니다.</li>
  <li>이것은 또한 실제 <strong>Dependency Inversion Principle</strong>(SOLID의 "D")입니다: 고수준 모듈과 저수준 모듈이 모두 추상화에 의존하며, 서로 직접 의존하지 않습니다.</li>
  <li><strong>Service Locator</strong>(숨겨진 전역 레지스트리), 과도한 주입(10인수 생성자), 컴포지션 루트 외부에서 컨테이너 접근을 피하세요.</li>
</ul>
