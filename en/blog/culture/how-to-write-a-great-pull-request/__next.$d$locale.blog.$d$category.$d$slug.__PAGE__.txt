e:T49ed,<p>You know the one. It arrives in your review queue on a Tuesday afternoon: a pull request titled <strong>"fixes"</strong>, diff spanning 2,347 lines across 41 files, no description, and a cheerful "PTAL" in the comment. Your heart sinks. You open it anyway. Twenty minutes later you've read the same method three times, still can't tell what problem it solves, and you've left a single nervous comment: "looks good overall, maybe rename this variable?" You've just rubber-stamped code you didn't understand.</p>
<p>The author wasn't malicious — they were probably heads-down, proud of the work, and just forgot that shipping code is only half the job. The other half is getting it reviewed, understood, and merged safely. A pull request isn't a formality you clear before merging; it's a conversation, and like all conversations, it goes better when someone puts in the effort to be understood.</p>
<p>This article is about that effort: how to write PRs that your teammates genuinely enjoy reviewing — small, purposeful, easy to follow, and easy to say yes to.</p>

<h2 id="pr-is-communication">A PR is communication, not just code</h2>
<p>Before a single line of diff, a pull request is a message to another human. That human has limited time, limited context about what you were thinking at 11 p.m., and a dozen other things competing for their attention. Your job as the author is to make their job easy.</p>
<p>Think of the reviewer as your audience, not your judge. They're not there to catch you out — they're a collaborator helping you ship something solid. The more you help them understand the <em>why</em> behind the change, the faster and better the review will be. A great PR description is like a cover letter for your code: it sets the scene, explains the motivation, and tells the reviewer exactly what to look at.</p>
<p>This mindset shift changes everything. Once you stop writing PRs for yourself and start writing them for your reviewer, you'll naturally want to make them smaller, describe them better, and clean them up before asking for eyes.</p>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">The reviewer's golden rule</span><p>If a reviewer has to read the code to understand <em>why</em> the change exists, the PR description hasn't done its job. The code explains <em>what</em> changed; the description explains <em>why it needed to change</em>.</p></div>

<h2 id="keep-it-small">Keep it small and focused</h2>
<p>The single most effective thing you can do to improve your pull requests is make them smaller. The larger a PR, the less thoroughly it gets reviewed, the longer it takes to merge, and the more bugs slip through.</p>
<p>A good PR contains <strong>one logical change</strong>. Not one ticket, not one feature, not one sprint's work — one coherent idea that can be understood, tested, and reverted independently. "Add user avatar upload" is one change. "Add user avatar upload and refactor the profile controller and bump three dependencies" is three changes wearing one coat.</p>

<h3 id="how-to-split">How to split big work</h3>
<p>Large features don't have to arrive as large PRs. Some practical splitting strategies:</p>
<ul>
  <li><strong>Separate refactoring from behaviour changes.</strong> First PR: rename, restructure, extract — no new logic. Second PR: the actual feature. Reviewers can verify the refactor has no side effects, then focus entirely on the new logic.</li>
  <li><strong>Ship the infrastructure first.</strong> Add the database column, the migration, the type — all with no UI. Then add the UI that uses it. Each PR is independently deployable.</li>
  <li><strong>Use feature flags.</strong> Merge incomplete work behind a flag that's off by default. The PR is small and safe; the feature isn't "done" until you flip the flag.</li>
  <li><strong>Stacked PRs.</strong> PR B depends on PR A. Some teams use stacked-PR tooling to review each layer independently. If your team doesn't use stacking, keep it simple: finish and merge A first, then open B.</li>
</ul>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">A useful heuristic</span><p>If you can't describe your PR in a single sentence without the word "and", it probably contains more than one logical change. Find the seam and split it.</p></div>

<h2 id="title-and-description">Writing a title and description worth reading</h2>
<p>The title is the headline. It should say <em>what</em> changed and why it matters, in one line. Avoid vague verbs like "update", "fix stuff", or "changes" and aim for specificity. Here are common PR title smells and how to fix them:</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>PR smell</th><th>What's wrong</th><th>Better version</th></tr>
  </thead>
  <tbody>
    <tr>
      <th>"fixes"</th>
      <td>No context whatsoever — reviewer must read every line to guess what's happening.</td>
      <td>"Fix race condition in cart checkout when concurrent sessions collide"</td>
    </tr>
    <tr>
      <th>"WIP: various changes"</th>
      <td>Signals the author isn't ready. Draft PRs exist for a reason.</td>
      <td>Open as a GitHub Draft until it's genuinely ready for review.</td>
    </tr>
    <tr>
      <th>"Update UserService"</th>
      <td>Describes where, not what or why. Every PR updates something.</td>
      <td>"Add email-change verification flow to UserService"</td>
    </tr>
    <tr>
      <th>"JIRA-1234"</th>
      <td>Forces the reviewer to open a second tab just to understand the motivation.</td>
      <td>"Limit API rate to 100 req/min per IP (JIRA-1234)" — ticket as context, not as a substitute.</td>
    </tr>
    <tr>
      <th>"Refactor + new feature + bump deps"</th>

<p>The <strong>description</strong> is where the real communication happens. A good description answers three questions every reviewer has in the first ten seconds:</p>
<ul>
  <li><strong>What</strong> changed and where?</li>
  <li><strong>Why</strong> was this change needed — the context, the bug, the user story?</li>
  <li><strong>How do I verify</strong> it works — steps to test, screenshots, affected edge cases?</li>
</ul>

<span class="kw">## What</span>

<span class="kw">## Why</span>

<h2 id="commit-hygiene">Commit hygiene</h2>
<p>How you organise your commits matters — not just to satisfy process, but because commits are the permanent record of how a change came to be. A year from now, someone (probably you) will run <code>git blame</code> on a line and follow the commit hash into the history. What they find should tell a story, not read like a stream of consciousness.</p>
<p>A good commit message follows the conventional format: a short imperative subject line (50 chars or fewer), an optional blank line, and an optional body explaining <em>why</em> — not what the diff already shows.</p>
<pre><code><span class="cmt"># The conventional format:</span>
<span class="kw">type(scope): short imperative subject</span>

<span class="str">Optional body: why this change was made, what problem it solves,
any non-obvious decisions and the reasoning behind them.</span>

<span class="cmt"># Real examples:</span>
<span class="kw">fix(cart):</span> add row-level lock to prevent oversell on concurrent checkout

<span class="kw">feat(auth):</span> add email-change verification flow

<span class="kw">refactor(profile):</span> extract avatar upload to dedicated service</code></pre>
<p>Beyond message format, think about commit <em>shape</em>. Each commit should represent one coherent step — a refactor, a test, a new behaviour. A reviewer who looks at individual commits (rather than the whole diff) should be able to follow the logical progression: "extract method → add test → implement behaviour" tells a better story than twenty "WIP" commits squashed at the last second.</p>
<p>Use interactive rebase to clean up a messy branch before opening the PR. You don't need perfect commits from the start — polish them before you ask for eyes.</p>

<h2 id="self-review">Self-review before requesting review</h2>
<p>Before you ping a teammate, become your own first reviewer. Open the diff on GitHub (or your code host of choice) and read it as if you'd never seen the code. This context switch catches more issues than you'd expect: leftover debug output, a function that grew too large, an import you forgot to remove, a comment that used to be true.</p>
<p>A self-review checklist worth running through:</p>
<ul>
  <li><strong>Read the diff top to bottom.</strong> Don't skim. Pretend you're seeing it for the first time.</li>
  <li><strong>Check for debug leftovers.</strong> <code>console.log</code>, TODO comments marked "remove", commented-out blocks, temporary hardcoded values.</li>
  <li><strong>Verify tests are there.</strong> If you added logic, is there a test? If you fixed a bug, is there a regression test?</li>
  <li><strong>Look for unintended changes.</strong> Whitespace churn, reformatted files, accidental scope creep. Move them to a separate PR or revert them.</li>
  <li><strong>Leave guiding comments yourself.</strong> If a section is non-obvious, add a PR comment explaining the reasoning before your reviewer sees it. This is not a sign of weakness — it's consideration.</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">A simple rule</span><p>If you wouldn't feel comfortable with a senior engineer reviewing the PR cold right now — without your explanation — it's not ready to go out. Fix it first, then request review.</p></div>
<p>Self-review also pays a selfish dividend: it catches embarrassing mistakes before they become permanent comments in the review history. Everyone has a story about the typo they spotted ten seconds after hitting "Request review". Ten minutes of self-review eliminates most of those.</p>

<h2 id="responding">Responding to review</h2>
<p>A PR isn't done when you push the code — it's done when it merges. Between those two events is the review cycle, and how you handle feedback matters as much as the code itself.</p>
<p>Some principles that keep reviews moving and relationships intact:</p>
<ul>
  <li><strong>Reply to every comment.</strong> Even if only to say "done" or "good catch, fixed in latest commit". Silence leaves reviewers wondering whether you saw their note.</li>
  <li><strong>Describe your fix when you push it.</strong> "Fixed in latest commit — moved the logic into a helper and added a test" is far clearer than a new commit with no context.</li>
  <li><strong>Resolve threads thoughtfully.</strong> In most teams, the commenter resolves when they're satisfied. Don't self-resolve without acknowledging feedback. If you're uncertain about the convention, ask.</li>
  <li><strong>Disagree respectfully.</strong> If you think a suggestion is wrong, say so clearly and with reasoning — don't just ignore it. "I considered that approach, but [reason] — happy to discuss if you feel strongly" keeps the conversation professional and productive. See also: <a href="/blog/culture/how-to-review-code-kindly">how to review code kindly</a> for the reviewer's perspective on the same conversation.</li>
  <li><strong>Re-request review explicitly.</strong> After addressing feedback, don't wait for reviewers to notice. Re-request review and leave a short summary: "Addressed all comments. Main changes: extracted the util, added two tests, kept the original approach for X because Y."</li>
</ul>

<h2 id="by-team-size">How PR norms scale by team size</h2>
<p>What makes a great PR isn't fixed — it shifts as the team grows and the codebase matures. Here's how the expectations evolve:</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>Team context</th><th>What matters most</th><th>Practical norms</th></tr>
  </thead>
  <tbody>
    <tr>
      <th>Solo / freelance</th>
      <td>Future-you is the reviewer. PRs are documentation of decisions for when you return to the code in six months.</td>
      <td>Even with no reviewer: a clear title, a brief "why" in the description. Commit messages that make <code>git log</code> readable later.</td>
    </tr>
    <tr>
      <th>Small team (2–8)</th>
      <td>Speed matters. Everyone knows the codebase. Reviews are conversational. Trust is high.</td>
      <td>Keep PRs small and fast-moving. Descriptions can be lighter — a sentence or two of context. Async review with same-day turnaround is a reasonable expectation.</td>
    </tr>
    <tr>
      <th>Mid-size team (10–50)</th>
      <td>Reviewers may not know the full context. Onboarding is constant. Changes span teams.</td>
      <td>Full what/why/how descriptions. Screenshots for UI work. A test plan. Cross-team changes need extra care and possibly wider review.</td>
    </tr>
    <tr>
      <th>Large / enterprise (50+)</th>
      <td>Reviewers are sometimes strangers. Compliance, audit trails, and correctness reviews matter. Turnaround SLAs exist.</td>
      <td>Formal PR templates enforced by tooling. Required approvals. Security review for sensitive changes. Links to design docs and tickets. Rollback plan for risky changes.</td>
    </tr>
  </tbody>
</table>
</div>

<p>A startup team of four doesn't need a rollback plan in every PR description. A bank does. Match the ceremony to the stakes, and revisit the norms as the team grows — what worked at ten engineers will feel inadequate at fifty.</p>

<div class="blog-takeaways">
<h3 id="takeaways">Key takeaways</h3>
<ul>
  <li><strong>A PR is a message to a human.</strong> Write it for your reviewer, not for yourself. Their time and context are limited.</li>
  <li><strong>Keep it small and focused.</strong> One logical change per PR. Separate refactoring from behaviour changes. Use feature flags to merge incomplete work safely behind a toggle.</li>
  <li><strong>Title + description = orientation.</strong> Answer what changed, why it needed to change, and how to verify it. The code shows the what; the description provides the why.</li>
  <li><strong>Clean commit history tells a story.</strong> Conventional messages, logical commits, a tidy rebase before you request review.</li>
  <li><strong>Self-review first.</strong> Read your own diff on GitHub before anyone else does. Leave guiding comments. Catch the obvious mistakes yourself.</li>
  <li><strong>Close the loop on feedback.</strong> Reply to every comment, re-request review after addressing notes, and disagree with reasoning — never with silence.</li>
  <li><strong>Match ceremony to team size.</strong> A four-person startup and a 500-person bank need different PR norms. Revisit yours as you grow.</li>
</ul>
