24:T6801,<p>You open a new project and the first thing you do — before writing a single line of logic — is create a folder. Maybe it's called <code>controllers/</code>. Maybe it's called <code>orders/</code>. Either way, that quiet choice is already an architectural decision, and it will quietly shape everything that follows: how fast a new teammate finds their footing, how many files you touch when a single feature changes, and whether six months from now your codebase reads like your <em>business</em> or like your <em>framework</em>.</p>
<p>This post is a practical tour of the three main approaches — <strong>by-layer</strong>, <strong>by-feature</strong>, and <strong>screaming (by-domain)</strong> — with honest trade-offs at each company size. No single answer wins for every situation, but you'll leave with a clear mental model for choosing deliberately instead of by default.</p>

<h2 id="why-it-matters">Why folder structure is an architectural decision, not bikeshedding</h2>
<p>People dismiss folder structure as "just organization". It's much more than that. The way you group code controls three things that turn out to matter enormously:</p>
<ul>
  <li><strong>Coupling.</strong> When code that changes together lives far apart, every small update turns into a cross-folder expedition. When it lives together, related changes cluster in one place.</li>
  <li><strong>Discoverability.</strong> A developer unfamiliar with your codebase will ask, "where is the code for billing?" The answer to that question — one folder, or five scattered folders — determines onboarding time.</li>
  <li><strong>Blast radius.</strong> How many files must you open to ship a new feature, or to safely delete an old one? Structure either constrains that blast radius or lets it sprawl unchecked.</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">The mental model</span><p>Think of your folder structure as a <strong>map</strong>. A good map groups things by how they relate in the real world — not by what material they're made of. Grouping files by type (controllers, models) is like sorting map icons by shape. Grouping by feature is like sorting by neighborhood. One of these helps you navigate; the other is technically correct but practically useless.</p></div>
<p>Conway's Law — the observation that software systems mirror the communication structures of the teams that build them — applies here too. If your team is organized around business capabilities (a payments squad, an orders squad), a by-feature structure will feel natural and enforce the right boundaries. A by-layer structure will fight you.</p>

<h2 id="by-layer">By layer: the familiar default</h2>
<p>By-layer organization groups files by their technical role. A typical Node or Spring Boot project ends up looking something like this:</p>

<figure class="blog-figure">
  <pre><code>src/
├─ controllers/
│  ├─ OrderController.ts
│  ├─ UserController.ts
│  └─ BillingController.ts
├─ services/
│  ├─ OrderService.ts
│  ├─ UserService.ts
│  └─ BillingService.ts
├─ repositories/
│  ├─ OrderRepository.ts
│  ├─ UserRepository.ts
│  └─ BillingRepository.ts
└─ models/
   ├─ Order.ts
   ├─ User.ts
   └─ Invoice.ts</code></pre>
  <figcaption>By-layer structure: every technical layer gets its own top-level folder. Adding a feature means adding files to <em>all four</em> folders.</figcaption>
</figure>

<p>Every web framework tutorial lands here. Rails does it, Spring Boot's starter does it, Express generators do it. That familiarity is its main asset.</p>
<p><strong>Where by-layer shines:</strong></p>
<ul>
  <li>Small apps with two or three features — the structure is so flat it barely matters.</li>
  <li>Teams where everyone works across all features; the layer is the unit of specialization.</li>
  <li>Prototypes and internal tools where speed of initial setup beats long-term clarity.</li>
</ul>
<p><strong>Where by-layer hurts:</strong></p>
<ul>
  <li><em>A single feature is scattered across folders.</em> Adding a "refund" feature means touching <code>controllers/</code>, <code>services/</code>, <code>repositories/</code>, and <code>models/</code>. That's four folders for what is conceptually one thing.</li>
  <li><em>Deleting a feature is an archaeology project.</em> How do you know which files in <code>services/</code> belong to billing versus orders? The folder name doesn't tell you.</li>
  <li><em>Merge conflicts multiply.</em> Every feature change touches the same layer-level files, causing collisions when two developers work in parallel.</li>
</ul>
<div class="blog-callout blog-callout--warn"><span class="blog-callout__label">The scaling cliff</span><p>By-layer works fine up to maybe 10–15 features. Beyond that, each top-level folder becomes a <strong>junk drawer</strong>: 40-file <code>services/</code> folders, no clear ownership, and pull requests that touch files a reviewer has never seen before.</p></div>

<h2 id="by-feature">By feature: a change lives in one folder</h2>
<p>By-feature organization groups files by the slice of the product they serve. The technical layers (controller, service, model) still exist — they just live <em>inside</em> each feature folder instead of at the top level:</p>

<figure class="blog-figure">
  <pre><code>src/
├─ orders/
│  ├─ OrderController.ts
│  ├─ OrderService.ts
│  ├─ OrderRepository.ts
│  └─ Order.ts
├─ billing/
│  ├─ BillingController.ts
│  ├─ BillingService.ts
│  ├─ BillingRepository.ts
│  └─ Invoice.ts
├─ auth/
│  ├─ AuthController.ts
│  ├─ AuthService.ts
│  └─ User.ts
└─ shared/
   ├─ database.ts
   ├─ logger.ts
   └─ Money.ts</code></pre>
  <figcaption>By-feature structure: each product concept owns its own folder. Adding a refund feature adds one folder — nothing else moves.</figcaption>
</figure>

<p><strong>Where by-feature shines:</strong></p>
<ul>
  <li><em>A change lives in one folder.</em> Adding a refund flow means adding files inside <code>billing/</code> — no other folder is touched. The blast radius is small and contained.</li>
  <li><em>Deleting a feature is simple.</em> Remove the folder. If something outside still imports from it, your IDE will tell you immediately.</li>
  <li><em>Ownership is obvious.</em> The payments team owns <code>billing/</code>; the identity team owns <code>auth/</code>. Monorepo tooling and code review rules can enforce this mechanically.</li>
  <li><em>Onboarding is faster.</em> A new hire working on orders reads only <code>orders/</code>. They don't need to understand the whole codebase first.</li>
</ul>
<p><strong>Where by-feature needs care:</strong></p>
<ul>
  <li><em>Shared code placement is awkward.</em> The <code>Money</code> type is used by both <code>orders/</code> and <code>billing/</code>. Does it live in <code>orders/</code>? That feels wrong. It ends up in <code>shared/</code> — which can itself grow into a second junk drawer if nobody owns it.</li>
  <li><em>Cross-feature dependencies need discipline.</em> If <code>orders/OrderService.ts</code> imports directly from <code>billing/BillingService.ts</code>, you've created a coupling that the folder structure can't see. Teams use linting rules (<code>eslint-plugin-import/no-restricted-paths</code>) or module-boundary tooling (Nx, NestJS modules) to make these explicit.</li>
</ul>

<h2 id="screaming">Screaming Architecture: what does your top level say?</h2>
<p>Uncle Bob (Robert C. Martin) coined the term <strong>Screaming Architecture</strong> in a famous blog post: "What does the architecture of your application scream? When you look at the top-level directory structure, and the source files in the highest-level package; do they scream: <em>Health Care System</em>, or <em>Accounting System</em>, or <em>Inventory Management System</em>? Or do they scream: <em>Rails</em>, or <em>Spring/Hibernate</em>, or <em>ASP.Net</em>?"</p>
<p>The insight is elegant: if your top-level folders are <code>controllers/</code>, <code>models/</code>, <code>views/</code>, your codebase is screaming <em>MVC framework</em>. The framework is a tool — it shouldn't be the first thing a reader encounters.</p>
<p>Screaming Architecture (also called <strong>by-domain</strong>) takes by-feature one step further. Not only are features at the top level — the names of those features are drawn directly from the <em>business language</em> (the Ubiquitous Language of Domain-Driven Design), not from technical roles:</p>

<pre><code><span class="cmt">// Screams "e-commerce platform"</span>
src/
├─ catalog/          <span class="cmt">// the domain concept: a product catalog</span>
├─ ordering/         <span class="cmt">// the domain concept: placing and tracking orders</span>
├─ fulfillment/      <span class="cmt">// the domain concept: picking, packing, shipping</span>
├─ payments/         <span class="cmt">// the domain concept: charges, refunds, invoices</span>
└─ identity/         <span class="cmt">// the domain concept: accounts, auth, permissions</span>

<span class="cmt">// Compare with the framework-screaming alternative:</span>
src/
├─ controllers/      <span class="cmt">// screams "MVC"</span>
├─ models/           <span class="cmt">// screams "ORM"</span>
├─ views/            <span class="cmt">// screams "templating engine"</span>
└─ services/         <span class="cmt">// screams "service layer pattern"</span></code></pre>

<p>Inside each domain folder the internal layout is up to the team — and that's fine. The key promise is that the <em>top level</em> communicates the business, not the infrastructure. A new developer (or your CTO, or a domain expert from the business side) can read the top-level folders and understand what the system <em>does</em> without knowing how it's built.</p>

<p>Here is an honest appraisal across five practical dimensions:</p>

<div class="blog-table-wrap">

<p>This hybrid buys you the best of both worlds:</p>
<ul>
  <li>The <em>top level screams your business</em> — newcomers know immediately that this is an e-commerce app, not a generic MVC skeleton.</li>
  <li><em>Each feature is self-contained</em> — a change to the billing flow touches only <code>billing/</code>.</li>
  <li><em>Technical layers still exist inside each folder</em>, so developers who think in MVC or Clean Architecture layers aren't lost. They just look inside the feature folder.</li>
  <li><em>A <code>shared/</code> folder</em> holds truly cross-cutting concerns (a <code>Money</code> value object, a logger, a database connection). The key rule: if something is added to <code>shared/</code>, it should be extracted and owned deliberately — not dumped there as an escape hatch.</li>
</ul>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">Naming the shared folder</span><p>Some teams call it <code>shared/</code>, others <code>common/</code>, <code>lib/</code>, or <code>core/</code>. The name matters less than the discipline: treat it like a mini internal library. If a file keeps getting modified alongside a specific feature, it doesn't belong in shared — it belongs inside that feature.</p></div>

<h2 id="at-scale">At team scale: enforced boundaries and monorepos</h2>
<p>As a codebase grows and multiple teams contribute, folder conventions alone aren't enough — a developer can always reach across a boundary with a quick relative import. The next step is making boundaries <em>enforced</em>:</p>
<ul>
  <li><strong>NestJS modules</strong> make inter-module dependencies explicit: you only get access to what another module <code>exports</code>. An import that crosses a module boundary without going through the public API is a compile error.</li>
  <li><strong>Nx boundary rules</strong> let you declare which features or libraries are allowed to depend on which others, and then enforce it as a lint error in CI.</li>
  <li><strong>Monorepos</strong> take this furthest: each domain (orders, billing, auth) becomes its own package with its own <code>package.json</code>, its own tests, and explicit published exports. Turborepo and Nx are the dominant tools in the JavaScript ecosystem for managing this. The structure looks less like a folder tree and more like a graph of packages — but the underlying idea is identical: group by business capability, make boundaries explicit, enforce them mechanically.</li>
</ul>
<p>The jump to a monorepo isn't always necessary. Many teams operate happily with a single package and a well-disciplined hybrid structure for years. Reach for monorepo tooling when cross-team coupling becomes a real pain — not preemptively.</p>

<h2 id="by-company-size">Recommendations by company size</h2>
<p>The right structure depends on where your organization is today. Here's a direct recommendation for each stage:</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr>
      <th>Stage</th>
      <th>Structure to reach for</th>
      <th>Why</th>
      <th>Real-world hint</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th>Solo / pre-launch startup</th>
      <td>By-layer <em>or</em> shallow by-feature</td>
      <td>You have 3–5 features. Any structure works. Optimize for speed and simplicity.</td>
      <td>Most Rails/Express tutorial defaults are fine here. Don't over-engineer.</td>
    </tr>
    <tr>
      <th>Growing startup (5–20 engineers)</th>
      <td>By-feature with a <code>shared/</code> folder</td>
      <td>Features are multiplying. By-layer is starting to hurt. A change in payments shouldn't ripple through orders.</td>
      <td>Many successful Series A companies live here: one repo, one service, features as top-level folders.</td>
    </tr>
    <tr>
      <th>Scale-up (20–100 engineers)</th>
      <td>Hybrid (domain on top, layers inside) + boundary lint rules</td>
      <td>Teams own domains. Accidental cross-domain coupling is a real risk. Enforce boundaries in CI.</td>
      <td>This is the Nx/NestJS module sweet spot. Shopify's core Rails app famously adopted strong domain separation at this stage.</td>
    </tr>
    <tr>
      <th>Enterprise (100+ engineers)</th>
      <td>Monorepo with domain packages, or independent services per bounded context</td>
      <td>Package-level isolation, independent deployment, and versioned public APIs between domains become necessary.</td>
      <td>Google, Meta, and Airbnb operate giant monorepos with strict ownership. Many enterprises go the other way: microservices per domain, which is screaming architecture taken to its logical conclusion.</td>
    </tr>
  </tbody>
</table>
</div>

<div class="blog-callout blog-callout--note"><span class="blog-callout__label">The common mistake</span><p>The most common structural mistake is not choosing the wrong approach — it's <strong>staying on the wrong approach too long</strong>. Teams start with by-layer, feel the pain at 20 features, and still don't refactor because "the structure works, technically". A one-afternoon folder reorganization at 15 features saves months of confusion at 50.</p></div>

<div class="blog-takeaways">
<h3 id="takeaways">Key takeaways</h3>
<ul>
  <li><strong>Structure is architecture.</strong> Your folder layout controls coupling, discoverability, and the blast radius of every future change.</li>
  <li><strong>By-layer</strong> is familiar and fine for small apps, but becomes a junk drawer past 10–15 features.</li>
  <li><strong>By-feature</strong> keeps changes local. A new feature touches one folder. Deleting a feature means deleting one folder.</li>
  <li><strong>Screaming Architecture</strong> says the top level should describe your <em>business</em>, not your framework. If a non-technical stakeholder can read your folder names and understand what the system does, you've succeeded.</li>
  <li><strong>The pragmatic hybrid</strong> most good teams land on: domain/feature names at the top, small technical layers inside each folder.</li>
  <li><strong>Enforce boundaries as you scale.</strong> Folder conventions are suggestions; module systems, lint rules, and monorepo packages make them guarantees.</li>
  <li><strong>Reorganize early</strong> — a one-afternoon folder refactor at 15 features avoids months of confusion at 50.</li>
</ul>
</div>
