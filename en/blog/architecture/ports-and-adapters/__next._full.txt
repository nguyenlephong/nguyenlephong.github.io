The payment code knows about the database, the database code knows about the web request, and somewhere in the middle — barely visible — are the actual business rules that make your product <em>your product</em>.</p>
<p><strong>Ports &amp; Adapters</strong> (you’ll also hear it called <em>Hexagonal Architecture</em>, coined by Alistair Cockburn in 2005) is a way of arranging code so that this stops happening. It’s not a framework, you don’t install it, and it works in any language. It’s really one idea wearing a fancy name. Let’s unwrap it slowly, with pictures and real examples, and by the end you’ll know exactly when to use it — and when not to.</p>

<h2 id="the-problem">The problem: code that grows up badly</h2>
<p>Most apps start the same friendly way — a controller calls a service, the service talks to a database and a couple of vendors, and everyone’s happy. The trouble is what happens <em>over time</em>. Here’s a checkout that’s only a few months old and already painful:</p>
<pre><code><span class="cmt">// checkout.ts — everything tangled together</span>
<span class="kw">class</span> OrderService {
  <span class="kw">async</span> checkout(cart: Cart) {
    <span class="kw">const</span> db = <span class="kw">new</span> PgClient(process.env.DB_URL)        <span class="cmt">// a specific database</span>
    <span class="kw">const</span> stripe = <span class="kw">new</span> Stripe(process.env.STRIPE_KEY)   <span class="cmt">// a specific vendor</span>
    <span class="kw">const</span> sg = <span class="kw">new</span> SendGrid(process.env.SENDGRID_KEY)   <span class="cmt">// another vendor</span>

    <span class="cmt">// the actual business rules are buried between vendor calls…</span>
    <span class="kw">const</span> total = cart.items.reduce((s, i) =&gt; s + i.price, 0)
    <span class="kw">await</span> stripe.charges.create({ amount: total, source: cart.token })
    <span class="kw">await</span> db.query(<span class="str">'INSERT INTO orders …'</span>)
    <span class="kw">await</span> sg.send({ to: cart.email, template: <span class="str">'order-ok'</span> })
  }
}</code></pre>
<p>Nothing here is “wrong”, exactly. But the business logic (what an order <em>is</em>, when it’s valid, what “confirmed” means) is tangled up with three vendors and a database driver. You can’t test the rules without a live database. You can’t move off SendGrid without surgery. And a new teammate reads this and learns about Stripe before they learn about your <em>orders</em>.</p>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">The core question</span><p>Ports &amp; Adapters answers one question: <strong>how do I keep the part that makes money (the business rules) separate from the parts I’ll inevitably swap out (databases, vendors, frameworks, UIs)?</strong></p></div>

<h2 id="core-idea">The core idea, in one picture</h2>
<p>Put your business rules — the <strong>domain</strong> — in the centre. Forbid it from importing any framework, database driver, or vendor SDK. Then let the outside world talk to it only through <strong>ports</strong> (interfaces), with <strong>adapters</strong> doing the messy real-world work on the other side.</p>

<p>That’s the whole shape. People draw a hexagon instead of a circle for a charming reason: the six sides are a reminder that an app has <em>many</em> ways in and out (a web UI, an API, tests, a cron job, a database, a queue…), not just “top” and “bottom”. The number six means nothing — don’t count your ports.</p>

<h2 id="vocabulary">The vocabulary, in plain words</h2>
<p>Three words do all the work. Here they are without jargon:</p>
<ul>
  <li><strong>Domain (the core):</strong> your business rules and concepts — <code>Order</code>, <code>Money</code>, “a cart with zero items can’t check out”. Pure logic. It should read like a description of your business, not your tech stack.</li>
  <li><strong>Port:</strong> an <em>interface</em> the domain owns, describing something it needs or offers — “I need to be able to <code>save(order)</code>” or “I need to <code>charge()</code> a payment”. A port is a promise, written in the domain’s language, with no vendor names in sight.</li>
  <li><strong>Adapter:</strong> the concrete code that fulfils a port using a real technology — a <code>PostgresOrderRepo</code>, a <code>StripeGateway</code>. Adapters are where Stripe, Postgres, and React are allowed to exist.</li>
</ul>
<p>And ports come in two flavours, which is the one subtlety worth learning:</p>

<ul>
  <li><strong>Driving (primary) adapters</strong> sit on the left and <em>call into</em> your app: a web controller, a CLI command, a test. They use an <strong>input port</strong> (your use-case interface).</li>

</ul>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">Quick mnemonic</span><p>If something <em>starts</em> a conversation with your app, it’s <strong>driving</strong>. If your app starts the conversation, the other thing is <strong>driven</strong>. A human clicking “Buy” drives; the payment vendor is driven.</p></div>

<h2 id="before-after">A concrete example: before and after</h2>
<p>Let’s untangle that checkout. First, write down what the domain needs — the <strong>ports</strong> — using only words from your business:</p>
<pre><code><span class="cmt">// ports.ts — the domain's vocabulary. No vendor names allowed.</span>
<span class="kw">interface</span> PaymentGateway { charge(amount: Money, token: string): Promise&lt;Receipt&gt; }
<span class="kw">interface</span> OrderRepository { save(order: Order): Promise&lt;void&gt; }
<span class="kw">interface</span> Notifier        { orderConfirmed(order: Order): Promise&lt;void&gt; }</code></pre>
<p>Now the use case depends on those promises, and <em>nothing else</em>. Read it out loud: it sounds like a description of checking out, not a list of vendors.</p>
<pre><code><span class="cmt">// checkout-service.ts — depends only on ports, never on vendors</span>
<span class="kw">class</span> CheckoutService {
  <span class="kw">constructor</span>(
    <span class="kw">private</span> payments: PaymentGateway,
    <span class="kw">private</span> orders: OrderRepository,
    <span class="kw">private</span> notify: Notifier,
  ) {}

  <span class="kw">async</span> checkout(cart: Cart): Promise&lt;Order&gt; {
    <span class="kw">const</span> order = Order.fromCart(cart)         <span class="cmt">// pure business rule</span>
    <span class="kw">await</span> <span class="kw">this</span>.payments.charge(order.total, cart.token)
    <span class="kw">await</span> <span class="kw">this</span>.orders.save(order)
    <span class="kw">await</span> <span class="kw">this</span>.notify.orderConfirmed(order)
    <span class="kw">return</span> order
  }
}</code></pre>
<p>Finally, the real vendors live in adapters that implement the ports. Swapping one is a localized change — and in tests, you swap them for fakes and run thousands of cases in the time one network call used to take:</p>
<pre><code><span class="cmt">// adapters/*.ts — vendors live here, behind the port they implement</span>
<span class="kw">class</span> StripeGateway      <span class="kw">implements</span> PaymentGateway { <span class="cmt">/* Stripe SDK */</span> }
<span class="kw">class</span> PostgresOrderRepo  <span class="kw">implements</span> OrderRepository { <span class="cmt">/* SQL here */</span> }
<span class="kw">class</span> SendGridNotifier   <span class="kw">implements</span> Notifier        { <span class="cmt">/* SendGrid */</span> }

<span class="cmt">// In a test, swap the real vendors for in-memory fakes — no network, milliseconds:</span>
<span class="kw">const</span> service = <span class="kw">new</span> CheckoutService(<span class="kw">new</span> FakePayments(), <span class="kw">new</span> InMemoryOrders(), <span class="kw">new</span> NullNotifier())</code></pre>
<p>Notice what moved: Stripe, Postgres, and SendGrid are now <em>details at the edge</em>. The thing that makes you money sits in the middle, readable and testable, and it has no idea any of those vendors exist.</p>

<h2 id="dependency-rule">The one rule that holds it together</h2>
<p>If you remember nothing else, remember this: <strong>source-code dependencies always point inward</strong>. The outer ring (adapters, frameworks, I/O) is allowed to know about the inner rings. The domain in the centre is allowed to know about <em>nothing</em> outside itself.</p>

<p>This is why you can replace your entire web framework, or migrate from MySQL to DynamoDB, without the domain noticing. The arrows of dependency never point out of the core, so changes at the edge can’t ripple into it. (If you’ve heard of the <em>Dependency Inversion Principle</em> — the “D” in SOLID — this is it in practice: the domain defines the interface, the adapter conforms to it.)</p>

<h2 id="primary-secondary">Driving vs driven, made practical</h2>
<p>Why does the primary/secondary split matter day to day? Because it tells you <em>who owns the interface</em>.</p>
<ul>
  <li>For a <strong>driven</strong> port (database, email), the <em>domain</em> owns the interface and the adapter bends to fit. That’s what lets you swap vendors freely.</li>
  <li>For a <strong>driving</strong> port (your use cases), the <em>application</em> owns the interface and the outside world (HTTP, CLI, a test harness) calls it. That’s what lets the same logic power a REST API today and a gRPC endpoint or a scheduled job tomorrow with zero rule changes.</li>
</ul>
<p>A useful litmus test: a brand-new delivery mechanism (say, a Slack bot) should be <em>only</em> a new driving adapter. A brand-new vendor (say, switching from Stripe to Adyen) should be <em>only</em> a new driven adapter. If either one forces you to open the domain, something has leaked.</p>

<h2 id="by-company-size">What it looks like at different company sizes</h2>
<p>This is where a lot of advice goes wrong — it treats architecture as one-size-fits-all. The <em>right</em> amount of Ports &amp; Adapters depends heavily on where your company is. Here’s the honest version:</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>Company stage</th><th>The pain it removes</th><th>How much to apply</th></tr>
  </thead>
  <tbody>
    <tr>
      <th>Solo / early startup</th>
      <td>You picked SQLite and a payment provider in a hurry and now fear you’re locked in.</td>
      <td>One or two ports max — usually the database and payments. Leave everything else direct. Speed beats purity here.</td>
    </tr>
    <tr>
      <th>Small / growing (≈ Series A)</th>
      <td>Tests hit the real database and a vendor sandbox, so they’re slow, flaky, and devs stop running them.</td>
      <td>Put ports around all I/O. Fakes make unit tests run in milliseconds and new hires understand the domain in a day.</td>
    </tr>
    <tr>
      <th>Mid-size / scale-up</th>
      <td>Several teams step on each other; a cost-cutting “move off SendGrid” turns into a month-long rewrite.</td>
      <td>Ports become <em>contracts</em> between teams. Swapping a vendor is one new adapter, not a migration.</td>
    </tr>
    <tr>
      <th>Big corp / enterprise</th>
      <td>Legacy systems, regional vendors, audits, and procurement rules that demand “no single-vendor lock-in”.</td>
      <td>Many adapters per port (old + new), strangler-fig migrations, and vendor independence as a written requirement.</td>
    </tr>
  </tbody>
</table>
</div>
<h3 id="size-stories">Three quick real-world stories</h3>
<p><strong>The small e-commerce shop (10 people).</strong> They wrapped email behind a one-method <code>Notifier</code> port “just in case”. Two years later their email costs spiked; moving from SendGrid to Amazon SES was a single new adapter and a one-line wiring change, shipped in an afternoon. The order logic — their crown jewels — wasn’t touched.</p>
<p><strong>The scale-up fintech (≈120 engineers).</strong> Payments had to run through a local provider in one country and an international one in another, with the choice made at runtime per transaction. Because <code>PaymentGateway</code> was a port, “which provider” became a routing decision behind the interface. Compliance auditors loved it too: the boundary was an obvious, testable seam.</p>
<p><strong>The enterprise bank (1,000+ engineers).</strong> A 20-year-old mainframe couldn’t be replaced overnight. They defined an <code>AccountLedger</code> port and wrote <em>two</em> adapters — one talking to the old mainframe, one to the new core-banking system — then migrated traffic gradually (a “strangler fig”). Teams built new features against the port while the migration happened underneath them, invisible.</p>

<h2 id="vs-layered">Ports &amp; Adapters vs the classic layered approach</h2>
<p>Most teams come from a traditional layered architecture (UI on top, services in the middle, database at the bottom). Here’s how the two compare:</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>Question</th><th>Classic layered (UI → service → DB)</th><th>Ports &amp; Adapters</th></tr>
  </thead>
  <tbody>
    <tr><th>Who depends on whom?</th><td>Top-down: every layer depends on the one below, ending at the database.</td><td>Everything depends <em>inward</em>, ending at the domain. The database is just another plug-in.</td></tr>
    <tr><th>Can you test rules without a DB?</th><td>Usually no — the service reaches the DB directly.</td><td>Yes — inject a fake repository; no DB, no network.</td></tr>
    <tr><th>Swap a vendor?</th><td>Touches the service layer and ripples outward.</td><td>Write one new adapter; the domain never changes.</td></tr>
    <tr><th>Where do business rules live?</th><td>Often smeared across services and the DB.</td><td>Concentrated in the domain, in plain language.</td></tr>
    <tr><th>Cost</th><td>Low ceremony, but couples you to your stack over time.</td><td>A few extra interfaces up front; pays off as change accelerates.</td></tr>
  </tbody>
</table>
</div>
<p>The headline difference is the direction of the arrows. In layered code, everything ultimately depends on the database. In Ports &amp; Adapters, the database depends on you. That inversion is the entire point.</p>

<h2 id="when-and-when-not">When to reach for it (and when not)</h2>
<p>Good architecture is about matching effort to stakes. Reach for Ports &amp; Adapters when:</p>
<ul>
  <li>The app has <strong>real business rules</strong> worth protecting (not just CRUD over a table).</li>
  <li>You expect to <strong>swap or add</strong> integrations — vendors, databases, channels — over its life.</li>
  <li><strong>Testing speed and confidence</strong> matter, and you want fast, deterministic unit tests.</li>
  <li>Multiple teams or a long lifespan mean <strong>clear boundaries</strong> pay for themselves.</li>
</ul>
<p>Be skeptical — or apply it very lightly — when:</p>
<ul>
  <li>It’s a <strong>thin CRUD app</strong> or a throwaway prototype; the indirection is pure overhead.</li>
  <li>The “domain” is basically your database schema; there’s nothing to protect.</li>
  <li>You’re a <strong>solo dev shipping fast</strong> and the cost of a future rewrite is lower than the cost of ceremony now.</li>
</ul>
<div class="blog-callout blog-callout--warn"><span class="blog-callout__label">Don’t cargo-cult it</span><p>Wrapping a value-less CRUD endpoint in three interfaces doesn’t make it “clean” — it makes it harder to read. Architecture is a cost you pay to buy <em>changeability</em>. If nothing’s going to change, you’re paying for nothing.</p></div>

<h2 id="mistakes">Common mistakes (and easy fixes)</h2>
<ul>
  <li><strong>Leaky ports.</strong> If your <code>PaymentGateway</code> returns a <code>Stripe.Charge</code> object, the vendor has leaked through the port. Fix: ports speak only in your own types (<code>Receipt</code>, <code>Money</code>).</li>
  <li><strong>Anemic domain.</strong> If all the real logic lives in adapters and the domain is just data bags, you’ve drawn the boundary in the wrong place. Fix: push rules <em>into</em> the core.</li>
  <li><strong>One giant port.</strong> A single <code>IRepository</code> with 40 methods isn’t a boundary, it’s a junk drawer. Fix: small, purpose-named ports (<code>OrderRepository</code>, <code>InventoryRepository</code>).</li>
  <li><strong>Confusing DTOs with domain models.</strong> The shape you send over HTTP isn’t your domain entity. Fix: let adapters translate between the wire format and the domain.</li>
  <li><strong>Interfaces for everything.</strong> Not every class needs a port — only the things that cross a boundary (I/O, vendors, delivery mechanisms). Fix: add a port when you feel the pain, not preemptively.</li>
</ul>

<h2 id="getting-started">How to start on Monday</h2>
<p>You don’t rewrite anything. You introduce one seam where it hurts most, and stop there:</p>
<ul>
  <li><strong>1. Pick your most painful dependency.</strong> Usually the database or a flaky vendor.</li>
  <li><strong>2. Write the interface you <em>wish</em> you had.</strong> In your business’s words: <code>save(order)</code>, not <code>execSql(...)</code>.</li>
  <li><strong>3. Move the vendor code behind an adapter</strong> that implements it. Nothing else changes yet.</li>
  <li><strong>4. Inject the adapter</strong> instead of constructing it inline, and write one fake for tests.</li>
  <li><strong>5. Feel the relief, then repeat</strong> — but only where churn or testing pain justifies another port.</li>
</ul>
<p>Within a week you’ll have fast tests around your nastiest integration and a domain that’s starting to read like your business. That’s the whole win, available incrementally.</p>

<div class="blog-takeaways">
<h3 id="takeaways">Key takeaways</h3>
<ul>
  <li><strong>One idea:</strong> business rules in the centre, technology at the edge, talking only through interfaces.</li>
  <li><strong>Port</strong> = an interface the domain owns. <strong>Adapter</strong> = the real implementation (Stripe, Postgres, React).</li>
  <li><strong>Dependencies point inward.</strong> The domain knows nothing about the outside world.</li>
  <li><strong>Driving</strong> adapters call your app; <strong>driven</strong> adapters are called by it.</li>
  <li><strong>Match the dose to the stakes:</strong> a couple of ports for a startup, vendor-independent boundaries for an enterprise, and almost none for a throwaway CRUD app.</li>
  <li><strong>Adopt it incrementally</strong> — one painful seam at a time. You never need a big rewrite.</li>
</ul>
</div>
