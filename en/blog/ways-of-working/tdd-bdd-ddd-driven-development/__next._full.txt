19:T485e,<p>Spend a year in software and you'll collect a confusing little alphabet: TDD, BDD, DDD, ATDD, EDD, CDD, FDD. They <em>sound</em> like competing religions you must choose between — and engineers argue about them as if they were. They're not. Most of them answer <strong>different questions</strong>, operate at <strong>different layers</strong>, and happily <strong>compose</strong> on the same project.</p>
<p>The shared suffix is the key. "X-Driven Development" means: <strong>you let X drive the order and shape of your work</strong> — X is the thing you start from and the thing that forces your decisions. Test-Driven means tests come first and shape the code. Domain-Driven means the business domain shapes the design. Event-Driven means events shape the architecture. Once you see that, the soup turns into a clear menu.</p>
<p>This guide is written for engineers who want to actually <em>use</em> these — not recite them. For each one: what it really means, the core loop or concept, a concrete example, when to reach for it, and the pitfalls. We'll end with a map showing how they all fit together on a single feature.</p>

<figure class="blog-figure blog-photo-panorama">
  <img src="https://nguyenlephong.github.io/dom-pub/icdn/blogs/tdd-bdd-ddd-driven-development/behavior-workshop-alignment.webp" width="1600" height="900" loading="lazy" decoding="async" alt="Phong and two teammates compare a tablet flow, printed diagrams, and abstract workflow cards to turn several development methods into one clear shared plan." />
  <figcaption>The alphabet soup gets calmer when a team treats each method as a different lens on the same feature, not as a religion to defend.</figcaption>
</figure>

<div class="blog-callout blog-callout--note"><span class="blog-callout__label">A quick orientation</span><p>They live on different axes, which is exactly why they combine rather than compete:<br/>• <strong>Process / testing</strong> — TDD, BDD, ATDD (how and when you write tests)<br/>• <strong>Design</strong> — DDD (how you model the problem)<br/>• <strong>Architecture</strong> — EDD / event-driven (how parts communicate)<br/>• <strong>Build approach</strong> — CDD, FDD (what unit you build around)</p></div>

<h2 id="tdd">TDD — Test-Driven Development</h2>
<p>The most famous of the family. You write a <strong>failing test first</strong>, then the minimum code to pass it, then clean up — the <strong>Red → Green → Refactor</strong> loop. The test isn't an afterthought to verify code; it's a design tool that forces you to define behavior before implementation.</p>
<pre><code><span class="cmt">// 1. RED — write the test first; it fails (function doesn't exist yet)</span>
test(<span class="str">'applies a 10% discount over $100'</span>, () => {
  expect(priceAfterDiscount(<span class="str">120</span>)).toBe(<span class="str">108</span>)
})

<span class="cmt">// 2. GREEN — the minimum code to make it pass</span>
<span class="kw">function</span> priceAfterDiscount(total) {
  <span class="kw">return</span> total > <span class="str">100</span> ? total * <span class="str">0.9</span> : total
}

<span class="cmt">// 3. REFACTOR — improve the code, tests stay green</span></code></pre>
<p><strong>Why it works:</strong> writing the test first forces you to think about the interface and the edge cases before you're attached to an implementation. It produces a safety net of regression tests as a side effect, and it tends to push you toward small, testable, decoupled units.</p>
<div class="blog-callout blog-callout--warn"><span class="blog-callout__label">Pitfalls</span><p>TDD tests <em>behavior at the unit level</em> — it doesn't guarantee the right product. Over-mocking leads to tests that pass while the system is broken. And TDD'ing throwaway spikes or trivial code is wasted ceremony. Use it where logic is non-trivial and correctness matters.</p></div>

<h2 id="bdd">BDD — Behavior-Driven Development</h2>
<p>BDD is TDD's evolution toward <em>shared understanding</em>. Instead of unit tests written in code-speak, you describe <strong>behavior in plain language</strong> that business and engineers agree on, using a structured <strong>Given / When / Then</strong> format (often written in Gherkin). It pushes the conversation up from "does this function return 108?" to "what should the system <em>do</em> for the user?"</p>
<pre><code><span class="kw">Feature</span>: Checkout discount

  <span class="kw">Scenario</span>: Order over $100 gets 10% off
    <span class="kw">Given</span> a cart totaling $120
    <span class="kw">When</span> the customer checks out
    <span class="kw">Then</span> the final price should be $108</code></pre>
<p>Those steps are then wired to test code, so the plain-language spec becomes an executable, living document. The big win is <strong>ubiquitous language</strong> and collaboration: the BA, PO, QA, and developer all read the same scenario and mean the same thing.</p>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">BDD vs ATDD vs TDD</span><p><strong>TDD</strong> drives <em>code</em> from unit tests (developer-facing). <strong>ATDD</strong> (Acceptance Test-Driven Development) drives <em>features</em> from acceptance tests agreed up front. <strong>BDD</strong> is essentially ATDD with a disciplined, behavior-focused language and tooling. In practice: BDD/ATDD define the <em>outer</em> loop (the feature is right), TDD runs the <em>inner</em> loop (the units are right). They nest.</p></div>

<h2 id="ddd">DDD — Domain-Driven Design</h2>
<p>Note the D: <strong>Design</strong>, not Development. DDD is about modeling complex business domains so the code reflects the real problem. It has two halves:</p>
<p><strong>Strategic DDD</strong> — the big-picture boundaries:</p>
<ul>
<li><strong>Ubiquitous language</strong> — one shared vocabulary used by domain experts <em>and</em> in the code. If the business says "policy," the class is <code>Policy</code>, not <code>InsuranceRecord</code>.</li>
<li><strong>Bounded context</strong> — a boundary within which a model and its language are consistent. "Customer" in <em>Billing</em> and "Customer" in <em>Support</em> are different models; trying to force one shared "Customer" everywhere is a classic mistake.</li>
</ul>
<p><strong>Tactical DDD</strong> — the building blocks inside a context:</p>
<ul>
<li><strong>Entity</strong> — has identity over time (an <code>Order</code> with an id).</li>
<li><strong>Value object</strong> — defined only by its values, immutable (a <code>Money</code> of {amount, currency}).</li>
<li><strong>Aggregate</strong> — a cluster of objects treated as one unit with one entry point (the <em>aggregate root</em>) that guards the rules (invariants).</li>
<li><strong>Repository</strong> — gives the illusion of an in-memory collection of aggregates.</li>
<li><strong>Domain event</strong> — something meaningful that happened (<code>OrderPlaced</code>).</li>
</ul>
<figure class="blog-figure blog-photo-figure">
  <img src="https://nguyenlephong.github.io/dom-pub/icdn/blogs/tdd-bdd-ddd-driven-development/domain-modeling-whiteboard.webp" width="1440" height="960" loading="lazy" decoding="async" alt="Phong points across a whiteboard of grouped domain regions and event paths while two teammates compare a laptop and printed model sketches." />
  <figcaption>Domain-Driven Design starts paying off when the conversation is about boundaries, language, and invariants before it is about classes.</figcaption>
</figure>
<pre><code><span class="cmt">// The Order aggregate root guards its own invariants</span>
<span class="kw">class</span> Order {
  addItem(product, qty) {
    <span class="kw">if</span> (<span class="kw">this</span>.status !== <span class="str">'DRAFT'</span>)
      <span class="kw">throw</span> <span class="kw">new</span> Error(<span class="str">'Cannot modify a placed order'</span>)
    <span class="kw">this</span>.items.push({ product, qty })
  }
  place() {
    <span class="kw">if</span> (<span class="kw">this</span>.items.length === <span class="str">0</span>)
      <span class="kw">throw</span> <span class="kw">new</span> Error(<span class="str">'Cannot place an empty order'</span>)
    <span class="kw">this</span>.status = <span class="str">'PLACED'</span>
    <span class="kw">this</span>.raise(<span class="kw">new</span> OrderPlaced(<span class="kw">this</span>.id))
  }
}</code></pre>
<p>The point: business rules live <em>in the domain model</em>, not scattered across controllers and services. DDD shines on <strong>complex domains</strong> with rich rules; it's overkill for a simple CRUD app. It also pairs naturally with the architectures in <a href="/blog/architecture/clean-vs-onion-vs-hexagonal">Clean / Onion / Hexagonal</a>, which exist precisely to keep that domain model pure.</p>
<div class="blog-callout blog-callout--warn"><span class="blog-callout__label">Pitfalls</span><p>DDD is frequently cargo-culted — teams adopt the tactical patterns (entities, repositories) while ignoring the strategic heart (ubiquitous language, bounded contexts), and end up with ceremony without benefit. If your domain is genuinely simple, plain code beats a DDD costume.</p></div>

<h2 id="edd">EDD — Event-Driven Development</h2>
<p>Event-Driven Development (and its bigger sibling, event-driven architecture) makes <strong>events</strong> the backbone: instead of components calling each other directly, they <em>emit</em> facts about what happened, and other components <em>react</em>. "Something happened" replaces "do this now."</p>
<pre><code><span class="cmt">// The order service just announces the fact — it doesn't know who cares</span>
eventBus.publish(<span class="kw">new</span> OrderPlaced({ orderId, total }))

<span class="cmt">// Independent consumers react on their own</span>
onEvent(<span class="str">'OrderPlaced'</span>, e => inventory.reserve(e.orderId))
onEvent(<span class="str">'OrderPlaced'</span>, e => email.sendConfirmation(e.orderId))</code></pre>
<p><strong>Why reach for it:</strong> loose coupling (the producer doesn't know its consumers), independent scaling, and a natural fit for asynchronous, distributed systems. It underpins patterns like event sourcing (store the events, derive state) and CQRS. Domain events from DDD are often what flow through an event-driven system — the two compose beautifully.</p>
<div class="blog-callout blog-callout--warn"><span class="blog-callout__label">Pitfalls</span><p>Events trade direct calls for indirection: flows become harder to trace, ordering and duplicate delivery become real problems, and "eventual consistency" surprises people who expected immediate reads. Great for decoupling at scale; needless pain for a small synchronous app.</p></div>

<h2 id="cdd">CDD — Component-Driven Development</h2>
<p>Most common in frontend: you build the UI <strong>from the bottom up, one isolated component at a time</strong>, before assembling them into pages. Each component is developed and reviewed in isolation (often in a tool like Storybook), with all its states visible at once.</p>
<pre><code><span class="cmt">// Build & review Button in isolation, every state, before any page uses it</span>
<span class="kw">export const</span> Primary = { args: { variant: <span class="str">'primary'</span>, label: <span class="str">'Buy now'</span> } }
<span class="kw">export const</span> Loading = { args: { loading: <span class="kw">true</span> } }
<span class="kw">export const</span> Disabled = { args: { disabled: <span class="kw">true</span> } }</code></pre>
<p><strong>Why it works:</strong> reusable, consistent UI; a living component library; parallel work; and components tested before integration. It's the build-side cousin of a design system.</p>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">CDD has a second meaning: Contract-Driven</strong></span><p>In backend/microservices, <strong>CDD</strong> often means <strong>Contract-Driven Development</strong>: services agree on an explicit API <em>contract</em> first (e.g. consumer-driven contracts with a tool like Pact), and both sides test against it independently. Same spirit — the contract drives the work — different layer.</p></div>

<h2 id="fdd">FDD — Feature-Driven Development</h2>
<p>An older, lightweight Agile methodology that organizes the whole process around a list of small, client-valued <strong>features</strong> ("calculate the total of a sale"). You build a domain model, build a feature list, then plan/design/build <em>by feature</em> in short iterations. Think of it as a feature-centric way to run delivery — closer in spirit to Scrum's planning than to TDD's code loop.</p>

<h2 id="edge">The newer and niche members</h2>
<ul>
<li><strong>Type-Driven Development</strong> — in strongly-typed languages, you design with the type system first ("make illegal states unrepresentable"), letting the compiler drive correctness before tests even run.</li>
<li><strong>Model-Driven Development (MDD)</strong> — generate code from higher-level models; common in regulated or low-code contexts.</li>
<li><strong>Eval-Driven Development</strong> — the AI-era newcomer. For LLM features, classic pass/fail tests break (outputs are non-deterministic), so you drive development with <strong>evals</strong>: graded datasets that measure output quality. It's TDD's spirit applied to probabilistic systems — see <a href="/blog/ai/how-ai-changes-software-roles">how AI is reshaping software roles</a>.</li>
</ul>

<h2 id="together">How they fit together — one feature, many "DD"s</h2>
<p>Here's the punchline: these aren't a menu to pick <em>one</em> from. A single well-built feature can use several at once, each on its own layer:</p>
<figure class="blog-figure blog-photo-figure">
  <img src="https://nguyenlephong.github.io/dom-pub/icdn/blogs/tdd-bdd-ddd-driven-development/feature-layer-workflow.webp" width="1440" height="960" loading="lazy" decoding="async" alt="Phong and two teammates align behavior cards, component tiles, and event-flow diagrams across paper, tablet, and laptop at one table." />
  <figcaption>A healthy feature uses different drivers at different layers, then pulls them back into one shared workflow.</figcaption>
</figure>
<div class="blog-table-wrap">
<table>
<thead><tr><th width="22%">Layer</th><th>Method</th><th>Its job on the feature</th></tr></thead>
<tbody>
<tr><td>Shared understanding</td><td><strong>BDD / ATDD</strong></td><td>Define the behavior everyone agrees on (Given/When/Then)</td></tr>
<tr><td>Design</td><td><strong>DDD</strong></td><td>Model the domain — the <code>Order</code> aggregate and its rules</td></tr>
<tr><td>Code</td><td><strong>TDD</strong></td><td>Drive each unit with failing tests first</td></tr>
<tr><td>Architecture</td><td><strong>EDD</strong></td><td>Emit <code>OrderPlaced</code>; let inventory and email react</td></tr>
<tr><td>UI build</td><td><strong>CDD</strong></td><td>Build the checkout components in isolation</td></tr>
</tbody>
</table>
</div>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">A nested loop, in practice</span><p>A common, healthy combination: <strong>BDD</strong> defines the outer loop ("the checkout discount works for the user"), <strong>TDD</strong> runs the inner loop (each function is correct), <strong>DDD</strong> shapes what those units <em>are</em>, and <strong>EDD</strong> connects the pieces. You don't announce "we do five DDs" — you just use the right driver at the right layer.</p></div>

<h2 id="choosing">Choosing wisely — and avoiding dogma</h2>
<p>The throughline across all of these: a methodology is a tool, not an identity. A few principles:</p>
<ul>
<li><strong>Match the method to the problem.</strong> Rich domain → DDD. Logic-heavy code → TDD. Cross-team features → BDD. Decoupled/async systems → EDD. Component libraries → CDD. Simple CRUD → don't over-engineer any of them.</li>
<li><strong>They compose; they don't compete.</strong> The question is rarely "TDD or DDD?" — it's "which drivers does this feature need, at which layers?"</li>
<li><strong>Beware cargo-culting.</strong> Adopting the rituals (folders named <code>aggregates/</code>, empty Gherkin files) without the underlying purpose adds ceremony and removes velocity.</li>
<li><strong>Dogma is the real anti-pattern.</strong> "100% TDD always" or "DDD everywhere" causes more harm than skipping them. Be deliberate, not religious.</li>
</ul>

<div class="blog-takeaways">
<h3 id="takeaways">Key takeaways</h3>
<ul>
<li><strong>"X-Driven" means X drives the order and shape of your work</strong> — the thing you start from and that forces your decisions.</li>
<li><strong>They live on different axes:</strong> process/testing (TDD, BDD, ATDD), design (DDD), architecture (EDD), build approach (CDD, FDD) — so they <em>compose</em> rather than compete.</li>
<li><strong>TDD:</strong> Red → Green → Refactor; tests as a design tool for the inner loop.</li>
<li><strong>BDD/ATDD:</strong> behavior in shared Given/When/Then language; the outer loop and ubiquitous understanding.</li>
<li><strong>DDD:</strong> model complex domains — ubiquitous language and bounded contexts (strategic) + entities, value objects, aggregates, events (tactical). It's Design, not Development.</li>
<li><strong>EDD:</strong> components emit events and react, for loose coupling in async/distributed systems.</li>
<li><strong>CDD:</strong> build UI from isolated components up (or, in backend, Contract-Driven). <strong>FDD:</strong> organize delivery around small client-valued features.</li>
<li><strong>Avoid dogma and cargo-culting.</strong> Match the driver to the problem and the layer; a methodology is a tool, not an identity.</li>
</ul>
</div>

<p>The "-Driven Development" family looks intimidating only until you see the trick hiding in the shared suffix: each one simply names <em>what you let lead</em>. Tests, behavior, the domain, events, components, features — different leaders for different parts of the work. The senior move isn't memorizing all of them or swearing allegiance to one; it's knowing, for the problem in front of you, which driver belongs at which layer — and having the judgment to leave the rest in the toolbox.
