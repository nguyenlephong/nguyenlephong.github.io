20:T4761,<p>你肯定遇到过这种情况：周二下午，你的审查队列里出现了一个 pull request，标题是<strong>"fixes"</strong>，diff 跨越 41 个文件共 2347 行，没有描述，评论里只有一个轻快的"PTAL"。你的心一沉。还是打开了。二十分钟后你把同一个方法看了三遍，仍然不知道它解决了什么问题，只留下了一条忐忑的评论："总体上看起来不错，也许把这个变量改个名？"你刚刚给一段你没看懂的代码盖了橡皮图章。</p>
<p>作者并非恶意——他们可能全神贯注于工作，为自己的成果感到骄傲，只是忘记了交付代码只是工作的一半。另一半是让它被审查、被理解、安全地合并。Pull request 不是合并前的手续；它是一次对话，像所有对话一样，当有人付出努力让对方理解时，效果会更好。</p>
<p>本文就是关于这种努力的：如何写出队友真正乐于审查的 PR——简小、有目的、易于跟进，也容易说"是"。</p>

<h2 id="pr-is-communication">PR 是沟通，不只是代码</h2>
<p>在第一行 diff 之前，pull request 首先是给另一个人的一条信息。那个人时间有限，对你晚上十一点在想什么缺乏上下文，还有十几件其他事情在争夺他的注意力。你作为作者的工作，是让他们的工作变得简单。</p>
<p>把审查者当作你的受众，而不是你的法官。他们不是来抓你把柄的——他们是帮你交付稳健成果的合作者。你越帮助他们理解变更背后的<em>为什么</em>，审查就会越快越好。一个出色的 PR 描述就像是你代码的求职信：它铺垫了场景，解释了动机，并告诉审查者具体看什么。</p>
<p>这种思维方式的转变改变了一切。一旦你停止为自己写 PR，开始为审查者写，你自然会希望把它写得更小、描述得更好，并在请人看之前整理好。</p>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">审查者的黄金法则</span><p>如果审查者必须通过读代码才能理解<em>为什么</em>这个变更存在，PR 描述就没有尽到职责。代码解释了<em>什么</em>发生了变化；描述解释了<em>为什么它需要变化</em>。</p></div>

<h2 id="keep-it-small">保持简小而聚焦</h2>
<p>改善 pull request 最有效的一件事就是把它写小。PR 越大，审查就越不彻底，合并时间越长，漏过的 bug 也越多。</p>
<p>一个好的 PR 包含<strong>一个逻辑变更</strong>。不是一张工单，不是一个功能，不是一个 sprint 的工作——而是一个连贯的想法，可以被独立理解、测试和回滚。"添加用户头像上传"是一个变更。"添加用户头像上传并重构 profile controller 并升级三个依赖"是三个变更穿了一件外衣。</p>

<h3 id="how-to-split">如何拆分大型工作</h3>
<p>大型功能不必以大型 PR 的形式出现。几个实用的拆分策略：</p>
<ul>
  <li><strong>将重构与行为变更分开。</strong>第一个 PR：重命名、重组、提取——没有新逻辑。第二个 PR：实际的功能。审查者可以验证重构没有副作用，然后完全专注于新逻辑。</li>
  <li><strong>先提交基础设施。</strong>添加数据库列、迁移、类型——全部不带 UI。然后添加使用它的 UI。每个 PR 可以独立部署。</li>
  <li><strong>使用功能开关。</strong>把未完成的工作合并到默认关闭的开关后面。PR 小而安全；功能在翻转开关之前还不算"完成"。</li>
  <li><strong>堆叠 PR。</strong>PR B 基于 PR A。一些团队使用堆叠 PR 工具独立审查每一层。如果你的团队没有堆叠工具，保持简单：先完成并合并 A，然后开 B。</li>
</ul>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">一个实用的启发</span><p>如果你无法用一句话、不带"并且"两字来描述你的 PR，它可能包含了多于一个逻辑变更。找到接缝，拆开它。</p></div>

<h2 id="title-and-description">写一个值得一读的标题和描述</h2>
<p>标题是标题行。它应该用一行说明<em>什么</em>发生了变化以及为什么重要。避免像"update"、"fix stuff"或"changes"这样模糊的动词，追求具体性。以下是常见的 PR 标题坏味道及改进方法：</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>PR 坏味道</th><th>问题所在</th><th>更好的版本</th></tr>
  </thead>
  <tbody>
    <tr>
      <th>"fixes"</th>
      <td>完全没有上下文——审查者必须读每一行来猜发生了什么。</td>
      <td>"修复并发会话碰撞时购物车结账的竞态条件"</td>
    </tr>
    <tr>
      <th>"WIP: various changes"</th>
      <td>表明作者还没准备好。草稿 PR 的存在就是为了这种情况。</td>
      <td>在真正准备好审查之前作为 GitHub 草稿开着。</td>
    </tr>
    <tr>
      <th>"Update UserService"</th>
      <td>描述了在哪里，而不是什么或为什么。每个 PR 都在更新某些东西。</td>
      <td>"为 UserService 添加邮件更改验证流程"</td>
    </tr>
    <tr>
      <th>"JIRA-1234"</th>
      <td>迫使审查者打开第二个标签页才能理解动机。</td>
      <td>"将每 IP 的 API 速率限制为 100 次/分钟（JIRA-1234）"——工单作为上下文，而不是替代品。</td>
    </tr>
    <tr>
      <th>"Refactor + new feature + bump deps"</th>
      <td>三个 PR 穿了一件外衣。每个关注点都应该有自己的审查。</td>
      <td>拆分为三个独立的、有焦点的 PR。</td>
    </tr>
  </tbody>
</table>
</div>

<p><strong>描述</strong>是真正沟通发生的地方。一个好的描述回答了审查者在最初十秒内提出的三个问题：</p>
<ul>
  <li><strong>什么</strong>发生了变化，在哪里？</li>
  <li><strong>为什么</strong>需要这个变更——背景、bug、用户故事？</li>
  <li><strong>如何验证</strong>它有效——测试步骤、截图、受影响的边界情况？</li>
</ul>
<p>来看一个前后对比，让它更具体：</p>
<pre><code><span class="cmt">--- 改之前（2000 行的 "fixes" PR）---</span>

<span class="kw">标题：</span> <span class="str">fixes</span>

<span class="kw">描述：</span>（空）


<span class="cmt">--- 改之后（为审查者写的同一变更）---</span>

<span class="kw">标题：</span> <span class="str">修复并发会话时购物车结账的竞态条件</span>

<span class="kw">## 是什么</span>
同一用户的两个并发结账请求都可能在任何一方递减库存数量之前通过库存检查，
导致超卖。本 PR 在库存递减操作周围添加了行级数据库锁。

<span class="kw">## 为什么</span>
客服在过去 30 天内反映了 12 起超卖事件（见 Sentry issue #4421）。
根本原因是两个请求在 inventory 表的 SELECT 和 UPDATE 之间产生竞争。

<span class="kw">## 如何测试</span>
1. 运行新的并发回归测试：npm test -- cart.concurrency
2. 在 staging：打开两个浏览器标签，对库存为 1 的商品同时点击"购买"。
   只有一个应该成功。

<span class="kw">## 风险/备注</span>
- 锁定范围仅限于单个商品行；不同商品同时结账不受影响。
- 无需迁移——使用现有表的 SELECT FOR UPDATE。</code></pre>

<h2 id="commit-hygiene">提交卫生</h2>
<p>如何组织你的提交很重要——不只是为了满足流程，而是因为提交是变更如何形成的永久记录。一年后，某人（可能是你）会在一行代码上运行 <code>git blame</code>，然后沿着提交哈希追溯历史。他们找到的应该讲述一个故事，而不是一段意识流。</p>
<p>一个好的提交信息遵循约定格式：一个简短的祈使句主题行（50 个字符或更少），一个可选的空行，以及一个可选的正文解释<em>为什么</em>——而不是 diff 已经显示的内容。</p>
<pre><code><span class="cmt"># 约定格式：</span>
<span class="kw">type(scope): 简短的祈使句主题</span>

<span class="str">可选正文：这次变更的原因，它解决了什么问题，
任何不明显的决策及其背后的推理。</span>

<span class="cmt"># 真实示例：</span>
<span class="kw">fix(cart):</span> 添加行级锁以防止并发结账时超卖

<span class="kw">feat(auth):</span> 添加邮件更改验证流程

<span class="kw">refactor(profile):</span> 提取头像上传为独立服务</code></pre>
<p>除了信息格式，还要考虑提交的<em>形态</em>。每个提交应该代表一个连贯的步骤——重构、测试、新行为。查看单个提交（而不是整个 diff）的审查者，应该能跟上逻辑进展："提取方法 → 添加测试 → 实现行为"比二十个在最后一刻压缩的"WIP"提交讲述的故事更好。</p>
<p>在开 PR 之前，使用交互式 rebase 整理混乱的分支。你不需要从一开始就有完美的提交——在请人看之前再打磨。</p>

<h2 id="self-review">请人审查前先自审</h2>
<p>在 ping 队友之前，先成为自己的第一个审查者。在 GitHub（或你选用的代码托管平台）上打开 diff，当作从未见过这段代码来读。这种上下文切换抓到的问题比你预想的更多：遗留的调试输出、增长得太大的函数、忘记移除的 import、曾经为真但现在不对的注释。</p>
<p>值得过一遍的自审清单：</p>
<ul>
  <li><strong>从头到尾读 diff。</strong>不要快速浏览。假装自己第一次看到它。</li>
  <li><strong>检查调试遗留物。</strong><code>console.log</code>、标注"待删除"的 TODO 注释、注释掉的代码块、临时硬编码的值。</li>
  <li><strong>验证测试是否存在。</strong>如果你添加了逻辑，有测试吗？如果你修复了 bug，有回归测试吗？</li>
  <li><strong>寻找无意改动。</strong>空白字符变动、被重新格式化的文件、意外的范围扩大。把它们移到单独的 PR 或者还原。</li>
  <li><strong>自己留下引导性评论。</strong>如果某个部分不明显，在审查者看到之前先加一条 PR 评论解释推理。这不是软弱的表现——这是体贴。</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">一个简单的规则</span><p>如果你不愿意让一个高级工程师现在立刻、不加解释地审查这个 PR——它还没准备好。先修改，再请求审查。</p></div>
<p>自审也有自私的好处：它在问题变成永久的审查历史评论之前抓住了令人尴尬的错误。每个人都有一个故事：按下"请求审查"后十秒就发现了那个拼写错误。十分钟的自审消除了大多数这类情况。</p>

<h2 id="responding">回复审查反馈</h2>
<p>PR 不是在你推送代码时就结束了——而是在它合并时才结束。在这两个事件之间是审查循环，你如何处理反馈与代码本身同样重要。</p>
<p>保持审查推进、关系不受损的几个原则：</p>
<ul>
  <li><strong>回复每一条评论。</strong>哪怕只是说"已完成"或"好发现，在最新提交中已修复"。沉默让审查者不确定你是否看到了他们的备注。</li>
  <li><strong>推送修复时描述你做了什么。</strong>"在最新提交中修复了——把逻辑移到了辅助函数里并添加了测试"比一个没有上下文的新提交要清晰得多。</li>
  <li><strong>审慎地解决线程。</strong>在大多数团队，评论者满意后才解决线程。不要在没有确认反馈的情况下自己关闭。如果你不确定约定，就问。</li>
  <li><strong>有礼貌地不同意。</strong>如果你认为某个建议是错的，清楚地说出来并给出理由——不要就是忽略它。"我考虑过那个方案，但[原因]——如果你有强烈感觉，很乐意讨论"让对话保持专业和建设性。同时也可以参见关于<a href="/blog/culture/how-to-review-code-kindly">如何善意地审查代码</a>——审查者对同一对话的视角。</li>
  <li><strong>明确地重新请求审查。</strong>解决反馈后，不要等审查者自己注意到。重新请求审查，并留下简短摘要："解决了所有评论。主要变更：提取了工具函数，添加了两个测试，因为 Y 原因保留了 X 的原始方案。"</li>
</ul>

<h2 id="by-team-size">PR 规范如何随团队规模变化</h2>
<p>好 PR 的标准不是固定的——随着团队成长和代码库成熟而变化。以下是期望如何演进的：</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>团队背景</th><th>最重要的是什么</th><th>实用规范</th></tr>
  </thead>
  <tbody>
    <tr>
      <th>独立 / 自由职业</th>
      <td>未来的你是审查者。PR 是六个月后你回看代码时决策的文档。</td>
      <td>即使没有审查者：一个清晰的标题，描述中简短的"为什么"。让 <code>git log</code> 以后能读懂的提交信息。</td>
    </tr>
    <tr>
      <th>小型团队（2–8 人）</th>
      <td>速度很重要。大家都了解代码库。审查是对话式的。信任度高。</td>
      <td>保持 PR 小而快。描述可以更轻——一两句上下文。当天回转的异步审查是合理的期望。</td>
    </tr>
    <tr>
      <th>中型团队（10–50 人）</th>
      <td>审查者可能不了解完整上下文。新人持续加入。变更跨越团队。</td>
      <td>完整的是什么/为什么/如何描述。UI 工作的截图。测试计划。跨团队变更需要额外关注，可能需要更广泛的审查。</td>
    </tr>
    <tr>
      <th>大型 / 企业（50+ 人）</th>
      <td>审查者有时是陌生人。合规、审计追踪和正确性审查很重要。有回转 SLA。</td>
      <td>工具强制执行的正式 PR 模板。必要的批准。敏感变更的安全审查。设计文档和工单的链接。高风险变更的回滚计划。</td>
    </tr>
  </tbody>
</table>
</div>

<p>四人创业团队不需要每个 PR 描述里都有回滚计划。银行需要。把仪式感与风险匹配，随着团队成长重新审视规范——十个人时有效的做法在五十人时会感觉不够用。</p>

<div class="blog-takeaways">
<h3 id="takeaways">核心要点</h3>
<ul>
  <li><strong>PR 是给人的信息。</strong>为你的审查者而写，而不是为自己。他们的时间和上下文都是有限的。</li>
  <li><strong>保持简小而聚焦。</strong>每个 PR 一个逻辑变更。将重构与行为变更分开。使用功能开关把未完成的工作安全地合并到开关后面。</li>
  <li><strong>标题 + 描述 = 定位。</strong>回答什么发生了变化、为什么需要变化、如何验证。代码展示了什么；描述提供了为什么。</li>
  <li><strong>整洁的提交历史讲述一个故事。</strong>约定式信息，逻辑提交，请求审查之前整齐的 rebase。</li>
  <li><strong>先自审。</strong>在其他人看之前，在 GitHub 上读你自己的 diff。留下引导性评论。自己抓住明显的错误。</li>
  <li><strong>关闭反馈循环。</strong>回复每一条评论，解决备注后重新请求审查，有分歧时给出理由——而不是沉默。</li>
  <li><strong>把仪式感与团队规模匹配。</strong>四人创业团队和 500 人银行需要不同的 PR 规范。随着成长重新审视你们的规范。</li>
