11:T19e1,<p>A QA engineer sits beside a failing checkout test with three tabs open: the acceptance criteria, the pull request diff, and an AI chat window. The feature is not large, but the possible states keep multiplying. New user, returning user, expired coupon, partial refund, slow payment provider, mobile viewport. The question is no longer whether AI can write a test. It is whether it can help the team think through the risk without pretending to understand the product better than the people building it.</p>
<p>AI is useful in QA because testing is partly a search problem. A good tester looks for paths the happy demo does not show. AI can help generate candidate test cases, edge conditions, data combinations, negative flows, accessibility checks, API contract examples, and regression ideas. It can take a short requirement and ask the boring but valuable questions: what happens when the field is empty, duplicated, too long, stale, unauthorized, or changed twice?</p>
<figure class="blog-figure blog-photo-figure">
  <img src="/assets/blog/ai-in-qa-automated-test-generation/candidate-test-map.png" width="1440" height="960" loading="lazy" decoding="async" alt="Phong and a teammate reviewing an AI-generated map of candidate test scenarios across several product flows on shared screens." />
  <figcaption>AI is most useful at the start of QA work when it expands the list of ordinary scenarios a tired team might miss.</figcaption>
</figure>
<p>That breadth is helpful. Many bugs survive because nobody had time to list the ordinary variations. An AI assistant can quickly produce a first map of scenarios for a login flow, billing rule, import screen, or notification workflow. The human reviewer still decides which scenarios matter, which are duplicates, and which are unrealistic. The value is not that the AI is right. The value is that it gives the team more starting points than a tired brain may produce at 5 p.m.</p>
<p>AI can also draft test skeletons. Given a component contract, API schema, or existing test style, it can propose unit tests, integration tests, Playwright flows, fixture data, and assertions. This is especially useful when a codebase already has clear patterns. The assistant can copy the shape of nearby tests and reduce the blank-page cost. But generated tests should be treated like generated code: reviewed, simplified, and questioned.</p>
<p>The risk is false confidence. A test that looks professional can still assert the wrong behavior. It may check implementation details instead of user-visible outcomes. It may mock the exact thing that should have been integrated. It may create data that can never exist in production. It may test the current bug as if the bug were the requirement. AI is very good at producing plausible structure. QA work still needs truth.</p>
<figure class="blog-figure blog-photo-figure">
  <img src="/assets/blog/ai-in-qa-automated-test-generation/generated-test-review.png" width="1440" height="960" loading="lazy" decoding="async" alt="Phong reviewing AI-generated test output on a dark monitor, checking whether the structure is trustworthy before accepting it." />
  <figcaption>Generated tests can look polished long before they prove the right behavior; human review is where confidence is earned.</figcaption>
</figure>
<p>Context quality decides a lot. If the prompt only says write tests for checkout, the output will be generic. If the prompt includes the business rule, user roles, known incidents, API response examples, validation rules, and existing test conventions, the output becomes more useful. The best AI-assisted QA feels less like asking for magic and more like briefing a careful teammate who needs evidence.</p>
<p>One practical workflow is to ask AI for a risk inventory before asking for code. Give it the story, acceptance criteria, and diff. Ask it to list user flows, boundary cases, state transitions, permissions, data integrity risks, and observability questions. Then mark each item as already covered, needs manual testing, needs automation, or not relevant. Only after that should the team ask for test code. This keeps the conversation about quality before syntax.</p>
<figure class="blog-figure blog-photo-figure">
  <img src="/assets/blog/ai-in-qa-automated-test-generation/risk-inventory-board.png" width="1440" height="960" loading="lazy" decoding="async" alt="Phong leading two teammates through a risk inventory board that separates flows, boundary cases, and automation decisions before test code is written." />
  <figcaption>A risk inventory before code keeps the conversation about product truth, not just about how quickly a tool can write syntax.</figcaption>
</figure>
<p>Another useful workflow is test review. Paste a generated or human-written test and ask what behavior it actually protects, what it misses, and whether it would fail for the right reason. This can reveal brittle assertions or over-mocking. It can also help junior engineers learn why one test gives confidence and another only increases line count.</p>
<p>Privacy and security still matter. Production data should not be pasted casually into an AI tool. Customer names, access tokens, private logs, and regulated data need the same care they needed before AI arrived. Synthetic fixtures, anonymized samples, or local models may be better choices depending on the product. A testing assistant should not become a new leak in the quality process.</p>
<p>AI changes the QA role, but not by removing judgment. It moves some effort from typing every scenario by hand toward curating risk, improving prompts, checking evidence, and deciding what deserves automation. The human tester remains the person who understands product trust, customer harm, release timing, and the subtle difference between a test that passes and a product that is safe to ship.</p>
<p>The teams that benefit most will probably be the ones that keep AI close to their existing quality habits. They will feed it real requirements, compare its suggestions against bug history, keep useful prompts, review generated tests like any other code, and measure whether the suite catches more meaningful regressions. The tool becomes part of the loop, not the owner of the loop.</p>
<p>The next time AI suggests twenty tests, it helps to pause before accepting them. Which user risk does each one protect? Which test would have caught the last bug? Which one only checks that the implementation still looks the same? Automated test generation is useful when it expands careful thinking.
