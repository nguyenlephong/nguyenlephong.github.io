12:T6fa9,<p>每个开发者都会在某个时刻碰到同一堵墙。你想为一段代码写测试，却发现代码在内部偷偷调用了 <code>new Database()</code>。无论你怎么做，都会触发一个真实的数据库连接。你想换掉邮件服务商，但邮件 SDK 被埋在三层深处。你想让同一段逻辑既能处理 HTTP 请求又能处理定时任务，但框架被如此紧密地嵌入，几乎不可能实现。</p>
<p>这些都是同一个根本原因的症状：代码<strong>自己创建依赖</strong>。解决方案有一个正式的名字——<em>Dependency Injection</em>——但它背后的思想令人清爽地简单。你停止让代码自己寻找协作者，而是<em>从外部把它们传进来</em>。就这样。其他一切——IoC 容器、DI 框架、injection token——都只是建立在这一个思想之上的机械装置。</p>
<p>本文从零开始拆解这一切：问题、原理、模式，以及在测试中的实际收益。全程使用 TypeScript，因为它对来自任何 Web 或服务端背景的工程师来说都清晰易读。</p>

<h2 id="the-problem">硬连接依赖：代码自己调用 <code>new</code> 时</h2>
<p>这是一个处理订单的 service。乍一看相当合理：</p>
<pre><code><span class="cmt">// order-service.ts — “改造前”版本</span>
<span class="kw">class</span> OrderService {
  <span class="kw">async</span> placeOrder(cart: Cart): Promise&lt;Order&gt; {
    <span class="kw">const</span> db     = <span class="kw">new</span> PostgresClient(process.env.DATABASE_URL!)  <span class="cmt">// 硬连接数据库</span>
    <span class="kw">const</span> stripe = <span class="kw">new</span> Stripe(process.env.STRIPE_SECRET_KEY!)        <span class="cmt">// 硬连接供应商</span>
    <span class="kw">const</span> mailer = <span class="kw">new</span> SendGridMailer(process.env.SENDGRID_KEY!)    <span class="cmt">// 硬连接供应商</span>

    <span class="kw">const</span> order = Order.fromCart(cart)
    <span class="kw">await</span> stripe.charges.create({ amount: order.total, source: cart.token })
    <span class="kw">await</span> db.query(<span class="str">'INSERT INTO orders …'</span>, [order])
    <span class="kw">await</span> mailer.send({ to: cart.email, template: <span class="str">'order-confirmed'</span> })
    <span class="kw">return</span> order
  }
}</code></pre>
<p>一旦你尝试使用它，三个问题就会浮现：</p>
<ul>
  <li><strong>无法孤立测试。</strong>你无法在没有真实凭据、真实数据库和真实 Stripe 沙箱的情况下运行 <code>placeOrder</code> 的单元测试。每个测试默认都是缓慢、脆弱的集成测试。</li>
  <li><strong>与具体实现死死绑定。</strong>如果业务从 Stripe 迁移到 Adyen，你就要打开这个文件做手术。如果 staging 环境需要一个假邮件服务，没有干净的方式插入。</li>
  <li><strong>依赖隐藏不可见。</strong>类签名的读者看到 <code>placeOrder(cart)</code>，完全不知道这个方法背后需要三个外部服务。依赖是不可见的，直到你读完每一行代码。</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">需要注意的模式</span><p>每当你在方法体或构造函数内部——而不是在组合根处——看到 <code>new SomeExternalThing()</code>，那就是一个硬连接依赖。该类既负责完成自己的工作，<em>又</em>负责寻找和构建完成这项工作所需的工具。这是两项工作，它们会朝相反的方向拉扯。</p></div>

<h2 id="inversion-of-control">Inversion of Control：谁来负责创建？</h2>
<p>在用代码解决问题之前，先给这个原理命个名。<strong>Inversion of Control</strong>（IoC）是这样一个思想：一段代码不应该负责获取自己的依赖。相反，某个外部的东西——框架、测试框架、组合根——负责创建并传入这些依赖。</p>
<p>你可能听说过所谓的<em>好莱坞原则</em>：<strong>“不要打电话给我们，我们会打给你。”</strong>在经典的命令式代码中，你的业务逻辑直接调用它的依赖——它主动伸手去拿数据库。有了 IoC，依赖主动来找你。那段关系的控制权被倒转了：你不再决定何时或如何获取你的协作者；外部世界负责决定并传递给你。</p>
<p>IoC 是一个宽泛的原理，而不是某种具体的技术。框架以多种方式实现它：</p>
<ul>
  <li><strong>Dependency Injection（DI）</strong>——把协作者作为构造函数参数或方法参数传入。最常见、最明确的形式。</li>
  <li><strong>Service Locator</strong>——一个全局注册表，代码在运行时通过名称检索其依赖。（稍后会讲为什么这是反模式。）</li>
  <li><strong>Template Method</strong>——基类调用子类覆写的钩子方法，基类控制流程，子类填写空白。</li>
  <li><strong>Event / Observer</strong>——组件发布事件；框架将它们路由给订阅者。两侧互不相知。</li>
</ul>
<p>在这些方式中，<strong>Dependency Injection 是最透明、最可测试的</strong>。它也是最常与 IoC 本身混淆的一种，这就是为什么这两个术语值得区分：IoC 是<em>原理</em>，DI 是该原理的一种<em>实现</em>。</p>

<h2 id="dependency-injection">Dependency Injection：把依赖传进来</h2>
<p>DI 的机制一旦你看清楚就简单得令人难为情。你不在类内部构造协作者，而是把它们声明为参数，让调用者来提供。</p>
<p>有三种形式——但在大多数情况下只有一个明确的赢家：</p>
<h3 id="constructor-injection">构造函数注入（首选）</h3>
<p>依赖在构造函数中声明。这是标准选择：每个依赖在类签名中可见，在构造时必须提供，并且可以标记为 <code>readonly</code>，防止后续被重新赋值。</p>
<pre><code><span class="cmt">// order-service.ts — 使用构造函数注入的“改造后”版本</span>
<span class="kw">interface</span> PaymentGateway { charge(amount: Money, token: string): Promise&lt;Receipt&gt; }
<span class="kw">interface</span> OrderRepository { save(order: Order): Promise&lt;void&gt; }
<span class="kw">interface</span> Mailer          { send(to: string, template: string): Promise&lt;void&gt; }

<span class="kw">class</span> OrderService {
  <span class="kw">constructor</span>(
    <span class="kw">private readonly</span> payments: PaymentGateway,
    <span class="kw">private readonly</span> orders:   OrderRepository,
    <span class="kw">private readonly</span> mailer:   Mailer,
  ) {}

  <span class="kw">async</span> placeOrder(cart: Cart): Promise&lt;Order&gt; {
    <span class="kw">const</span> order = Order.fromCart(cart)                   <span class="cmt">// 纯业务规则</span>
    <span class="kw">await</span> <span class="kw">this</span>.payments.charge(order.total, cart.token)
    <span class="kw">await</span> <span class="kw">this</span>.orders.save(order)
    <span class="kw">await</span> <span class="kw">this</span>.mailer.send(cart.email, <span class="str">'order-confirmed'</span>)
    <span class="kw">return</span> order
  }
}</code></pre>
<p>大声读出那个构造函数签名：“要创建一个 <code>OrderService</code>，你需要一个支付网关、一个订单仓储和一个邮件服务。”这是一份契约，用普通代码写出，无需读一行方法体就能看到。</p>
<h3 id="setter-injection">Setter / 方法注入（偶尔使用）</h3>
<p>依赖在构造之后通过方法设置。对于可选协作者，或者框架需要分两个阶段构造对象时（比如某些测试框架、某些 IoC 容器）很有用。谨慎使用——它允许对象处于部分配置的状态，这是一个微妙的 bug 来源。</p>
<pre><code><span class="cmt">// setter 注入——只在构造时无法连接时才使用</span>
<span class="kw">class</span> ReportGenerator {
  <span class="kw">private</span> formatter: ReportFormatter = <span class="kw">new</span> DefaultFormatter()

  setFormatter(f: ReportFormatter) { <span class="kw">this</span>.formatter = f }  <span class="cmt">// 可选覆盖</span>

  generate(data: ReportData): string {
    <span class="kw">return this</span>.formatter.format(data)
  }
}</code></pre>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">简单的经验法则</span><p>默认用<strong>构造函数注入</strong>。类<em>始终</em>需要的每个依赖都应该在构造函数里。只有当依赖确实是可选的，或者框架强制要求时，才使用 setter 注入。不要把 setter 注入当作逃避思考依赖图的逃生舱口。</p></div>

<h2 id="composition-root">组合根：把一切连接在一起的那一处</h2>
<p>一旦你通过构造函数注入依赖，一个新问题随之而来：<em>究竟谁来实际调用 <code>new</code> 并把各部分连接在一起？</em>答案是<strong>组合根（composition root）</strong>：应用入口处的一个单一位置，在这里所有真实对象被构造并相互注入。</p>
<p>把它想象成整个代码库中唯一一个允许自由出现 <code>new ConcreteImplementation()</code> 的地方。其他一切都用接口说话。组合根是“抽象系统”与“真实世界”之间的缝合处。</p>
<pre><code><span class="cmt">// composition-root.ts — 真实对象在这里被连接起来的唯一位置</span>
<span class="kw">import</span> { PostgresOrderRepository } <span class="kw">from</span> <span class="str">'./adapters/postgres-order-repository'</span>
<span class="kw">import</span> { StripePaymentGateway }    <span class="kw">from</span> <span class="str">'./adapters/stripe-payment-gateway'</span>
<span class="kw">import</span> { SendGridMailer }          <span class="kw">from</span> <span class="str">'./adapters/sendgrid-mailer'</span>
<span class="kw">import</span> { OrderService }            <span class="kw">from</span> <span class="str">'./domain/order-service'</span>
<span class="kw">import</span> { OrderController }         <span class="kw">from</span> <span class="str">'./http/order-controller'</span>

<span class="kw">export function</span> buildApp() {
  <span class="cmt">// 1. 先构建最底层的依赖（自身没有依赖）</span>
  <span class="kw">const</span> db      = <span class="kw">new</span> PostgresOrderRepository(process.env.DATABASE_URL!)
  <span class="kw">const</span> stripe  = <span class="kw">new</span> StripePaymentGateway(process.env.STRIPE_SECRET_KEY!)
  <span class="kw">const</span> mailer  = <span class="kw">new</span> SendGridMailer(process.env.SENDGRID_KEY!)

  <span class="cmt">// 2. 构建 service，注入其依赖</span>
  <span class="kw">const</span> orderSvc = <span class="kw">new</span> OrderService(stripe, db, mailer)

  <span class="cmt">// 3. 构建交付层，注入 service</span>
  <span class="kw">const</span> orderCtrl = <span class="kw">new</span> OrderController(orderSvc)

  <span class="kw">return</span> orderCtrl
}</code></pre>
<p>注意组合根读起来几乎像一份配方：“准备这些原料，将它们组合成 service，再把 service 放入 controller。”整个代码库中只有一处，你去那里就能理解<em>一切是如何连接的</em>。在引导新工程师入职，或者需要替换某个原料时，这是一笔巨大的财富。</p>
<p>在测试中，你写一个<em>测试专用的组合根</em>——一个用 fake 代替真实 adapter 的微型等价物：</p>
<pre><code><span class="cmt">// 在测试文件中——同一个 service，假的协作者</span>
<span class="kw">const</span> fakePayments = <span class="kw">new</span> InMemoryPaymentGateway()
<span class="kw">const</span> fakeOrders   = <span class="kw">new</span> InMemoryOrderRepository()
<span class="kw">const</span> fakeMailer   = <span class="kw">new</span> NoOpMailer()

<span class="kw">const</span> svc = <span class="kw">new</span> OrderService(fakePayments, fakeOrders, fakeMailer)
<span class="cmt">// 毫秒级运行；无网络，无数据库</span></code></pre>

<h2 id="by-hand-vs-container">手动 DI 与 DI 容器——有什么变化，何时重要</h2>
<p>以上所有都是<em>手动 DI</em>，也叫<em>Pure DI</em>。你自己写连接代码。对于很多项目——尤其是小项目——这是正确的选择：它是普通代码，没有魔法，可以立即调试，新开发者只需读一个文件就能理解整个连接方式。</p>
<p>随着应用增长，手动方式开始让人不舒服。DI 容器（有时也叫 IoC 容器）自动化了依赖的注册和解析，并在此之上增加了生命周期管理。</p>

<p>容器开始物有所值的阈值因项目而异，但粗略的参考：如果你的组合根感觉本身正在成为一个维护负担——你总是在滚动它、忘记添加新的绑定，或者在与对象生命周期打架——那就是应该考虑容器的信号。在此之前，普通版本更容易理解和调试。</p>

<h2 id="testing">测试收益：注入 fake，得到快速测试</h2>
<p>学习 DI 最值得的理由——超越任何架构上的优雅——是它对你的测试套件做的事情。当每个依赖都通过构造函数到来，写测试就意味着写一个用 fake 代替真实 adapter 的微型组合根。没有数据库，没有网络，没有供应商沙箱。测试在毫秒内运行，永远不会因为 staging Stripe 账户触发了速率限制而失败。</p>
<pre><code><span class="cmt">// order-service.test.ts</span>
<span class="kw">class</span> InMemoryPaymentGateway <span class="kw">implements</span> PaymentGateway {
  receipts: Receipt[] = []
  <span class="kw">async</span> charge(amount: Money, token: string): Promise&lt;Receipt&gt; {
    <span class="kw">const</span> r = { id: <span class="str">'fake-receipt'</span>, amount }
    <span class="kw">this</span>.receipts.push(r)
    <span class="kw">return</span> r
  }
}

<span class="kw">class</span> InMemoryOrderRepository <span class="kw">implements</span> OrderRepository {
  store: Order[] = []
  <span class="kw">async</span> save(order: Order) { <span class="kw">this</span>.store.push(order) }
}

<span class="kw">class</span> SpyMailer <span class="kw">implements</span> Mailer {
  sent: string[] = []
  <span class="kw">async</span> send(to: string) { <span class="kw">this</span>.sent.push(to) }
}

<span class="cmt">// 测试本身是纯粹的、快速的、确定性的：</span>
<span class="kw">it</span>(<span class="str">'确认订单并通知客户'</span>, <span class="kw">async</span> () =&gt; {
  <span class="kw">const</span> payments = <span class="kw">new</span> InMemoryPaymentGateway()
  <span class="kw">const</span> repo     = <span class="kw">new</span> InMemoryOrderRepository()
  <span class="kw">const</span> mailer   = <span class="kw">new</span> SpyMailer()

  <span class="kw">const</span> svc   = <span class="kw">new</span> OrderService(payments, repo, mailer)
  <span class="kw">const</span> order = <span class="kw">await</span> svc.placeOrder(testCart)

  expect(payments.receipts).toHaveLength(1)
  expect(repo.store).toContainEqual(order)
  expect(mailer.sent).toContain(testCart.email)
})</code></pre>
<p>这与 <a href="/blog/architecture/ports-and-adapters">Ports &amp; Adapters</a> 的核心洞见是同一个——内存 fake 就是测试时的 adapter，真实的 Stripe 或 Postgres 实现就是生产时的 adapter。DI 是让这种替换成为可能的机制；ports &amp; adapters 是让边界有意图、有明确表达的架构模式。</p>
<p>这里也值得停下来谈谈 SOLID。“D”代表<strong>依赖倒置原则</strong>：高层模块不应该依赖低层模块；两者都应该依赖抽象。上面的接口（<code>PaymentGateway</code>、<code>OrderRepository</code>、<code>Mailer</code>）就是那些抽象。<code>OrderService</code> 是高层模块。<code>StripePaymentGateway</code> 和 <code>PostgresOrderRepository</code> 是低层模块。两者都不直接知道对方的存在——它们在接口处相遇，组合根把它们扣在一起。</p>

<h2 id="pitfalls">反模式：DI 变坏的那些方式</h2>
<p>思想很简单，但有几种经典的方式会让它失效。</p>
<h3 id="service-locator">Service Locator</h3>
<p>Service Locator 是一个全局注册表，代码在运行时通过它检索依赖：</p>
<pre><code><span class="cmt">// 反模式：Service Locator——看起来像 DI，但不是</span>
<span class="kw">class</span> OrderService {
  <span class="kw">async</span> placeOrder(cart: Cart) {
    <span class="kw">const</span> payments = ServiceLocator.get&lt;PaymentGateway&gt;(<span class="str">'PaymentGateway'</span>)
    <span class="kw">const</span> repo     = ServiceLocator.get&lt;OrderRepository&gt;(<span class="str">'OrderRepository'</span>)
    <span class="cmt">// … 继续</span>
  }
}</code></pre>
<p>这比内联 <code>new Stripe()</code> 看起来更干净，但隐藏了同样的问题。依赖在签名中仍然不可见。测试必须在运行前配置一个全局注册表。而且类可以在任何时间点请求<em>任何东西</em>——你失去了让 DI 有价值的那种明确性。</p>
<h3 id="over-injection">过度注入（10 个参数的构造函数）</h3>
<p>当一个构造函数积累了七、八或十个参数，通常是该类做了太多事情的信号——而不是 DI 用错了工具。解决方案是把类拆解成更小、更聚焦的协作者，每个都有简短的依赖列表。构造函数长度是一个有用的健康指标。</p>
<h3 id="hidden-globals">隐藏的全局容器</h3>
<p>在 service 内部直接访问 DI 容器——而不是只在组合根处访问——是换了个名字的 Service Locator 模式。规则很严格：只有组合根与容器对话。service 只与被注入的接口对话。</p>
<h3 id="magic-you-cant-trace">无法追踪的魔法</h3>
<p>有些框架通过装饰器和反射元数据连接依赖，方式上在出错时确实很难追踪。如果你花在阅读容器文档上的时间多于阅读自己的业务逻辑，那就是应该转向更简单方案的信号——或者至少添加一个明文说明主要绑定关系的组合根文档。</p>

<h2 id="by-company-size">按公司规模调整 DI：从个人项目到企业级</h2>
<p>像大多数架构思想一样，DI 随着问题的规模而扩展。目标始终相同——可测试、可变更的代码——但随着团队和代码库的增长，实现这一目标的合适机制也会变化。</p>
<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>公司阶段</th><th>推荐方案</th><th>典型形态</th></tr>
  </thead>

    <tr>
      <th>独立开发者 / 早期创业</th>
      <td>手动 DI，加一个组合根文件。</td>
      <td>一个 <code>buildApp()</code> 函数；也许 10–20 行。写起来快，读起来易。</td>
    </tr>
    <tr>
      <th>小型团队（5–20 名工程师）</th>
      <td>仍然是手动 DI，可能拆分成若干 domain 模块，每个有自己的连接函数。</td>
      <td><code>buildOrderModule()</code>、<code>buildBillingModule()</code>——各自独立，在启动时组合一次。</td>
    </tr>
    <tr>
      <th>成长中的规模化公司（20–100 名工程师）</th>
      <td>轻量容器开始物有所值。考虑 <em>tsyringe</em>、<em>InversifyJS</em>，或框架内置的容器。</td>
      <td>类上的装饰器，每个模块配置一个容器；Web 应用使用请求作用域的 service。</td>
    </tr>
    <tr>
      <th>企业级 / 大型平台</th>
      <td>完整的 DI 框架几乎是必须的。NestJS、Spring Boot 或 ASP.NET Core 都围绕这一思想构建。</td>
      <td>约定优于配置，生命周期钩子，模块导入/导出，多租户作用域。</td>
    </tr>
  </tbody>
</table>

<p>一家金融科技初创公司可能从一个 30 行的组合根开始，连接三个 service。两年后，同一代码库有了 60 个 service：手动连接本身变成了一项维护任务，团队开始使用 NestJS 或 InversifyJS 自动管理依赖图。切换很顺滑，因为<em>底层模式</em>——构造函数注入、接口、清晰的组合根——始终不变。容器只是一个机械助手，不是一个不同的思想。</p>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">实用建议</span><p>从手动 DI 和组合根开始。任何时候当连接文件成为负担，你都可以引入容器，而且迁移是机械性的：你只是把显式的 <code>new</code> 调用替换为注册。你的 domain、service 或测试中没有任何东西需要改变——只有组合根。</p></div>

<div class="blog-takeaways">
<h3 id="takeaways">核心要点</h3>
<ul>
  <li><strong>根本原因：</strong>代码刚硬、难以测试的根本原因是一个类对自己的依赖调用 <code>new</code>。解决方案是把依赖传进来。</li>
  <li><strong>Inversion of Control</strong> 是原理（由别人负责提供你的依赖）。<strong>Dependency Injection</strong> 是实现它最明确、最可测试的方式。</li>
  <li><strong>构造函数注入</strong>是默认选择。依赖在签名中可见，在构造时必须提供，在测试中易于替换。</li>
  <li><strong>组合根</strong>是代码库中真实对象被连接在一起的唯一位置。其他一切通过接口对话。</li>
  <li><strong>手动 DI</strong> 对小型代码库更简单、更易调试。<strong>DI 容器</strong>在组合根本身成为维护负担时才物有所值。</li>
  <li><strong>测试收益立竿见影：</strong>在测试中注入 fake，跳过网络和数据库，在几秒内运行数千个用例。</li>
  <li>这也是 SOLID 中<strong>依赖倒置原则</strong>（“D”）的实践：高层模块和低层模块都依赖抽象，永远不直接依赖对方。</li>
  <li>避免 <strong>Service Locator</strong>（隐藏的全局注册表）、过度注入（十个参数的构造函数），以及在组合根之外访问容器。</li>
</ul>
</div>
