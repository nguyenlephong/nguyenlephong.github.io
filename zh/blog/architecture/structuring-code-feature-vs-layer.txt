<p>本文是对三种主要方式——<strong>按层</strong>、<strong>按功能</strong>和<strong>尖叫式（按领域）</strong>——的实用导览，附上各公司规模下诚实的权衡取舍。没有一个答案在所有情况下都是赢家，但读完之后你会有一个清晰的思维模型，能够有意识地选择，而不是随波逐流。</p>

<h2 id="why-it-matters">为什么文件夹结构是架构决策，而不是无聊的细枝末节</h2>
<p>人们把文件夹结构当作“只是组织方式”来忽视。它远不止于此。你对代码的分组方式控制着三件事，而这三件事事后证明极为重要：</p>
<ul>
  <li><strong>耦合。</strong>当一起变化的代码相距遥远，每次小的更新都会变成一场跨文件夹的远征。当它们住在一起，相关的改动就聚集在一处。</li>
  <li><strong>可发现性。</strong>不熟悉代码库的开发者会问：“billing 的代码在哪里？”这个问题的答案——一个文件夹，还是五个散落的文件夹——决定了入职所需的时间。</li>
  <li><strong>影响范围。</strong>你需要打开多少文件才能发布一个新功能，或者安全地删除一个旧功能？结构要么约束这个影响范围，要么让它无边蔓延。</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">思维模型</span><p>把你的文件夹结构想象成一张<strong>地图</strong>。好的地图按照事物在现实世界中的关联方式分组——而不是按照构成它们的材料。按类型（controllers、models）分组文件，就像按形状对地图图标排序。按功能分组，就像按街区排序。其中一种帮你导航；另一种技术上正确，实际毫无用处。</p></div>
<p>康威定律——软件系统反映构建它的团队的沟通结构——在这里同样适用。如果你的团队围绕业务能力组织（支付小队、订单小队），按功能的结构会感觉自然，并强化正确的边界。按层的结构会与你对抗。</p>

<h2 id="by-layer">按层：熟悉的默认方式</h2>
<p>按层组织按照技术角色对文件进行分组。一个典型的 Node 或 Spring Boot 项目最终看起来大致是这样：</p>

<figure class="blog-figure">
  <pre><code>src/
├─ controllers/
│  ├─ OrderController.ts
│  ├─ UserController.ts
│  └─ BillingController.ts
├─ services/
│  ├─ OrderService.ts
│  ├─ UserService.ts
│  └─ BillingService.ts
├─ repositories/
│  ├─ OrderRepository.ts
│  ├─ UserRepository.ts
│  └─ BillingRepository.ts
└─ models/
   ├─ Order.ts
   ├─ User.ts
   └─ Invoice.ts</code></pre>
  <figcaption>按层结构：每个技术层都有自己的顶层文件夹。添加一个功能意味着在<em>全部四个</em>文件夹中添加文件。</figcaption>
</figure>

<p>每个 Web 框架教程都会落到这里。Rails 这样做，Spring Boot 的脚手架这样做，Express 生成器也这样做。这种熟悉感是它的主要资产。</p>
<p><strong>按层的优势：</strong></p>
<ul>
  <li>只有两三个功能的小型应用——结构如此扁平，几乎没什么影响。</li>
  <li>所有人都在所有功能上工作的团队；层是专业化的单元。</li>
  <li>速度比长期清晰度更重要的原型和内部工具。</li>
</ul>
<p><strong>按层的痛点：</strong></p>
<ul>
  <li><em>一个功能散落在多个文件夹里。</em>添加一个“退款”功能意味着要触碰 <code>controllers/</code>、<code>services/</code>、<code>repositories/</code> 和 <code>models/</code>。这是四个文件夹，对应概念上的一件事。</li>
  <li><em>删除一个功能是一项考古工作。</em>你怎么知道 <code>services/</code> 里哪些文件属于 billing，哪些属于 orders？文件夹名称告诉不了你。</li>
  <li><em>合并冲突成倍增加。</em>每次功能变更都会触碰同一层级的文件，当两名开发者并行工作时就会发生碰撞。</li>
</ul>
<div class="blog-callout blog-callout--warn"><span class="blog-callout__label">规模化悬崖</span><p>按层在约 10–15 个功能以内运转良好。超过这个数量，每个顶层文件夹就变成了一个<strong>杂物抽屉</strong>：40 个文件的 <code>services/</code> 文件夹，没有清晰的归属，拉取请求触碰了审查者从未见过的文件。</p></div>

<h2 id="by-feature">按功能：一次变更只在一个文件夹里</h2>
<p>按功能组织按照文件服务的产品切片对其进行分组。技术层（controller、service、model）仍然存在——它们只是<em>住在</em>各自的功能文件夹内部，而不是在顶层：</p>

<figure class="blog-figure">
  <pre><code>src/
├─ orders/
│  ├─ OrderController.ts
│  ├─ OrderService.ts
│  ├─ OrderRepository.ts
│  └─ Order.ts
├─ billing/
│  ├─ BillingController.ts
│  ├─ BillingService.ts
│  ├─ BillingRepository.ts
│  └─ Invoice.ts
├─ auth/
│  ├─ AuthController.ts
│  ├─ AuthService.ts
│  └─ User.ts
└─ shared/
   ├─ database.ts
   ├─ logger.ts
   └─ Money.ts</code></pre>
  <figcaption>按功能结构：每个产品概念拥有自己的文件夹。添加一个退款功能只需添加一个文件夹——其他地方不需要改动。</figcaption>
</figure>

<p><strong>按功能的优势：</strong></p>
<ul>
  <li><em>一次变更住在一个文件夹里。</em>添加退款流程意味着在 <code>billing/</code> 里添加文件——不会触碰其他文件夹。影响范围小且可控。</li>
  <li><em>删除一个功能很简单。</em>删掉文件夹。如果外部还有什么 import 了它，你的 IDE 会立刻告诉你。</li>
  <li><em>归属关系一目了然。</em>支付团队拥有 <code>billing/</code>；身份团队拥有 <code>auth/</code>。Monorepo 工具和代码审查规则可以机械地执行这一点。</li>
  <li><em>入职更快。</em>一个新人负责 orders，只需阅读 <code>orders/</code>。他们不需要先理解整个代码库。</li>
</ul>
<p><strong>按功能需要注意的地方：</strong></p>
<ul>
  <li><em>共享代码的放置很尴尬。</em><code>Money</code> 类型被 <code>orders/</code> 和 <code>billing/</code> 同时使用。它应该放在 <code>orders/</code> 吗？那感觉不对。它最终放到了 <code>shared/</code> 里——而 <code>shared/</code> 本身可能因为没有人真正负责它，而逐渐变成第二个杂物抽屉。</li>
  <li><em>跨功能依赖需要规律。</em>如果 <code>orders/OrderService.ts</code> 直接 import <code>billing/BillingService.ts</code>，你创建了一个文件夹结构看不见的耦合。团队用 lint 规则（<code>eslint-plugin-import/no-restricted-paths</code>）或模块边界工具（Nx、NestJS modules）来让这些显式化。</li>
</ul>

<h2 id="screaming">尖叫式架构：你的顶层说了什么？</h2>
<p>Uncle Bob（Robert C. Martin）在一篇著名的博文中创造了<strong>尖叫式架构</strong>这个说法：“你应用的架构在尖叫什么？当你看顶层目录结构和最高层包里的源文件时；它们在尖叫：<em>医疗系统</em>？还是<em>会计系统</em>？还是<em>库存管理系统</em>？还是在尖叫：<em>Rails</em>？或 <em>Spring/Hibernate</em>？或 <em>ASP.Net</em>？”</p>
<p>这个洞见很优雅：如果你的顶层文件夹是 <code>controllers/</code>、<code>models/</code>、<code>views/</code>，你的代码库在尖叫 <em>MVC 框架</em>。框架是工具——它不应该是读者首先遇到的东西。</p>
<p>尖叫式架构（也叫<strong>按领域</strong>）把按功能推进了一步。不仅功能在顶层——那些功能的名字直接来自<em>业务语言</em>（领域驱动设计的通用语言），而不是技术角色：</p>

<pre><code><span class="cmt">// 尖叫“电商平台”</span>
src/
├─ catalog/          <span class="cmt">// 领域概念：商品目录</span>
├─ ordering/         <span class="cmt">// 领域概念：下单与跟踪订单</span>
├─ fulfillment/      <span class="cmt">// 领域概念：拣货、打包、发货</span>
├─ payments/         <span class="cmt">// 领域概念：收费、退款、发票</span>
└─ identity/         <span class="cmt">// 领域概念：账户、认证、权限</span>

<span class="cmt">// 与“框架优先”的反例相比：</span>
src/
├─ controllers/      <span class="cmt">// 尖叫“MVC”</span>
├─ models/           <span class="cmt">// 尖叫“ORM”</span>
├─ views/            <span class="cmt">// 尖叫“模板引擎”</span>
└─ services/         <span class="cmt">// 尖叫“service 层模式”</span></code></pre>

<p>每个领域文件夹内部的布局由团队决定——这完全没问题。关键的承诺是<em>顶层</em>传达业务，而不是基础设施。一个新开发者（或者你的 CTO，或者业务侧的领域专家）可以读一下顶层文件夹名，就理解系统<em>做什么</em>，而不需要知道它是怎么构建的。</p>

<p>这种混合方案兼得两全：</p>
<ul>
  <li><em>顶层在尖叫你的业务</em>——新来的人立刻知道这是一个电商应用，而不是一个通用的 MVC 骨架。</li>
  <li><em>每个功能自成一体</em>——对 billing 流程的变更只触碰 <code>billing/</code>。</li>
  <li><em>技术层仍然存在于每个文件夹内部</em>，所以用 MVC 或 Clean Architecture 层来思考的开发者不会迷失。他们只需看功能文件夹的内部。</li>
  <li><em><code>shared/</code> 文件夹</em>存放真正横切关注点（<code>Money</code> 值对象、logger、数据库连接）。关键规则：添加到 <code>shared/</code> 的东西应该是有意识地提取并拥有的——不是当逃生舱口扔进去的。</li>
</ul>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">给 shared 文件夹命名</span><p>有些团队叫它 <code>shared/</code>，有些叫 <code>common/</code>、<code>lib/</code> 或 <code>core/</code>。名字没有规律重要：把它当作一个迷你内部库来对待。如果一个文件总是和某个特定功能一起被修改，它不属于 shared——它属于那个功能内部。</p></div>

<h2 id="at-scale">团队规模下：强制边界与 monorepo</h2>
<p>随着代码库增长，多个团队参与贡献，仅靠文件夹约定是不够的——开发者随时可以用一个快速的相对 import 跨越边界。下一步是让边界变得<em>被强制执行</em>：</p>
<ul>
  <li><strong>NestJS modules</strong> 让模块间依赖变得显式：你只能访问另一个模块 <code>exports</code> 的东西。跨越模块边界而没有通过公开 API 的 import 会是一个编译错误。</li>
  <li><strong>Nx 边界规则</strong>让你声明哪些功能或库允许依赖哪些，然后在 CI 中作为 lint 错误来强制执行。</li>
  <li><strong>Monorepo</strong> 走得最远：每个领域（orders、billing、auth）变成自己的包，有自己的 <code>package.json</code>、自己的测试，以及明确的公开导出。Turborepo 和 Nx 是 JavaScript 生态中管理这一切的主要工具。结构看起来不像文件树，更像一个包的图——但底层思想完全相同：按业务能力分组，让边界显式化，机械地执行它们。</li>
</ul>
<p>跳到 monorepo 并不总是必要的。很多团队用单个包加上有规律的混合结构，幸福地运转多年。当跨团队耦合成为真实的痛点时再考虑 monorepo 工具——不要预防性地。</p>

<h2 id="by-company-size">按公司规模的建议</h2>
<p>正确的结构取决于你的组织今天处于哪个阶段。以下是各阶段的直接建议：</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr>

  <tbody>
    <tr>

      <td>Google、Meta 和 Airbnb 运营着严格所有权的大型 monorepo。很多企业走另一条路：每个领域一个微服务，这是尖叫式架构推向逻辑终点的样子。</td>

<div class="blog-callout blog-callout--note"><span class="blog-callout__label">常见错误</span><p>最常见的结构错误不是选错了方式——而是<strong>在错误的方式上待太久</strong>。团队从按层开始，在 20 个功能时感到痛苦，却仍然不重构，因为“结构技术上还能用”。在 15 个功能时花一个下午重组文件夹，能避免在 50 个功能时数月的困惑。</p></div>

<div class="blog-takeaways">
<h3 id="takeaways">核心要点</h3>
<ul>
  <li><strong>结构即架构。</strong>你的文件夹布局控制着耦合、可发现性，以及每次未来变更的影响范围。</li>
  <li><strong>按层</strong>很熟悉，对小型应用没问题，但超过 10–15 个功能后变成杂物抽屉。</li>
  <li><strong>按功能</strong>让变更保持局部。一个新功能触碰一个文件夹。删除一个功能意味着删除一个文件夹。</li>
  <li><strong>尖叫式架构</strong>说顶层应该描述你的<em>业务</em>，而不是你的框架。如果非技术利益相关方能读懂你的文件夹名并理解系统做什么，你就成功了。</li>
  <li><strong>大多数优秀团队落脚的务实混合方案：</strong>顶层是领域/功能名称，每个文件夹内部是小的技术层。</li>
  <li><strong>随着规模扩大强制执行边界。</strong>文件夹约定是建议；模块系统、lint 规则和 monorepo 包让它们变成保证。</li>
  <li><strong>早早重组</strong>——在 15 个功能时花一个下午重组文件夹，避免在 50 个功能时数月的困惑。</li>
</ul>
