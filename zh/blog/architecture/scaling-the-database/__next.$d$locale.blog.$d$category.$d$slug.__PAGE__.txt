e:T6158,<p>App servers are the easy part of scaling. They hold no state, so when traffic doubles you clone them behind a load balancer and walk away. The database is the hard part, because it holds the <em>state</em> — the one thing you cannot simply duplicate and forget. That is why, in almost every growing system, <strong>the database is the first thing to buckle.</strong></p>
<p>It is also the first thing engineers over-engineer in a panic. The instinct under load is to reach for the most dramatic move on the menu — &ldquo;let's shard,&rdquo; &ldquo;let's go NoSQL,&rdquo; &ldquo;let's split into microservices.&rdquo; Almost always the wrong order. There is a ladder here, with the cheapest and least invasive moves at the bottom, and the right answer is to climb it one rung at a time — only stepping up when the rung below genuinely runs out.</p>

<h2 id="rung-0">Rung 0: Measure first — it's probably one query</h2>
<p>Before you scale anything, find out <em>why</em> it is slow. Nine times out of ten, early on, the answer is not &ldquo;the database is too small.&rdquo; It is one missing index, one <code>N+1</code> loop firing a hundred queries to render a page, or one innocent-looking query doing a full table scan across a million rows.</p>
<p>An <strong>index</strong> is the single highest-leverage, cheapest thing in this entire article. The right one can turn a two-second query into a two-millisecond one without adding a single server — because it lets the database jump straight to the rows it needs instead of reading the whole table.</p>
<pre><code><span class="cmt">-- Ask the database what it is actually doing</span>
<span class="kw">EXPLAIN ANALYZE</span> <span class="kw">SELECT</span> * <span class="kw">FROM</span> orders <span class="kw">WHERE</span> customer_id = <span class="str">42</span>;
<span class="cmt">-- "Seq Scan on orders ... rows=2,000,000"  ← reading everything</span>

<span class="cmt">-- One line that changes everything</span>
<span class="kw">CREATE INDEX</span> idx_orders_customer <span class="kw">ON</span> orders (customer_id);
<span class="cmt">-- now: "Index Scan ... rows=37"  ← jumps straight to the matches</span></code></pre>
<div class="blog-callout blog-callout--warn"><span class="blog-callout__label">The most expensive mistake on this page</span><p>Throwing hardware at a missing index. You will pay a bigger server bill every month, forever, to avoid writing a one-line migration once. Always reach for <code>EXPLAIN</code> before you reach for your wallet.</p></div>

<h2 id="rung-1">Rung 1: Scale up before you scale out</h2>
<p>When the box itself is genuinely the limit — CPU pegged, RAM exhausted, disk I/O saturated — the next move is the boring one: <strong>vertical scaling</strong>. Give the database a bigger machine. More RAM (so more of your data lives in memory), faster disks, more cores.</p>
<p>It is unglamorous, and that is exactly the point. No code changes, no new failure modes, no new concepts for the team to learn. It buys you time and headroom while you do something smarter.</p>
<p>It does have three honest limits. There is a <em>ceiling</em> — the biggest instance money can rent. There is a <em>cost cliff</em> — pricing turns sharply super-linear at the top end, so the last doubling costs far more than the first. And it remains a <em>single point of failure</em>: one box, and rebooting it for that upgrade is downtime. But do not let those scare you off prematurely. The largest managed instances today are enormous; a single well-tuned database serves more traffic than the vast majority of companies will ever have. Do not distribute until you have actually hit the wall.</p>

<h2 id="rung-2">Rung 2: The read/write asymmetry → read replicas</h2>
<p>Here is the insight that the next two rungs are built on: <strong>almost every application reads far, far more than it writes.</strong> A social feed, a product catalogue, a news site — they might serve a hundred or a thousand reads for every write. Your write load is tiny; your read load is the firehose.</p>
<p>So stop making one machine do both. Keep a single <strong>primary</strong> that takes every write, and add <strong>read replicas</strong> — copies that continuously stream the primary's data and serve read queries. Now you can add replicas to soak up read traffic almost linearly, and you get a failover candidate for free if the primary dies.</p>

<p>The catch has a name: <strong>replication lag</strong>. Replication is asynchronous, so a replica is always a little behind — usually milliseconds, occasionally seconds when it's busy. This is <a href="/blog/architecture/data-in-distributed-systems">eventual consistency</a> arriving in your own database, and it produces a classic bug: a user posts a comment (write → primary), the page reloads and reads a replica that hasn't caught up yet, and their comment appears to vanish. They post it again. Now there are two.</p>
<figure class="blog-figure blog-photo-figure">
  <img src="https://nguyenlephong.github.io/dom-pub/icdn/blogs/scaling-the-database/read-replica-lag-review.webp" width="1440" height="960" loading="lazy" decoding="async" alt="A Vietnamese backend team reviews one primary database and two read replicas on a whiteboard while deciding which reads must stay fresh." />
  <figcaption>Read replicas help only when the team is explicit about which requests can tolerate a little staleness and which ones must go back to the primary.</figcaption>
</figure>

<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">Living with the lag</span><p>Route reads that <em>must</em> see the latest write back to the primary — &ldquo;read-your-own-writes&rdquo; for the few flows that need it (a user looking at the thing they just changed). Let everything else — search, feeds, dashboards, analytics — happily read a slightly stale replica. The skill is deciding, per query, whether a one-second-old answer is fine. Usually it is.</p></div>

<h2 id="rung-3">Rung 3: Don't hit the database at all → caching</h2>
<p>The fastest query is the one you never run. For data that is read often, expensive to compute, and changes rarely, put a <strong>cache</strong> (Redis, Memcached) in front of the database and answer from memory.</p>
<p>The everyday pattern is <strong>cache-aside</strong>: look in the cache first; on a miss, read the database, store the result, and return it.</p>
<pre><code><span class="kw">async function</span> getProduct(id) {
  <span class="kw">const</span> hit = <span class="kw">await</span> cache.get(<span class="str">`product:${id}`</span>)
  <span class="kw">if</span> (hit) <span class="kw">return</span> JSON.parse(hit)        <span class="cmt">// fast path: served from memory</span>

  <span class="kw">const</span> row = <span class="kw">await</span> db.products.findById(id)  <span class="cmt">// miss: pay for the query once</span>
  <span class="kw">await</span> cache.set(<span class="str">`product:${id}`</span>, JSON.stringify(row), { ttl: <span class="str">300</span> })
  <span class="kw">return</span> row
}</code></pre>
<p>Caching is the highest reward-to-effort move on the ladder — and the one with the sharpest edges. There's an old joke that the two hardest problems in computer science are cache invalidation and naming things. The joke is about the first one. The moment the underlying data changes, your cache is <em>wrong</em>, and deciding when and how to refresh it (expire on a timer? delete on write? both?) is the real work. A second trap is the <strong>cache stampede</strong>: a hot key expires, a thousand requests all miss at once, and they all hammer the database together — the exact spike the cache existed to prevent.</p>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">A cache is an optimisation, never the truth</span><p>The database is the source of record; the cache is a fast, disposable opinion of it. Design so that if the entire cache vanished this second, you'd serve slower for a few minutes — not lose data or serve corruption. If losing the cache loses data, you've built a fragile database, not a cache.</p></div>

<h2 id="rung-4">Rung 4: When one box can't hold the writes → partitioning &amp; sharding</h2>
<p>Everything so far scales <em>reads</em> and keeps a single primary for <em>writes</em>. Eventually — and for most companies this is much later than they fear — the write volume or the sheer size of the data outgrows one machine. Now you have no choice but to split the data itself. There are two shapes.</p>
<p><strong>Vertical partitioning</strong> splits by columns or tables: move a rarely-used giant <code>blob</code> column, or a whole table, onto its own database. Taken to its logical end across a business, this is just <a href="/blog/architecture/data-in-distributed-systems">database-per-service</a> — each service owning its slice.</p>
<p><strong>Horizontal partitioning</strong>, better known as <strong>sharding</strong>, splits the <em>rows</em> of one big table across many databases by a <strong>shard key</strong> — <code>user_id</code> by hash, customer by region, order by date range. Each shard is a normal database holding its slice of the rows.</p>

<p>The shard key is <strong>the most important and most permanent decision in this whole article.</strong> A good key does two things: it spreads load evenly (so you don't get a &ldquo;hot shard&rdquo; melting while the others idle), and it aligns with your most common query (so that query lands on a single shard). Get it wrong and you live with the consequences for years.</p>
<figure class="blog-figure blog-photo-figure">

  <figcaption>Sharding gets expensive long before the migration starts; the real pressure is choosing a key that spreads load without breaking the questions the product asks every day.</figcaption>
</figure>

<p>And the bill is steep, which is exactly why this is the top rung:</p>
<ul>
<li><strong>Cross-shard queries hurt.</strong> &ldquo;Top 10 orders across all customers&rdquo; now means asking every shard and merging the results — a scatter-gather that is slow and fiddly.</li>
<li><strong>Cross-shard transactions are gone.</strong> You're back to <a href="/blog/architecture/data-in-distributed-systems">sagas and eventual consistency</a> — the distributed-data problem, now inside your storage layer.</li>
<li><strong>Resharding is a real project.</strong> Splitting a shard that grew too hot, or changing the key, means migrating live data without downtime. Teams plan for this months ahead.</li>
</ul>

<h2 id="sql-vs-nosql">A note on SQL vs NoSQL: it's about access patterns, not &ldquo;web scale&rdquo;</h2>
<p>Somewhere along the way someone will say you need to &ldquo;switch to NoSQL to scale.&rdquo; Usually that's a myth. A relational database, climbed up the ladder above, scales to a genuinely enormous size — companies you've heard of run staggering load on boring, sharded Postgres or MySQL.</p>
<p>You reach for a different store when the <em>access pattern</em> is genuinely different, not because relational &ldquo;doesn't scale.&rdquo; A key-value store for massive simple-lookup throughput; a document store for flexible, nested schemas; a search engine (Elasticsearch) for full-text; a time-series database for metrics; a wide-column store (Cassandra) for write-heavy workloads with a known query shape. Each is a precision tool for a pattern relational SQL serves awkwardly.</p>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">The honest default</span><p>Stay on your boring relational database far longer than the internet suggests. &ldquo;Polyglot persistence&rdquo; — the right store for each job — is a real and good idea at scale, but every new datastore is one more thing to operate, back up, monitor, and reason about consistency across. Add the second database when a specific pattern demands it, not because one sounds more advanced.</p></div>

<h2 id="decision-table">Climb the ladder in order</h2>
<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>The symptom you actually have</th><th>The cheapest fix that works</th></tr>
  </thead>
  <tbody>
    <tr><th>One endpoint is slow</th><td><code>EXPLAIN</code> it — add the missing index, kill the N+1</td></tr>
    <tr><th>The whole DB is near its CPU/RAM limit</th><td>Scale up to a bigger instance first</td></tr>
    <tr><th>Reads are the bottleneck, writes are fine</th><td>Read replicas — and route must-be-fresh reads to the primary</td></tr>
    <tr><th>The same expensive reads, over and over</th><td>A cache in front (cache-aside + TTL)</td></tr>
    <tr><th>Write volume or data size exceeds one box</th><td>Partition, then shard by a carefully chosen key</td></tr>
    <tr><th>A specific pattern fights the relational model</th><td>Add a purpose-built store — and accept operating it</td></tr>
  </tbody>
</table>
</div>

<h2 id="by-company-size">The honest view by company size</h2>
<ul>
<li><strong>Solo / early startup.</strong> One database, well-indexed, backed up regularly, restore tested once. That is your entire scaling strategy, and it is the correct one. The biggest mistake at this stage is sharding for a load you don't have — a single managed Postgres serves more traffic than 99% of startups will ever reach.</li>
<li><strong>Growing scale-up.</strong> Add a read replica when reads dominate (and for failover peace of mind). Put a cache on the few obvious hot paths. Scale the primary up when you must. Resist sharding until write volume genuinely forces your hand — and when it does, spend real time choosing the shard key. Measure relentlessly; let data, not fear, move you up a rung.</li>
<li><strong>Enterprise.</strong> Replicas, caching tiers, and sharding (or a managed distributed-SQL database that shards for you) are normal furniture. The investment shifts from &ldquo;which rung&rdquo; to operational discipline: shard-key governance, cache invalidation as a platform concern, automated replica failover, and capacity planning. Polyglot persistence here is deliberate and owned — not the accidental sprawl of a dozen teams each picking a favourite database.</li>
</ul>

<div class="blog-takeaways">
<h3 id="takeaways">Key takeaways</h3>
<ul>
  <li><strong>The database falls over first because it holds state you can't just clone.</strong> Climb the ladder; don't leap to the top.</li>
  <li><strong>Measure before you scale.</strong> An index is the cheapest, highest-leverage fix on the whole page — reach for <code>EXPLAIN</code> before hardware.</li>
  <li><strong>Scale up before you scale out.</strong> One well-tuned box goes further than most teams believe.</li>
  <li><strong>Read replicas exploit the read/write asymmetry — but bring replication lag.</strong> Route reads that must see the latest write to the primary; let the rest read slightly stale.</li>
  <li><strong>Caching makes the fastest query the one you never run.</strong> Invalidation is the hard part, and the cache is never the source of truth.</li>
  <li><strong>Sharding is the top rung because the shard key is hard to undo.</strong> Reach for it only when writes or size truly exceed one machine — and choose the key like it's permanent, because it is.</li>
</ul>
</div>
<p>Scaling the data layer is mostly the discipline of <em>not</em> doing the exciting thing first. Together with splitting code into services, deciding <a href="/blog/architecture/data-in-distributed-systems">who owns the data</a>, and paying the <a href="/blog/architecture/resilience-in-distributed-systems">resilience tax</a>, it's the last quiet piece of keeping a system standing as it grows — one cheap, reversible step at a time, for as long as that step keeps working.</p>
