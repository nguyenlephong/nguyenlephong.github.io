19:T41f0,<p>You have probably seen the pull request already. It arrives late in the afternoon with a friendly note: <em>AI helped with most of this, tests are green.</em> The diff is large. The naming is mostly fine. The code compiles. The unit tests pass. The branch deploys to staging. Nothing looks obviously broken.</p>
<p>And still, while reviewing it, something in your stomach stays tense. You cannot quite explain the authorization edge case. You are not sure why the cache TTL is five minutes. The fallback path looks reasonable, but no one remembers whether a revoked user is allowed to keep access during propagation. The code may be correct. The problem is that the team does not understand it well enough to be confident.</p>
<figure class="blog-figure blog-photo-panorama">
  <img src="/assets/blog/cognitive-debt-ai-assisted-development/green-tests-still-need-understanding.png" width="1600" height="900" loading="lazy" decoding="async" alt="Nguyen Le Phong and a teammate study an AI-assisted pull request and flow diagram on a large monitor while late-afternoon light fills the review room." />
  <figcaption>A green test suite does not remove the moment when a reviewer realizes the team still cannot explain the change with enough confidence.</figcaption>
</figure>
<p>That is the shape of <strong>cognitive debt</strong> in AI-assisted development. It is not simply bad code. It is the gap between the system that exists and the shared understanding the team has about how to change, debug, extend, and operate it safely.</p>
<h2 id="what-cognitive-debt-is">What cognitive debt is</h2>
<p>Technical debt is familiar to most engineers. We borrow against the future when we accept duplication, tight coupling, thin tests, rushed design, or an abstraction that will need to be cleaned later. The debt lives mostly in the code and architecture.</p>
<p>Cognitive debt lives in people. It appears when the team no longer carries a reliable mental model of the system. Code may be typed, formatted, tested, and merged, but the team cannot explain why a flow exists, which invariant protects it, what trade-off was chosen, or what might break if one module changes.</p>
<p>Margaret-Anne Storey has recently framed this as part of a broader <a href="https://arxiv.org/abs/2603.22106">triple debt model</a>: technical debt in code, cognitive debt in shared understanding, and intent debt in the externalized rationale that people and AI agents need later. That distinction matters. A codebase can look clean and still be hard to change if the reasoning behind it has disappeared.</p>
<div class="blog-table-wrap">
<table>
<thead><tr><th width="24%">Debt type</th><th>Where it lives</th><th>The broken question</th></tr></thead>
<tbody>
<tr><td><strong>Technical debt</strong></td><td>Code, design, architecture</td><td>Is this easy to change?</td></tr>
<tr><td><strong>Cognitive debt</strong></td><td>The team's mental model</td><td>Do we understand this well enough to change it safely?</td></tr>
<tr><td><strong>Intent debt</strong></td><td>Decision records, specs, docs, prompts, rationale</td><td>Do we know why it was designed this way?</td></tr>
</tbody>
</table>
</div>
<p>The painful part is that cognitive debt rarely fails a build. It does not show up as a TypeScript error or a lint warning. It usually announces itself later, when someone says, <em>do not touch that area, nobody knows how it works.</em></p>
<h2 id="why-ai-makes-it-worse">Why AI makes it worse</h2>
<p>AI changes the speed ratio. Before AI, writing code was often also a way of building the mental model. You discovered the constraints while you typed, hit the edge cases while you tested, and felt the shape of the system while changing it by hand.</p>
<p>With an AI coding assistant or agent, a developer can accept a large patch, a refactor, a SQL query, an integration layer, or a DevOps workflow before building the same depth of understanding. The artifact grows faster than the human model around it. The team gets velocity without always getting comprehension.</p>
<figure class="blog-figure blog-photo-figure">
  <img src="/assets/blog/cognitive-debt-ai-assisted-development/rebuild-the-mental-model-at-night.png" width="1600" height="900" loading="lazy" decoding="async" alt="Nguyen Le Phong sits alone at a home desk at night, comparing code, dependency maps, and handwritten questions as he rebuilds his mental model." />
  <figcaption>Cognitive debt starts to shrink only when someone slows down long enough to turn generated output back into a human mental model.</figcaption>
</figure>
<p>For example, imagine asking an agent to “make payment retries more robust.” It adds exponential backoff, extracts a shared retry helper, catches more provider errors, and updates a few tests. The diff looks like a reliability improvement. But if nobody checks which provider errors are safe to retry, whether every charge carries an idempotency key, and how the partner rate limit behaves under retry storms, the team has not only changed code. It has changed an operational contract it may not understand.</p>
<p>This is why the recent delivery numbers are worth reading carefully. DORA's report on <a href="https://dora.dev/ai/gen-ai-report/">generative AI in software development</a> found that a 25% increase in AI adoption was associated with a 1.5% decrease in delivery throughput and a 7.2% decrease in delivery stability, with larger batch sizes named as one likely mechanism. Sonar's survey on AI coding described a similar verification gap: <a href="https://www.sonarsource.com/company/press-releases/sonar-data-reveals-critical-verification-gap-in-ai-coding/">96% of developers did not fully trust AI-generated code, but only 48% said they always checked it before committing</a>.</p>
<p>Those numbers do not mean AI is bad for software. They mean speed is not the same thing as throughput, and generated code is not the same thing as understood code. If AI moves the bottleneck from writing to verifying, then the team has to redesign the process around verification, not pretend the old review habits still scale.</p>
<h2 id="cognitive-offloading">The human side: cognitive offloading</h2>
<p>There is also a more general cognitive pattern underneath this. Michael Gerlich's 2025 study in <a href="https://www.mdpi.com/2075-4698/15/1/6"><em>Societies</em></a>, based on 666 participants, found a strong relationship between AI tool usage, cognitive offloading, and lower critical thinking scores. The important reading is not that using AI automatically makes people worse thinkers. The more useful reading is that when we use AI to avoid doing the hard thinking ourselves, the thinking muscle receives less training.</p>
<p>Software engineering has its own version of that pattern. If the assistant writes the implementation, the tests, the migration, and the explanation, and the human only accepts the final bundle, the human may not have built the mental model needed for the next incident. The code exists, but the understanding was outsourced.</p>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">A fair distinction</span><p>AI can also reduce cognitive debt when used deliberately: summarizing dependencies, generating test matrices, drafting ADRs, finding missing edge cases, or turning code into a diagram the team can verify. The risk is not AI itself. The risk is using AI only to increase output while shared understanding stays flat.</p></div>
<h2 id="authorization-example">A concrete example: the authorization layer</h2>
<p>Imagine a team asks an AI agent to generate an authorization layer:</p>
<p><code>User -&gt; Role -&gt; Policy -&gt; Resource -&gt; Action</code></p>
<p>The demo works. The middleware is clean. There is a cache. There are basic tests. But nobody writes down the real operating rules:</p>
<ul>
<li>Why can admins bypass some policies?</li>
<li>Why is the cache TTL five minutes?</li>
<li>How long is acceptable between permission revocation and propagation?</li>
<li>Which policies are business-critical and must fail closed?</li>
<li>What must the audit log prove during an incident?</li>
</ul>
<p>Six months later, there is an incident. A user whose access was revoked can still reach a resource for a few minutes. The code is not necessarily ugly. The failure lives in the lost or never-created understanding around the code. The team accepted a working artifact without capturing the intent and invariants that make the artifact safe to operate.</p>
<h2 id="symptoms">Signs your team is carrying cognitive debt</h2>
<p>The signs are usually ordinary at first.</p>
<ul>
<li><strong>Code runs, but nobody wants to touch it.</strong> Every small change requires three people, ten files, and too much manual regression.</li>
<li><strong>PR review becomes a ritual.</strong> Reviewers comment on names, style, and test snapshots, but cannot really judge intent because the patch is too large or too unfamiliar.</li>
<li><strong>Onboarding slows down.</strong> New teammates read the docs, then read the code, then realize each person explains the system differently.</li>
<li><strong>Debugging depends on the person who remembers.</strong> The system has a few human routers. When they are away, delivery stalls.</li>
<li><strong>The team asks AI to explain code no human can verify.</strong> That can help, but it becomes dangerous when the explanation has no tests, telemetry, domain rules, or design notes to check against.</li>
</ul>
<p>The common thread is not that the system is complex. Most useful systems become complex. The problem is that the shared theory of the system is no longer available to the people responsible for it.</p>
<h2 id="slow-down">Slowing down is an engineering control</h2>
<p>It is tempting to answer this with a slogan: slow down. But the point is not nostalgia for hand-written code or suspicion toward tools. The point is more precise: slow down at the moments where understanding is created or lost.</p>
<p>Some friction is not waste. Writing a short design note is friction. Explaining a PR's assumptions is friction. Splitting an AI-generated change into smaller batches is friction. Making a reviewer restate the invariant is friction. But these frictions are the same places where the team rebuilds the mental model that makes future speed possible.</p>
<p>A useful rule for teams is simple: <strong>do not merge code that no one understands well enough to operate.</strong> At least one human should be able to explain what changed, why it changed, what could fail, how to detect failure, and how to roll back. That person does not need to have typed every line. They do need to be accountable for the system becoming understandable again before it ships.</p>
<figure class="blog-figure blog-photo-figure">
  <img src="/assets/blog/cognitive-debt-ai-assisted-development/human-checkpoints-make-the-batch-smaller.png" width="1600" height="900" loading="lazy" decoding="async" alt="Nguyen Le Phong leads two teammates through a board of small review steps, invariants, and rollback paths during a daylight engineering workshop." />
  <figcaption>Smaller batches and human-owned checkpoints are not ceremony; they are how a team makes an AI-assisted change understandable again before it becomes production responsibility.</figcaption>
</figure>
<h2 id="pr-template">A PR template for the AI era</h2>
<p>One practical place to start is the pull request. Do not ask only <em>what changed</em>. Ask for the thinking that lets someone else safely inherit the change.</p>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">A lightweight checklist</span><p>For meaningful AI-assisted changes, add these fields to the PR: Why this change? Key assumptions. New or protected invariants. Failure modes. Rollback path. What AI generated or assisted. What was manually verified.</p></div>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">What this looks like</span><p>For the authorization example, a useful PR note might say: Why: revoked permissions must stop working within 60 seconds. Assumptions: the policy cache may be stale for at most one minute; finance resources must fail closed. Invariants: every access decision writes an audit event. Rollback: disable the cache behind the feature flag. Manual verification: revoke a user in staging, retry the resource, and confirm the audit trail.</p></div>
<p>This is not bureaucracy for its own sake. It is intent capture. It gives future humans and future agents something better than archeology.</p>
<h2 id="team-practices">Six practices that reduce cognitive debt</h2>
<ol>
<li><strong>Limit AI-generated batch size.</strong> If a patch is too large for a reviewer to reason about, it is too large to merge safely. Ask the agent to produce smaller steps.</li>
<li><strong>Treat AI prototypes as disposable.</strong> Use them to explore, then refactor through the architecture, naming, constraints, and test strategy of your system.</li>
<li><strong>Review for understanding, not only correctness.</strong> Ask whether the reviewer can explain the change back. If not, the review is not done.</li>
<li><strong>Externalize intent.</strong> Write ADRs, decision logs, domain glossaries, sequence diagrams, runbooks, threat models, and migration notes when the change is meaningful.</li>
<li><strong>Use AI to repay debt.</strong> Ask it to generate a test matrix, list edge cases, summarize dependencies, compare implementation with a spec, or draft documentation that humans verify.</li>
<li><strong>Protect human checkpoints.</strong> Architecture, public APIs, security boundaries, data migrations, and irreversible operations should remain human-owned decisions, even when AI helps produce the draft.</li>
</ol>
<p>A practical difference: do not ask an agent to “modernize the whole auth module” and merge the result as one impressive diff. Ask it first to add characterization tests around current behavior. Then extract policy lookup. Then add cache behind a flag. Then document the revoke semantics. Each step leaves the reviewer able to explain one idea, instead of pretending to understand five intertwined changes at once.</p>
<h2 id="leader-role">The leader's role: permission to slow down</h2>
<p>The teams most at risk are not always the teams using AI badly. Sometimes they are the teams using AI enthusiastically while leadership quietly rewards only visible output. If the only celebrated number is how much faster code is produced, people will optimize for code volume and hide the cost of understanding.</p>
<p>Leaders need to make a different permission explicit: we are not trying to do everything AI can do. We are trying to do the important work with enough understanding that we can own the result. That means making space for deep work, design review, knowledge sharing, and honest conversations about where AI is helping and where it is creating pressure.</p>
<p>A team can move fast with AI. But it has to move at the speed of verified understanding, not at the speed of generated text.</p>

<h3 id="takeaways">Key takeaways</h3>
<ul>
<li><strong>Cognitive debt is the gap between the system and the team's understanding of it.</strong> Code can compile, tests can pass, and the team can still be unable to change it safely.</li>
<li><strong>Technical debt lives mostly in code; cognitive debt lives in people; intent debt lives in missing rationale.</strong> All three interact in AI-assisted development.</li>
<li><strong>AI increases the risk because generation can outrun comprehension.</strong> The patch arrives faster than the team's mental model can form.</li>
<li><strong>The bottleneck moves from writing to verification.</strong> If review habits do not change, AI can create larger batches that are harder to reason about.</li>
<li><strong>Some friction is healthy.</strong> Design notes, smaller PRs, explicit invariants, and rollback plans are not slowdown for its own sake. They are how shared understanding is rebuilt.</li>
<li><strong>Use AI to repay debt, not only create artifacts.</strong> Let it draft test matrices, dependency maps, runbooks, and ADRs, then make humans verify them.</li>
<li><strong>Do not merge what no one can operate.</strong> At least one human should understand the change deeply enough to debug it at 2 a.m.</li>
</ul>
</div>
<p>The most useful posture toward AI coding is not fear, and it is not surrender. It is stewardship. Let the machine draft, search, compare, and accelerate. But keep the team's understanding close to the system it owns. The future codebase will not be maintained by the prompt that created it.
