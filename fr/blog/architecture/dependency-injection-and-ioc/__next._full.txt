17:T816c,<p>À un moment donné, chaque développeur se heurte au même mur. Vous voulez écrire un test pour un bout de code, mais le code appelle secrètement <code>new Database()</code> en interne. Quoi que vous fassiez, une vraie connexion de base de données se lance. Vous voulez changer de fournisseur d'e-mail, mais le SDK e-mail est enfoui trois couches plus bas. Vous voulez exécuter la même logique pour une requête HTTP et un job planifié, mais le framework est tellement vissé dedans que c'est pratiquement impossible.</p>
<p>Ce sont les symptômes d'une seule cause profonde : un code qui <strong>crée ses propres dépendances</strong>. Le remède a un nom formel — <em>Dependency Injection</em> — mais l'idée qu'il cache est d'une simplicité rafraîchissante. Vous arrêtez de laisser le code trouver ses propres collaborateurs, et vous <em>les lui passez depuis l'extérieur</em>. C'est tout. Tout le reste — conteneurs IoC, frameworks DI, tokens d'injection — n'est que la mécanique construite sur cette seule idée.</p>
<p>Ce guide la démonte depuis zéro : le problème, le principe, le pattern et le bénéfice pratique pour les tests. TypeScript est utilisé tout au long car il se lit clairement pour les ingénieurs venant de n'importe quel contexte web ou serveur.</p>

<h2 id="the-problem">Dépendances codées en dur : quand le code appelle <code>new</code> tout seul</h2>
<p>Voici un service qui traite une commande. Il semble parfaitement raisonnable au premier coup d'œil :</p>
<pre><code><span class="cmt">// order-service.ts — version « avant »</span>
<span class="kw">class</span> OrderService {
  <span class="kw">async</span> placeOrder(cart: Cart): Promise&lt;Order&gt; {
    <span class="kw">const</span> db     = <span class="kw">new</span> PostgresClient(process.env.DATABASE_URL!)  <span class="cmt">// DB codée en dur</span>
    <span class="kw">const</span> stripe = <span class="kw">new</span> Stripe(process.env.STRIPE_SECRET_KEY!)        <span class="cmt">// fournisseur codé en dur</span>
    <span class="kw">const</span> mailer = <span class="kw">new</span> SendGridMailer(process.env.SENDGRID_KEY!)    <span class="cmt">// fournisseur codé en dur</span>

    <span class="kw">const</span> order = Order.fromCart(cart)
    <span class="kw">await</span> stripe.charges.create({ amount: order.total, source: cart.token })
    <span class="kw">await</span> db.query(<span class="str">'INSERT INTO orders …'</span>, [order])
    <span class="kw">await</span> mailer.send({ to: cart.email, template: <span class="str">'order-confirmed'</span> })
    <span class="kw">return</span> order
  }
}</code></pre>
<p>Trois problèmes se cristallisent dès que vous essayez de travailler avec ça :</p>
<ul>
  <li><strong>Impossible à tester de façon isolée.</strong> Vous ne pouvez pas exécuter <code>placeOrder</code> dans un test unitaire sans de vraies credentials, une base de données active et un sandbox Stripe actif. Par défaut, chaque test est un test d'intégration lent et fragile.</li>
  <li><strong>Couplé rigidement à des implémentations spécifiques.</strong> Si l'entreprise passe de Stripe à Adyen, vous ouvrez ce fichier et faites de la chirurgie. Si le staging a besoin d'un mailer factice, il n'y a pas de moyen propre d'en brancher un.</li>
  <li><strong>Dépendances cachées.</strong> Un lecteur de la signature de la classe voit <code>placeOrder(cart)</code> et n'a aucune idée que la méthode a secrètement besoin de trois services externes. Les dépendances sont invisibles jusqu'à ce que vous lisiez chaque ligne.</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">Le pattern à remarquer</span><p>Chaque fois que vous voyez <code>new UneTrucExterne()</code> <em>à l'intérieur</em> d'un corps de méthode ou de constructeur — et non à la racine de composition — c'est une dépendance codée en dur. La classe est responsable à la fois de son propre travail <em>et</em> de trouver et construire les outils dont elle a besoin. Ce sont deux missions, et elles tirent dans des directions opposées.</p></div>

<h2 id="inversion-of-control">Inversion of Control : qui est responsable de la création ?</h2>
<p>Avant de résoudre le problème avec du code, il est utile de nommer le principe en jeu. L'<strong>Inversion of Control</strong> (IoC) est l'idée qu'un bout de code ne devrait pas être responsable d'obtenir ses propres dépendances. Au contraire, quelque chose d'externe — un framework, un harnais de test, une racine de composition — se charge de créer les choses et de les passer.</p>
<p>Vous avez peut-être entendu parler du soi-disant <em>Principe Hollywood</em> : <strong>« Ne nous appelez pas, nous vous appellerons. »</strong> Dans le code impératif classique, votre logique métier appelle ses dépendances directement — elle tend la main et attrape une base de données. Avec IoC, la dépendance vient à vous. Le contrôle de cette relation est inversé : vous ne décidez plus quand ni comment obtenir vos collaborateurs ; le monde extérieur décide et les livre.</p>
<p>L'IoC est un principe large, pas une technique spécifique. Les frameworks l'implémentent de différentes façons :</p>
<ul>
  <li><strong>Dependency Injection (DI)</strong> — passer les collaborateurs comme arguments du constructeur ou paramètres de méthode. La forme la plus courante et la plus explicite.</li>
  <li><strong>Service Locator</strong> — un registre global depuis lequel le code récupère ses dépendances par leur nom. (Plus de détails sur pourquoi c'est un anti-pattern plus loin.)</li>
  <li><strong>Template Method</strong> — une classe de base appelle des méthodes crochet que les sous-classes surchargent, de sorte que la base contrôle le flux et les sous-classes remplissent les blancs.</li>
  <li><strong>Événement / Observateur</strong> — les composants publient des événements ; le framework les route vers les abonnés. Aucun côté ne connaît l'autre.</li>
</ul>
<p>Parmi ceux-ci, la <strong>Dependency Injection est la plus transparente et la plus testable</strong>. C'est aussi celle qu'on confond le plus souvent avec l'IoC elle-même, ce qui explique pourquoi les deux termes méritent d'être distingués : l'IoC est le <em>principe</em>, la DI est une <em>implémentation</em> de ce principe.</p>

<h2 id="dependency-injection">Dependency Injection : passer les dépendances vers l'intérieur</h2>
<p>La mécanique de la DI est presque embarrassante de simplicité une fois qu'on la voit. Au lieu de construire les collaborateurs à l'intérieur de votre classe, vous les déclarez comme paramètres et laissez l'appelant les fournir.</p>
<p>Il existe trois saveurs — mais une gagnante claire pour la plupart des situations :</p>
<h3 id="constructor-injection">Injection par constructeur (recommandée)</h3>
<p>Les dépendances sont déclarées dans le constructeur. C'est le choix standard : chaque dépendance est visible dans la signature de la classe, requise au moment de la construction, et peut être rendue <code>readonly</code> pour qu'rien ne puisse la réassigner plus tard.</p>
<pre><code><span class="cmt">// order-service.ts — version « après » avec injection par constructeur</span>
<span class="kw">interface</span> PaymentGateway { charge(amount: Money, token: string): Promise&lt;Receipt&gt; }
<span class="kw">interface</span> OrderRepository { save(order: Order): Promise&lt;void&gt; }
<span class="kw">interface</span> Mailer          { send(to: string, template: string): Promise&lt;void&gt; }

<span class="kw">class</span> OrderService {
  <span class="kw">constructor</span>(
    <span class="kw">private readonly</span> payments: PaymentGateway,
    <span class="kw">private readonly</span> orders:   OrderRepository,
    <span class="kw">private readonly</span> mailer:   Mailer,
  ) {}

  <span class="kw">async</span> placeOrder(cart: Cart): Promise&lt;Order&gt; {
    <span class="kw">const</span> order = Order.fromCart(cart)                   <span class="cmt">// règle métier pure</span>
    <span class="kw">await</span> <span class="kw">this</span>.payments.charge(order.total, cart.token)
    <span class="kw">await</span> <span class="kw">this</span>.orders.save(order)
    <span class="kw">await</span> <span class="kw">this</span>.mailer.send(cart.email, <span class="str">'order-confirmed'</span>)
    <span class="kw">return</span> order
  }
}</code></pre>
<p>Lisez cette signature de constructeur à voix haute : « Pour créer un <code>OrderService</code>, vous avez besoin d'une passerelle de paiement, d'un dépôt de commandes et d'un mailer. » C'est un contrat, écrit en code simple, visible sans avoir à lire une seule ligne de méthode.</p>
<h3 id="setter-injection">Injection par setter / méthode (usage occasionnel)</h3>
<p>Les dépendances sont définies via une méthode après la construction. Utile pour les collaborateurs optionnels ou pour les frameworks qui construisent les objets en deux phases (par exemple certains frameworks de test, certains conteneurs IoC). À utiliser avec parcimonie — cela permet à un objet d'exister dans un état partiellement configuré, ce qui est une source subtile de bugs.</p>
<pre><code><span class="cmt">// injection par setter — à utiliser seulement quand le câblage à la construction est impossible</span>
<span class="kw">class</span> ReportGenerator {
  <span class="kw">private</span> formatter: ReportFormatter = <span class="kw">new</span> DefaultFormatter()

  setFormatter(f: ReportFormatter) { <span class="kw">this</span>.formatter = f }  <span class="cmt">// surcharge optionnelle</span>

  generate(data: ReportData): string {
    <span class="kw">return this</span>.formatter.format(data)
  }
}</code></pre>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">Règle de base simple</span><p>Préférez <strong>l'injection par constructeur par défaut</strong>. Chaque dépendance dont la classe a <em>toujours</em> besoin appartient au constructeur. Ne recourez à l'injection par setter que quand une dépendance est réellement optionnelle, ou quand un framework l'impose. N'utilisez jamais l'injection par setter comme échappatoire pour éviter de réfléchir à votre graphe de dépendances.</p></div>

<h2 id="composition-root">La racine de composition : un seul endroit pour tout câbler</h2>
<p>Une fois que vous injectez des dépendances par les constructeurs, une nouvelle question se pose : <em>qui appelle réellement <code>new</code> et câble les pièces ensemble ?</em> La réponse est la <strong>racine de composition</strong> : un endroit unique au point d'entrée de votre application où tous les objets réels sont construits et injectés les uns dans les autres.</p>
<p>Pensez-y comme le seul endroit dans tout votre codebase où <code>new ImplémantationConcrète()</code> peut apparaître librement. Tout le reste parle en termes d'interfaces. La racine de composition est la couture entre « le système abstrait » et « le monde réel ».</p>
<pre><code><span class="cmt">// composition-root.ts — le seul endroit où les vraies choses sont câblées</span>
<span class="kw">import</span> { PostgresOrderRepository } <span class="kw">from</span> <span class="str">'./adapters/postgres-order-repository'</span>
<span class="kw">import</span> { StripePaymentGateway }    <span class="kw">from</span> <span class="str">'./adapters/stripe-payment-gateway'</span>
<span class="kw">import</span> { SendGridMailer }          <span class="kw">from</span> <span class="str">'./adapters/sendgrid-mailer'</span>
<span class="kw">import</span> { OrderService }            <span class="kw">from</span> <span class="str">'./domain/order-service'</span>
<span class="kw">import</span> { OrderController }         <span class="kw">from</span> <span class="str">'./http/order-controller'</span>

<span class="kw">export function</span> buildApp() {
  <span class="cmt">// 1. Construire d'abord les dépendances feuilles (sans dépendances elles-mêmes)</span>
  <span class="kw">const</span> db      = <span class="kw">new</span> PostgresOrderRepository(process.env.DATABASE_URL!)
  <span class="kw">const</span> stripe  = <span class="kw">new</span> StripePaymentGateway(process.env.STRIPE_SECRET_KEY!)
  <span class="kw">const</span> mailer  = <span class="kw">new</span> SendGridMailer(process.env.SENDGRID_KEY!)

  <span class="cmt">// 2. Construire les services, en injectant leurs dépendances</span>
  <span class="kw">const</span> orderSvc = <span class="kw">new</span> OrderService(stripe, db, mailer)

  <span class="cmt">// 3. Construire la couche de livraison, en injectant le service</span>
  <span class="kw">const</span> orderCtrl = <span class="kw">new</span> OrderController(orderSvc)

  <span class="kw">return</span> orderCtrl
}</code></pre>
<p>Remarquez comment la racine de composition se lit presque comme une recette : « préparez ces ingrédients, assemblez-les en services, mettez les services dans les contrôleurs. » Il y a un seul endroit dans tout le codebase où vous allez pour comprendre <em>comment tout se connecte</em>. C'est un atout considérable pour intégrer un nouvel ingénieur, ou quand vous devez échanger un ingrédient.</p>
<p>Dans un test, vous écrivez une <em>racine de composition spécifique au test</em> — un équivalent minuscule qui câble des faux à la place des vrais adapters :</p>
<pre><code><span class="cmt">// dans un fichier de test — même service, collaborateurs factices</span>
<span class="kw">const</span> fakePayments = <span class="kw">new</span> InMemoryPaymentGateway()
<span class="kw">const</span> fakeOrders   = <span class="kw">new</span> InMemoryOrderRepository()
<span class="kw">const</span> fakeMailer   = <span class="kw">new</span> NoOpMailer()

<span class="kw">const</span> svc = <span class="kw">new</span> OrderService(fakePayments, fakeOrders, fakeMailer)
<span class="cmt">// s'exécute en quelques millisecondes ; pas de réseau, pas de base de données</span></code></pre>

<h2 id="by-hand-vs-container">DI manuelle vs conteneur DI — ce qui change et quand ça compte</h2>
<p>Tout ce qui précède est de la <em>DI manuelle</em>, aussi appelée <em>DI Pure</em>. Vous écrivez le code de câblage vous-même. Pour beaucoup de projets — surtout les petits — c'est le bon choix : c'est du code simple, sans magie, immédiatement débogable, et un nouveau développeur peut comprendre tout le câblage en lisant un seul fichier.</p>
<p>À mesure que les applications grandissent, l'approche manuelle commence à faire mal. Un conteneur DI (parfois appelé conteneur IoC) automatise l'enregistrement et la résolution des dépendances, et ajoute par-dessus la gestion des durées de vie.</p>

<p>Le seuil à partir duquel un conteneur rapporte dépend du contexte, mais voici un repère approximatif : si votre racine de composition commence à devenir elle-même un fardeau de maintenance — vous la faites défiler en permanence, vous oubliez d'ajouter un nouveau binding, ou vous vous battez avec les durées de vie des objets — c'est le signal pour vous tourner vers un conteneur. Jusqu'alors, la version simple est plus facile à comprendre et à déboguer.</p>

<h2 id="testing">Le bénéfice pour les tests : injectez des faux, obtenez des tests rapides</h2>
<p>La raison pour laquelle la DI vaut la peine d'être apprise — plus que toute élégance architecturale — c'est ce qu'elle fait à votre suite de tests. Quand chaque dépendance arrive par le constructeur, écrire un test signifie écrire une toute petite racine de composition avec des faux à la place des vrais adapters. Pas de base de données, pas de réseau, pas de sandbox fournisseur. Les tests s'exécutent en quelques millisecondes et n'échouent jamais parce que le compte Stripe de staging a une limite de taux.</p>
<pre><code><span class="cmt">// order-service.test.ts</span>
<span class="kw">class</span> InMemoryPaymentGateway <span class="kw">implements</span> PaymentGateway {
  receipts: Receipt[] = []
  <span class="kw">async</span> charge(amount: Money, token: string): Promise&lt;Receipt&gt; {
    <span class="kw">const</span> r = { id: <span class="str">'fake-receipt'</span>, amount }
    <span class="kw">this</span>.receipts.push(r)
    <span class="kw">return</span> r
  }
}

<span class="kw">class</span> InMemoryOrderRepository <span class="kw">implements</span> OrderRepository {
  store: Order[] = []
  <span class="kw">async</span> save(order: Order) { <span class="kw">this</span>.store.push(order) }
}

<span class="kw">class</span> SpyMailer <span class="kw">implements</span> Mailer {
  sent: string[] = []
  <span class="kw">async</span> send(to: string) { <span class="kw">this</span>.sent.push(to) }
}

<span class="cmt">// Le test lui-même est pur, rapide, déterministe :</span>
<span class="kw">it</span>(<span class="str">'confirme une commande et notifie le client'</span>, <span class="kw">async</span> () =&gt; {
  <span class="kw">const</span> payments = <span class="kw">new</span> InMemoryPaymentGateway()
  <span class="kw">const</span> repo     = <span class="kw">new</span> InMemoryOrderRepository()
  <span class="kw">const</span> mailer   = <span class="kw">new</span> SpyMailer()

  <span class="kw">const</span> svc   = <span class="kw">new</span> OrderService(payments, repo, mailer)
  <span class="kw">const</span> order = <span class="kw">await</span> svc.placeOrder(testCart)

  expect(payments.receipts).toHaveLength(1)
  expect(repo.store).toContainEqual(order)
  expect(mailer.sent).toContain(testCart.email)
})</code></pre>
<p>C'est la même intuition au cœur des <a href="/blog/architecture/ports-and-adapters">Ports &amp; Adapters</a> — le faux en mémoire n'est que l'adapter pour le moment des tests, et la vraie implémentation Stripe ou Postgres est l'adapter de production. La DI est le mécanisme qui rend l'échange possible ; les ports &amp; adapters sont le pattern architectural qui rend la frontière intentionnelle et explicite.</p>
<p>Il vaut aussi la peine de s'arrêter sur SOLID ici. Le « D » représente le <strong>Dependency Inversion Principle</strong> : les modules de haut niveau ne devraient pas dépendre des modules de bas niveau ; les deux devraient dépendre d'abstractions. Les interfaces ci-dessus (<code>PaymentGateway</code>, <code>OrderRepository</code>, <code>Mailer</code>) sont ces abstractions. <code>OrderService</code> est le module de haut niveau. <code>StripePaymentGateway</code> et <code>PostgresOrderRepository</code> sont les modules de bas niveau. Aucun ne connaît directement l'autre — ils se rencontrent à l'interface, et la racine de composition les assemble.</p>

<h2 id="pitfalls">Anti-patterns : les façons dont la DI tourne mal</h2>
<p>L'idée est simple, mais il existe quelques façons bien connues de la saper.</p>
<h3 id="service-locator">Le Service Locator</h3>
<p>Un Service Locator est un registre global que le code appelle pour récupérer ses dépendances à l'exécution :</p>
<pre><code><span class="cmt">// anti-pattern : Service Locator — ressemble à de la DI mais n'en est pas</span>
<span class="kw">class</span> OrderService {
  <span class="kw">async</span> placeOrder(cart: Cart) {
    <span class="kw">const</span> payments = ServiceLocator.get&lt;PaymentGateway&gt;(<span class="str">'PaymentGateway'</span>)
    <span class="kw">const</span> repo     = ServiceLocator.get&lt;OrderRepository&gt;(<span class="str">'OrderRepository'</span>)
    <span class="cmt">// … continue</span>
  }
}</code></pre>
<p>Ça semble plus propre que <code>new Stripe()</code> en ligne, mais ça cache le même problème. Les dépendances sont toujours invisibles dans la signature. Les tests doivent configurer un registre global avant de s'exécuter. Et la classe peut demander <em>n'importe quoi</em> à tout moment — vous avez perdu l'explicité qui rend la DI précieuse.</p>
<h3 id="over-injection">Sur-injection (le constructeur à 10 arguments)</h3>
<p>Quand un constructeur accumule sept, huit ou dix paramètres, c'est généralement le signe que la classe en fait trop — pas que la DI est le mauvais outil. La correction consiste à décomposer la classe en collaborateurs plus petits et plus focalisés, chacun avec une liste de dépendances courte. La longueur du constructeur est un indicateur de santé utile.</p>
<h3 id="hidden-globals">Conteneurs globaux cachés</h3>
<p>Accéder directement au conteneur DI depuis un service — plutôt qu'uniquement depuis la racine de composition — est le pattern Service Locator sous un nom différent. La règle est stricte : seule la racine de composition parle au conteneur. Les services ne parlent qu'à leurs interfaces injectées.</p>
<h3 id="magic-you-cant-trace">Magie impossible à tracer</h3>
<p>Certains frameworks câblent les dépendances via des décorateurs et des métadonnées de réflexion d'une façon qui est réellement difficile à suivre quand quelque chose se casse. Si vous passez plus de temps à lire la documentation du conteneur qu'à lire votre propre logique métier, c'est un signe de recourir à une approche plus simple — ou au minimum d'ajouter une racine de composition bien documentée qui explicite les bindings majeurs.</p>

<h2 id="by-company-size">Calibrer la DI : du projet solo à l'entreprise</h2>
<p>Comme la plupart des idées architecturales, la DI s'adapte au problème. L'objectif reste toujours le même — du code testable et modifiable — mais la bonne mécanique pour y arriver change à mesure que l'équipe et le codebase grandissent.</p>

<table>

<p>Une startup fintech peut commencer avec une racine de composition de 30 lignes qui câble trois services. Deux ans plus tard, le même codebase compte 60 services : le câblage manuel est devenu une tâche de maintenance en soi, et l'équipe se tourne vers NestJS ou InversifyJS pour gérer le graphe automatiquement. La transition est simple parce que le <em>pattern sous-jacent</em> — injection par constructeur, interfaces, racine de composition claire — reste le même. Le conteneur est un assistant mécanique, pas une idée différente.</p>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">Conseil pratique</span><p>Commencez avec la DI manuelle et une racine de composition. Vous pouvez introduire un conteneur à tout moment quand le fichier de câblage devient un fardeau, et la migration est mécanique : vous remplacez les appels <code>new</code> explicites par des enregistrements. Rien dans votre domaine, vos services ou vos tests n'a besoin de changer — seule la racine de composition.</p></div>

<div class="blog-takeaways">
<h3 id="takeaways">Points clés à retenir</h3>
<ul>
  <li><strong>La cause profonde</strong> d'un code rigide et non testable est une classe qui appelle <code>new</code> sur ses propres dépendances. Le remède est de les passer de l'extérieur.</li>
  <li><strong>L'Inversion of Control</strong> est le principe (quelqu'un d'autre est responsable de fournir vos dépendances). La <strong>Dependency Injection</strong> est la façon la plus explicite et la plus testable de l'implémenter.</li>
  <li><strong>L'injection par constructeur</strong> est le choix par défaut. Les dépendances sont visibles dans la signature, requises à la construction et facilement échangeables dans les tests.</li>
  <li><strong>La racine de composition</strong> est l'endroit unique dans votre codebase où les objets réels sont câblés ensemble. Tout le reste parle via des interfaces.</li>
  <li><strong>La DI manuelle</strong> est plus simple et plus débogable pour les petits codebases. Un <strong>conteneur DI</strong> rapporte quand la racine de composition elle-même devient un fardeau de maintenance.</li>
  <li><strong>Le bénéfice pour les tests</strong> est immédiat : injectez des faux dans les tests, évitez le réseau et la base de données, et exécutez des milliers de cas en quelques secondes.</li>
  <li>C'est aussi le <strong>Dependency Inversion Principle</strong> (le « D » dans SOLID) en action : les modules de haut niveau et de bas niveau dépendent tous deux d'abstractions, jamais directement l'un de l'autre.</li>
  <li>Évitez le <strong>Service Locator</strong> (registre global caché), la sur-injection (constructeurs à dix arguments) et l'accès au conteneur en dehors de la racine de composition.</li>
</ul>
