20:T5705,<p>Vous connaissez celle-là. Elle arrive dans votre file de revue un mardi après-midi : une pull request intitulée <strong>« fixes »</strong>, un diff couvrant 2 347 lignes sur 41 fichiers, aucune description, et un joyeux « PTAL » en commentaire. Le cœur vous tombe. Vous l'ouvrez quand même. Vingt minutes plus tard, vous avez lu la même méthode trois fois, vous n'arrivez toujours pas à savoir quel problème elle résout, et vous avez laissé un seul commentaire nerveux : « ça a l'air bien dans l'ensemble, peut-être renommer cette variable ? » Vous venez de valider du code que vous n'avez pas compris.</p>
<p>L'auteur n'était pas malveillant — il était probablement la tête dans le guidon, fier du travail, et a simplement oublié que livrer du code n'est que la moitié du travail. L'autre moitié consiste à le faire réviser, comprendre et merger en toute sécurité. Une pull request n'est pas une formalité à franchir avant de merger ; c'est une conversation, et comme toutes les conversations, elle se passe mieux quand quelqu'un fait l'effort d'être compris.</p>
<p>Cet article porte sur cet effort : comment écrire des PRs que vos coéquipiers prennent vraiment plaisir à réviser — petites, ciblées, faciles à suivre et faciles à accepter.</p>

<h2 id="pr-is-communication">Une PR est de la communication, pas seulement du code</h2>
<p>Avant une seule ligne de diff, une pull request est un message à un autre être humain. Cet être humain a un temps limité, un contexte limité sur ce à quoi vous pensiez à 23h, et une dizaine d'autres choses qui se disputent son attention. Votre travail en tant qu'auteur est de faciliter son travail.</p>
<p>Pensez au relecteur comme à votre audience, pas à votre juge. Il n'est pas là pour vous piéger — c'est un collaborateur qui vous aide à livrer quelque chose de solide. Plus vous l'aidez à comprendre le <em>pourquoi</em> derrière le changement, plus la revue sera rapide et de qualité. Une bonne description de PR est comme une lettre de motivation pour votre code : elle plante le décor, explique la motivation et dit au relecteur exactement quoi regarder.</p>
<p>Ce changement de mentalité change tout. Une fois que vous arrêtez d'écrire des PRs pour vous-même et que vous commencez à les écrire pour votre relecteur, vous voudrez naturellement les rendre plus petites, les décrire mieux et les soigner avant de demander des regards.</p>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">La règle d'or du relecteur</span><p>Si un relecteur doit lire le code pour comprendre <em>pourquoi</em> le changement existe, la description de la PR n'a pas fait son travail. Le code explique <em>ce qui</em> a changé ; la description explique <em>pourquoi c'était nécessaire</em>.</p></div>

<h2 id="keep-it-small">Gardez-la petite et focalisée</h2>
<p>La chose la plus efficace que vous puissiez faire pour améliorer vos pull requests est de les rendre plus petites. Plus une PR est grande, moins elle est révisée minutieusement, plus elle prend de temps à merger, et plus de bugs passent au travers.</p>
<p>Une bonne PR contient <strong>un seul changement logique</strong>. Pas un ticket, pas une fonctionnalité, pas le travail d'un sprint — une idée cohérente qui peut être comprise, testée et annulée indépendamment. « Ajouter l'upload d'avatar utilisateur » est un changement. « Ajouter l'upload d'avatar utilisateur et refactorer le contrôleur de profil et mettre à jour trois dépendances » est trois changements dans un seul manteau.</p>

<h3 id="how-to-split">Comment diviser un grand travail</h3>
<p>Les grandes fonctionnalités n'ont pas besoin d'arriver sous forme de grandes PRs. Quelques stratégies de division pratiques :</p>
<ul>
  <li><strong>Séparez le refactoring des changements de comportement.</strong> Première PR : renommer, restructurer, extraire — pas de nouvelle logique. Deuxième PR : la fonctionnalité réelle. Les relecteurs peuvent vérifier que le refactoring n'a pas d'effets secondaires, puis se concentrer entièrement sur la nouvelle logique.</li>
  <li><strong>Livrez l'infrastructure en premier.</strong> Ajoutez la colonne de base de données, la migration, le type — tout ça sans aucune interface. Puis ajoutez l'interface qui l'utilise. Chaque PR est déployable indépendamment.</li>
  <li><strong>Utilisez les feature flags.</strong> Mergez du travail incomplet derrière un flag désactivé par défaut. La PR est petite et sûre ; la fonctionnalité n'est pas « terminée » avant que vous activiez le flag.</li>
  <li><strong>PRs empilées.</strong> La PR B dépend de la PR A. Certaines équipes utilisent des outils de PR empilées pour réviser chaque couche indépendamment. Si votre équipe n'utilise pas l'empilement, restez simple : terminez et mergez A d'abord, puis ouvrez B.</li>
</ul>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">Une heuristique utile</span><p>Si vous ne pouvez pas décrire votre PR en une seule phrase sans le mot « et », elle contient probablement plus d'un changement logique. Trouvez la couture et divisez.</p></div>

<h2 id="title-and-description">Rédiger un titre et une description qui valent la lecture</h2>
<p>Le titre est le titre de l'article. Il doit dire <em>ce qui</em> a changé et pourquoi ça compte, en une ligne. Évitez les verbes vagues comme « mettre à jour », « corriger des trucs », ou « modifications » et visez la spécificité. Voici des odeurs courantes de titre de PR et comment les corriger :</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>Odeur de PR</th><th>Ce qui ne va pas</th><th>Version améliorée</th></tr>
  </thead>
  <tbody>
    <tr>
      <th>« fixes »</th>
      <td>Aucun contexte whatsoever — le relecteur doit lire chaque ligne pour deviner ce qui se passe.</td>
      <td>« Fix race condition dans le checkout panier lors de sessions concurrentes »</td>
    </tr>
    <tr>
      <th>« WIP : divers changements »</th>
      <td>Signale que l'auteur n'est pas prêt. Les PR en Draft existent pour une raison.</td>
      <td>Ouvrez en tant que Draft GitHub jusqu'à ce que ce soit vraiment prêt pour la revue.</td>
    </tr>
    <tr>
      <th>« Update UserService »</th>
      <td>Décrit où, pas quoi ni pourquoi. Chaque PR met à jour quelque chose.</td>
      <td>« Ajouter le flux de vérification de changement d'email à UserService »</td>
    </tr>
    <tr>
      <th>« JIRA-1234 »</th>
      <td>Force le relecteur à ouvrir un deuxième onglet juste pour comprendre la motivation.</td>
      <td>« Limiter l'API à 100 req/min par IP (JIRA-1234) » — le ticket comme contexte, pas comme substitut.</td>
    </tr>
    <tr>
      <th>« Refactor + nouvelle fonctionnalité + bump deps »</th>
      <td>Trois PRs dans un trench-coat. Chaque préoccupation mérite sa propre revue.</td>

    </tr>
  </tbody>
</table>
</div>

<p>La <strong>description</strong> est là où la vraie communication se passe. Une bonne description répond aux trois questions que tout relecteur se pose dans les dix premières secondes :</p>
<ul>
  <li><strong>Quoi</strong> a changé et où ?</li>
  <li><strong>Pourquoi</strong> ce changement était-il nécessaire — le contexte, le bug, la user story ?</li>
  <li><strong>Comment vérifier</strong> que ça fonctionne — étapes de test, captures d'écran, cas limites concernés ?</li>
</ul>
<p>Voici un avant-après pour rendre ça concret :</p>
<pre><code><span class="cmt">--- AVANT (la PR « fixes » de 2 000 lignes) ---</span>

<span class="kw">Titre :</span> <span class="str">fixes</span>

<span class="kw">Description :</span> (vide)


<span class="cmt">--- APRÈS (le même changement, rédigé pour un relecteur) ---</span>

<span class="kw">Titre :</span> <span class="str">Fix race condition dans le checkout panier sur sessions concurrentes</span>

<span class="kw">## Quoi</span>
Deux requêtes de checkout concurrentes du même utilisateur pouvaient toutes les deux
passer la vérification d'inventaire avant que l'une ou l'autre ne décrémente le
compteur de stock, entraînant des surventes. Cette PR ajoute un verrou de ligne DB
autour du décrément du stock.

<span class="kw">## Pourquoi</span>
Le support client a signalé 12 incidents de survente au cours des 30 derniers jours
(voir Sentry issue #4421). La cause racine est deux requêtes en concurrence entre le
SELECT et l'UPDATE sur la table d'inventaire.

<span class="kw">## Comment tester</span>
1. Lancer le nouveau test de régression de concurrence : npm test -- cart.concurrency
2. En staging : ouvrir deux onglets de navigateur, cliquer sur « Acheter »
   simultanément sur un produit avec stock = 1. Un seul devrait réussir.

<span class="kw">## Risques / notes</span>
- Le verrou est limité à une seule ligne produit ; n'affecte pas le débit
  quand différents produits passent en caisse en même temps.
- Pas de migration nécessaire — utilise SELECT FOR UPDATE sur la table existante.</code></pre>

<h2 id="commit-hygiene">Hygiène des commits</h2>
<p>La façon dont vous organisez vos commits compte — pas seulement pour satisfaire le processus, mais parce que les commits sont la trace permanente de comment un changement est venu à exister. Dans un an, quelqu'un (probablement vous) lancera <code>git blame</code> sur une ligne et suivra le hash de commit dans l'historique. Ce qu'il trouvera devrait raconter une histoire, pas ressembler à un flux de conscience.</p>
<p>Un bon message de commit suit le format conventionnel : une ligne de sujet courte à l'impératif (50 caractères ou moins), une ligne vide optionnelle, et un corps optionnel expliquant <em>pourquoi</em> — pas ce que le diff montre déjà.</p>
<pre><code><span class="cmt"># Le format conventionnel :</span>
<span class="kw">type(scope): sujet impératif court</span>

<span class="str">Corps optionnel : pourquoi ce changement a été fait, quel problème il résout,
les décisions non évidentes et le raisonnement derrière elles.</span>

<span class="cmt"># Exemples réels :</span>
<span class="kw">fix(cart):</span> add row-level lock to prevent oversell on concurrent checkout

<span class="kw">feat(auth):</span> add email-change verification flow

<span class="kw">refactor(profile):</span> extract avatar upload to dedicated service</code></pre>
<p>Au-delà du format du message, pensez à la <em>forme</em> des commits. Chaque commit devrait représenter une étape cohérente — un refactoring, un test, un nouveau comportement. Un relecteur qui regarde les commits individuels (plutôt que le diff entier) devrait pouvoir suivre la progression logique : « extraire la méthode → ajouter un test → implémenter le comportement » raconte une meilleure histoire que vingt commits « WIP » squashés à la dernière seconde.</p>
<p>Utilisez le rebase interactif pour nettoyer une branche désordonnée avant d'ouvrir la PR. Vous n'avez pas besoin de commits parfaits dès le début — polissez-les avant de demander des regards.</p>

<h2 id="self-review">Auto-revue avant de demander la revue</h2>
<p>Avant de contacter un coéquipier, devenez votre propre premier relecteur. Ouvrez le diff sur GitHub (ou votre hébergeur de code) et lisez-le comme si vous n'aviez jamais vu le code. Ce changement de contexte détecte plus de problèmes que vous ne l'attendriez : les sorties de débogage oubliées, une fonction qui a trop grossi, un import que vous avez oublié de supprimer, un commentaire qui était vrai autrefois.</p>
<p>Une checklist d'auto-revue qui mérite d'être parcourue :</p>
<ul>
  <li><strong>Lisez le diff de haut en bas.</strong> Ne survolez pas. Faites semblant de le voir pour la première fois.</li>
  <li><strong>Vérifiez les restes de débogage.</strong> <code>console.log</code>, commentaires TODO marqués « supprimer », blocs commentés, valeurs temporaires codées en dur.</li>
  <li><strong>Vérifiez que les tests sont là.</strong> Si vous avez ajouté de la logique, y a-t-il un test ? Si vous avez corrigé un bug, y a-t-il un test de régression ?</li>
  <li><strong>Cherchez les changements non intentionnels.</strong> Agitation des espaces blancs, fichiers reformatés, dérive de portée accidentelle. Déplacez-les dans une PR séparée ou annulez-les.</li>
  <li><strong>Laissez vous-même des commentaires guidants.</strong> Si une section n'est pas évidente, ajoutez un commentaire PR expliquant le raisonnement avant que votre relecteur le voie. Ce n'est pas un signe de faiblesse — c'est de la considération.</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">Une règle simple</span><p>Si vous ne seriez pas à l'aise avec un ingénieur senior révisant la PR à froid en ce moment — sans votre explication — elle n'est pas prête à sortir. Corrigez d'abord, puis demandez la revue.</p></div>
<p>L'auto-revue rapporte aussi un dividende égoïste : elle détecte les erreurs embarrassantes avant qu'elles ne deviennent des commentaires permanents dans l'historique de revue. Tout le monde a une histoire de la faute de frappe qu'il a repérée dix secondes après avoir appuyé sur « Demander la revue ». Dix minutes d'auto-revue élimine la plupart de celles-là.</p>

<h2 id="responding">Répondre à la revue</h2>
<p>Une PR n'est pas terminée quand vous poussez le code — elle est terminée quand elle merge. Entre ces deux événements se trouve le cycle de revue, et la façon dont vous gérez les retours compte autant que le code lui-même.</p>
<p>Quelques principes qui gardent les revues en mouvement et les relations intactes :</p>
<ul>
  <li><strong>Répondez à chaque commentaire.</strong> Même juste pour dire « fait » ou « bon point, corrigé dans le dernier commit ». Le silence laisse les relecteurs se demander si vous avez vu leur note.</li>
  <li><strong>Décrivez votre correctif quand vous le poussez.</strong> « Corrigé dans le dernier commit — déplacé la logique dans un helper et ajouté un test » est bien plus clair qu'un nouveau commit sans contexte.</li>
  <li><strong>Résolvez les fils de manière réfléchie.</strong> Dans la plupart des équipes, c'est le commentateur qui résout quand il est satisfait. Ne vous auto-résolvez pas sans reconnaître le retour. Si vous n'êtes pas sûr de la convention, demandez.</li>
  <li><strong>Soyez en désaccord avec respect.</strong> Si vous pensez qu'une suggestion est fausse, dites-le clairement et avec du raisonnement — ne l'ignorez pas simplement. « J'ai envisagé cette approche, mais [raison] — heureux d'en discuter si vous vous sentez fortement » maintient la conversation professionnelle et productive. Voir aussi : <a href="/blog/culture/how-to-review-code-kindly">comment réviser le code avec bienveillance</a> pour la perspective du relecteur sur la même conversation.</li>
  <li><strong>Re-demandez la revue explicitement.</strong> Après avoir adressé les retours, n'attendez pas que les relecteurs le remarquent. Re-demandez la revue et laissez un court résumé : « Adressé tous les commentaires. Changements principaux : extrait l'utilitaire, ajouté deux tests, gardé l'approche originale pour X parce que Y. »</li>
</ul>

<h2 id="by-team-size">Comment les normes de PR évoluent selon la taille de l'équipe</h2>
<p>Ce qui fait une excellente PR n'est pas fixe — ça évolue à mesure que l'équipe grandit et que la base de code mûrit. Voici comment les attentes évoluent :</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr><th>Contexte d'équipe</th><th>Ce qui compte le plus</th><th>Normes pratiques</th></tr>
  </thead>
  <tbody>
    <tr>
      <th>Solo / freelance</th>
      <td>Vous-du-futur est le relecteur. Les PRs sont la documentation des décisions pour quand vous reviendrez au code dans six mois.</td>
      <td>Même sans relecteur : un titre clair, un bref « pourquoi » dans la description. Des messages de commit qui rendent <code>git log</code> lisible plus tard.</td>
    </tr>
    <tr>
      <th>Petite équipe (2–8)</th>
      <td>La vitesse compte. Tout le monde connaît la base de code. Les revues sont conversationnelles. La confiance est élevée.</td>
      <td>Gardez les PRs petites et rapides. Les descriptions peuvent être plus légères — une phrase ou deux de contexte. Une revue async avec un retour le même jour est une attente raisonnable.</td>
    </tr>
    <tr>
      <th>Équipe de taille moyenne (10–50)</th>
      <td>Les relecteurs peuvent ne pas connaître tout le contexte. L'intégration est constante. Les changements couvrent les équipes.</td>
      <td>Descriptions complètes quoi/pourquoi/comment. Captures d'écran pour le travail UI. Un plan de test. Les changements inter-équipes nécessitent un soin supplémentaire et possiblement une revue plus large.</td>
    </tr>
    <tr>
      <th>Grande / entreprise (50+)</th>
      <td>Les relecteurs sont parfois des inconnus. La conformité, les pistes d'audit et les revues d'exactitude comptent. Des SLAs de délai d'exécution existent.</td>
      <td>Templates PR formels imposés par les outils. Approbations requises. Revue de sécurité pour les changements sensibles. Liens vers les docs de conception et les tickets. Plan de rollback pour les changements risqués.</td>
    </tr>
  </tbody>
</table>
</div>

<p>Une équipe startup de quatre personnes n'a pas besoin d'un plan de rollback dans chaque description de PR. Une banque en a besoin. Adaptez la cérémonie aux enjeux, et revisitez les normes à mesure que l'équipe grandit — ce qui fonctionnait à dix ingénieurs semblera insuffisant à cinquante.</p>

<div class="blog-takeaways">
<h3 id="takeaways">Points clés à retenir</h3>
<ul>
  <li><strong>Une PR est un message à un humain.</strong> Écrivez-la pour votre relecteur, pas pour vous-même. Son temps et son contexte sont limités.</li>
  <li><strong>Gardez-la petite et focalisée.</strong> Un changement logique par PR. Séparez le refactoring des changements de comportement. Utilisez les feature flags pour merger du travail incomplet en toute sécurité derrière un toggle.</li>
  <li><strong>Titre + description = orientation.</strong> Répondez à ce qui a changé, pourquoi c'était nécessaire et comment vérifier. Le code montre le quoi ; la description fournit le pourquoi.</li>
  <li><strong>Un historique de commits propre raconte une histoire.</strong> Messages conventionnels, commits logiques, un rebase soigné avant de demander la revue.</li>
  <li><strong>Auto-revue en premier.</strong> Lisez votre propre diff sur GitHub avant quiconque d'autre. Laissez des commentaires guidants. Détectez les erreurs évidentes vous-même.</li>
  <li><strong>Fermez la boucle sur les retours.</strong> Répondez à chaque commentaire, re-demandez la revue après avoir adressé les notes, et soyez en désaccord avec du raisonnement — jamais avec le silence.</li>
  <li><strong>Adaptez la cérémonie à la taille de l'équipe.</strong> Une startup de quatre personnes et une banque de 500 personnes ont besoin de normes PR différentes. Revisitez les vôtres à mesure que vous grandissez.</li>
</ul>
</div>
