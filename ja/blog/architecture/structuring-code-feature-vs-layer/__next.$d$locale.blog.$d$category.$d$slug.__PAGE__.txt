14:T81e8,<p>新しいプロジェクトを開いて、ロジックを一行も書く前に最初にすることは――フォルダーを作ることです。<code>controllers/</code> と名付けるかもしれません。<code>orders/</code> かもしれません。どちらにしても、その静かな選択はすでにアーキテクチャの決定であり、その後のすべてを静かに形作ります：新しいチームメンバーがどれほど速く足場を見つけるか、一つの機能が変わったときにいくつのファイルを触るか、そして六ヶ月後にコードベースがあなたの<em>ビジネス</em>のように読めるか<em>フレームワーク</em>のように読めるかです。</p>
<p>この記事は三つの主なアプローチ――<strong>レイヤー別</strong>・<strong>フィーチャー別</strong>・<strong>叫ぶ（ドメイン別）</strong>――の実践的なツアーです。各会社の規模ごとの正直なトレードオフを添えています。一つの答えがすべての状況に勝つわけではありませんが、デフォルトではなく意図的に選択するための明確なメンタルモデルを持ってここを離れられるはずです。</p>

<h2 id="why-it-matters">フォルダー構造がアーキテクチャの決定である理由――細かい話ではない</h2>
<p>人々はフォルダー構造を「ただの整理」と軽視します。それははるかに多くのことを意味します。コードをどうグループ化するかは、非常に重要であることがわかる三つのことを制御します：</p>
<ul>
  <li><strong>カップリング。</strong>一緒に変わるコードが遠く離れて住んでいると、小さな更新が毎回フォルダーをまたいだ遠征になります。一緒に住んでいれば、関連する変更は一つの場所にまとまります。</li>
  <li><strong>発見可能性。</strong>コードベースに不慣れな開発者は「課金コードはどこにあるか？」と尋ねます。その答え――一つのフォルダー、または五つの散らばったフォルダー――がオンボーディング時間を決めます。</li>
  <li><strong>影響範囲。</strong>新しい機能を追加したり古い機能を安全に削除したりするためにいくつのファイルを開かなければならないか？構造はその影響範囲を制限するか、制御なく広がるままにするかのどちらかです。</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">メンタルモデル</span><p>フォルダー構造を<strong>地図</strong>として考えてください。良い地図は、何でできているかではなく、現実世界でどう関連するかでものをグループ化します。ファイルをタイプ別（コントローラー・モデル）にグループ化するのは、地図のアイコンを形で並び替えるようなものです。フィーチャー別にグループ化するのは近所で並び替えるようなものです。一方はナビゲートするのに役立ちます；もう一方は技術的には正しいですが実践的には役に立ちません。</p></div>
<p>ソフトウェアシステムは、それを構築したチームのコミュニケーション構造を反映するという Conway の法則もここに適用されます。チームがビジネス機能（決済スクワッド・注文スクワッド）を中心に組織されているなら、フィーチャー別の構造は自然に感じられ、正しいバウンダリーを強制します。レイヤー別の構造はあなたと戦うことになります。</p>

<h2 id="by-layer">レイヤー別：おなじみのデフォルト</h2>
<p>レイヤー別の整理は、技術的な役割でファイルをグループ化します。典型的な Node や Spring Boot プロジェクトはこのような形になります：</p>

<figure class="blog-figure">
  <pre><code>src/
├─ controllers/
│  ├─ OrderController.ts
│  ├─ UserController.ts
│  └─ BillingController.ts
├─ services/
│  ├─ OrderService.ts
│  ├─ UserService.ts
│  └─ BillingService.ts
├─ repositories/
│  ├─ OrderRepository.ts
│  ├─ UserRepository.ts
│  └─ BillingRepository.ts
└─ models/
   ├─ Order.ts
   ├─ User.ts
   └─ Invoice.ts</code></pre>
  <figcaption>レイヤー別の構造：各技術レイヤーが独自のトップレベルフォルダーを持ちます。機能を追加するには<em>四つすべての</em>フォルダーにファイルを追加します。</figcaption>
</figure>

<p>すべてのWebフレームワークのチュートリアルがここに行き着きます。Rails もそうです。Spring Boot のスターターもそうです。Express のジェネレーターもそうです。そのなじみ深さが主な資産です。</p>
<p><strong>レイヤー別が輝く場所：</strong></p>
<ul>
  <li>二〜三の機能を持つ小さなアプリ――構造が非常にフラットなので、ほとんど問題になりません。</li>
  <li>全員がすべての機能をまたいで作業するチーム；レイヤーが専門化の単位です。</li>
  <li>初期セットアップのスピードが長期的な明確さに勝るプロトタイプや内部ツール。</li>
</ul>
<p><strong>レイヤー別が痛む場所：</strong></p>
<ul>
  <li><em>一つの機能がフォルダー全体に散らばっている。</em>「払い戻し」機能を追加するには <code>controllers/</code>・<code>services/</code>・<code>repositories/</code>・<code>models/</code> を触る必要があります。概念的には一つのことなのに四つのフォルダーです。</li>
  <li><em>機能を削除することが考古学の作業になる。</em><code>services/</code> のどのファイルが課金に属し、どれが注文に属するか、どうやってわかりますか？フォルダー名は教えてくれません。</li>
  <li><em>マージコンフリクトが増える。</em>すべての機能変更が同じレイヤーレベルのファイルを触り、二人の開発者が並行して作業するときに衝突が起きます。</li>
</ul>
<div class="blog-callout blog-callout--warn"><span class="blog-callout__label">スケーリングの崖</span><p>レイヤー別は10〜15の機能まで問題なく機能します。それを超えると、各トップレベルフォルダーは<strong>がらくた引き出し</strong>になります：40ファイルの <code>services/</code> フォルダー、明確なオーナーシップなし、レビュアーが以前に見たことのないファイルに触れるプルリクエスト。</p></div>

<h2 id="by-feature">フィーチャー別：変更は一つのフォルダーに収まる</h2>
<p>フィーチャー別の整理は、それが担うプロダクトのスライスでファイルをグループ化します。技術レイヤー（コントローラー・サービス・モデル）は依然として存在します――ただし、トップレベルではなく各フィーチャーフォルダーの<em>内部</em>に住みます：</p>

<figure class="blog-figure">
  <pre><code>src/
├─ orders/
│  ├─ OrderController.ts
│  ├─ OrderService.ts
│  ├─ OrderRepository.ts
│  └─ Order.ts
├─ billing/
│  ├─ BillingController.ts
│  ├─ BillingService.ts
│  ├─ BillingRepository.ts
│  └─ Invoice.ts
├─ auth/
│  ├─ AuthController.ts
│  ├─ AuthService.ts
│  └─ User.ts
└─ shared/
   ├─ database.ts
   ├─ logger.ts
   └─ Money.ts</code></pre>
  <figcaption>フィーチャー別の構造：各プロダクト概念が独自のフォルダーを持ちます。払い戻し機能を追加しても一つのフォルダーが追加されるだけで、他は何も動きません。</figcaption>
</figure>

<p><strong>フィーチャー別が輝く場所：</strong></p>
<ul>
  <li><em>変更は一つのフォルダーに収まる。</em>払い戻しフローを追加するには <code>billing/</code> 内にファイルを追加します――他のフォルダーは触られません。影響範囲は小さく封じ込められています。</li>
  <li><em>機能の削除がシンプル。</em>フォルダーを削除します。外から何かがまだインポートしているなら、IDEがすぐに教えてくれます。</li>
  <li><em>オーナーシップが明白。</em>決済チームが <code>billing/</code> を所有し、アイデンティティチームが <code>auth/</code> を所有します。モノレポのツールとコードレビュールールがこれを機械的に強制できます。</li>
  <li><em>オンボーディングが速い。</em>注文に取り組む新入社員は <code>orders/</code> だけを読みます。最初にコードベース全体を理解する必要はありません。</li>
</ul>
<p><strong>フィーチャー別が注意を要する場所：</strong></p>
<ul>
  <li><em>共有コードの配置が厄介。</em><code>Money</code> 型は <code>orders/</code> と <code>billing/</code> の両方で使われます。<code>orders/</code> に置くのは違和感があります。結局 <code>shared/</code> に入りますが、誰も所有しなければそれ自体が第二のがらくた引き出しになりえます。</li>
  <li><em>フィーチャーをまたぐ依存には規律が必要。</em><code>orders/OrderService.ts</code> が <code>billing/BillingService.ts</code> から直接インポートすると、フォルダー構造が見えないカップリングを作ってしまいます。チームはリンティングルール（<code>eslint-plugin-import/no-restricted-paths</code>）やモジュール境界ツール（Nx・NestJS モジュール）を使って、これらを明示的にします。</li>
</ul>

<h2 id="screaming">Screaming Architecture：トップレベルは何を語るか？</h2>
<p>Uncle Bob（Robert C. Martin）は有名なブログ投稿で <strong>Screaming Architecture</strong> という用語を作りました：「あなたのアプリケーションのアーキテクチャは何を叫んでいますか？トップレベルのディレクトリ構造と最上位パッケージのソースファイルを見たとき、それは叫んでいますか：<em>ヘルスケアシステム</em>、または<em>会計システム</em>、または<em>在庫管理システム</em>？それとも叫んでいますか：<em>Rails</em>、または<em>Spring/Hibernate</em>、または<em>ASP.Net</em>？」</p>
<p>洞察はエレガントです：トップレベルのフォルダーが <code>controllers/</code>・<code>models/</code>・<code>views/</code> なら、コードベースは <em>MVCフレームワーク</em>を叫んでいます。フレームワークはツールです――読者が最初に出会うべきものではありません。</p>
<p>Screaming Architecture（<strong>ドメイン別</strong>とも呼ばれる）はフィーチャー別をもう一歩進めます。機能がトップレベルにあるだけでなく――それらの機能の名前は技術的な役割ではなく、<em>ビジネス言語</em>（ドメイン駆動設計のユビキタス言語）から直接引かれています：</p>

<pre><code><span class="cmt">// 「eコマースプラットフォーム」を叫ぶ</span>
src/
├─ catalog/          <span class="cmt">// ドメイン概念：商品カタログ</span>
├─ ordering/         <span class="cmt">// ドメイン概念：注文の配置と追跡</span>
├─ fulfillment/      <span class="cmt">// ドメイン概念：ピッキング・梱包・出荷</span>
├─ payments/         <span class="cmt">// ドメイン概念：請求・返金・請求書</span>
└─ identity/         <span class="cmt">// ドメイン概念：アカウント・認証・権限</span>

<span class="cmt">// フレームワークを叫ぶ代替案と比較：</span>
src/
├─ controllers/      <span class="cmt">// 「MVC」を叫ぶ</span>
├─ models/           <span class="cmt">// 「ORM」を叫ぶ</span>
├─ views/            <span class="cmt">// 「テンプレートエンジン」を叫ぶ</span>
└─ services/         <span class="cmt">// 「サービスレイヤーパターン」を叫ぶ</span></code></pre>

<p>各ドメインフォルダーの内部レイアウトはチームに任されています――それで構いません。肝心なのは、<em>トップレベル</em>がインフラストラクチャではなくビジネスを伝えることです。新しい開発者（あるいはCTO・ビジネス側のドメインエキスパート）がトップレベルのフォルダーを読んで、どうやって作られているかを知らなくてもシステムが<em>何をするか</em>を理解できれば、成功です。</p>

<h2 id="side-by-side">並べて比較</h2>
<p>五つの実践的な次元にわたる正直な評価です：</p>

<div class="blog-table-wrap">
<table>
  <thead>
    <tr>
      <th>次元</th>
      <th>レイヤー別</th>
      <th>フィーチャー別</th>
      <th>ドメイン別（叫ぶ）</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th>発見可能性</th>
      <td>スケールで低い――「課金コードはどこ？」に速い答えがない</td>
      <td>高い――概念ごとに一つのフォルダー</td>
      <td>最高――フォルダー名はビジネス自体から来る</td>
    </tr>
    <tr>
      <th>変更の局所性</th>
      <td>低い――一つの機能変更がすべてのレイヤーに散らばる</td>
      <td>高い――変更は一つのフォルダーにまとまる</td>
      <td>高い――フィーチャー別と同じで、加えて明示的なドメイン言語</td>
    </tr>
    <tr>
      <th>オンボーディング速度</th>
      <td>遅い――新メンバーは何かに触る前にレイヤー全体を理解しなければならない</td>
      <td>速い――一つの機能を作業するために一つのフォルダーを学ぶ</td>
      <td>最速――ドメイン名が事前の文脈なしに理解を導く</td>
    </tr>
    <tr>
      <th>カップリングリスク</th>
      <td>高い――共有レイヤーが暗黙のカップリング面になる</td>
      <td>中程度――フィーチャーをまたぐインポートは可能だが見える</td>
      <td>低い――明示的な境界付けられたコンテキストと強制されたバウンダリー</td>
    </tr>
    <tr>
      <th>「大きな泥団子」のリスク</th>
      <td>高い――機能が増えるにつれてレイヤーがごちゃ混ぜ引き出しになる</td>
      <td>中程度――shared/ フォルダーが無制限に成長しうる</td>
      <td>低い――ドメインバウンダリーが泥を早期に見えるようにする</td>
    </tr>
  </tbody>
</table>
</div>

<p>このハイブリッドは両方の良いところを手に入れます：</p>
<ul>
  <li><em>トップレベルはビジネスを叫びます</em>――新メンバーはすぐにこれがeコマースアプリであり、一般的なMVCのスケルトンではないことを知ります。</li>
  <li><em>各フィーチャーが自己完結しています</em>――課金フローへの変更は <code>billing/</code> だけを触ります。</li>
  <li><em>技術レイヤーは各フォルダーの内部に依然として存在します</em>ので、MVCや Clean Architecture レイヤーで考える開発者は迷いません。フィーチャーフォルダーの内側を見るだけです。</li>
  <li><em><code>shared/</code> フォルダー</em>が本当に横断的な関心事（<code>Money</code> 値オブジェクト・ロガー・データベース接続）を保持します。肝心なルール：<code>shared/</code> に何かを追加する場合、それは逃げ道として捨てるのではなく、意図的に抽出・所有されるべきです。</li>
</ul>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">共有フォルダーの命名</span><p>一部のチームは <code>shared/</code>・他は <code>common/</code>・<code>lib/</code>・または <code>core/</code> と呼びます。名前は規律よりも重要ではありません：ミニ内部ライブラリとして扱ってください。特定の機能と一緒に常に変更されるファイルがあれば、それは shared に属さず、その機能の内部に属します。</p></div>

<h2 id="at-scale">チームのスケールで：強制されたバウンダリーとモノレポ</h2>
<p>コードベースが成長して複数のチームが貢献するようになると、フォルダーの慣習だけでは不十分です――開発者はいつでも相対インポートでバウンダリーをまたぐことができます。次のステップはバウンダリーを<em>強制</em>することです：</p>
<ul>
  <li><strong>NestJS モジュール</strong>はモジュール間の依存を明示的にします：別のモジュールが <code>exports</code> するものへのアクセスだけが得られます。パブリックAPIを通らないモジュール境界をまたぐインポートはコンパイルエラーです。</li>
  <li><strong>Nx のバウンダリールール</strong>で、どのフィーチャーまたはライブラリが他のどれに依存することを許可されるかを宣言し、CIのリントエラーとして強制できます。</li>
  <li><strong>モノレポ</strong>はこれを最も遠くまで進めます：各ドメイン（注文・課金・認証）が独自の <code>package.json</code>・独自のテスト・明示的に公開されたエクスポートを持つ独自パッケージになります。Turborepo と Nx が JavaScript エコシステムでこれを管理する主要なツールです。構造はフォルダーツリーよりパッケージのグラフに近くなります――しかし根本的なアイデアは同じです：ビジネス機能別にグループ化し、バウンダリーを明示的にし、機械的に強制する。</li>
</ul>
<p>モノレポへの移行が常に必要なわけではありません。多くのチームは単一パッケージとよく規律されたハイブリッド構造で何年もうまくやっています。クロスチームのカップリングが本当の痛みになったとき――予防的にではなく――モノレポツールに手を伸ばしましょう。</p>

<div class="blog-callout blog-callout--note"><span class="blog-callout__label">よくある間違い</span><p>最も一般的な構造的な間違いは間違ったアプローチを選ぶことではありません――<strong>間違ったアプローチに長く留まりすぎること</strong>です。チームはレイヤー別で始め、20の機能で痛みを感じ、それでも「構造は技術的には機能している」という理由でリファクタリングしません。15の機能で一午後のフォルダー再編成は、50の機能での何ヶ月もの混乱を防ぎます。</p></div>

<div class="blog-takeaways">
<h3 id="takeaways">まとめ</h3>
<ul>
  <li><strong>構造はアーキテクチャです。</strong>フォルダーレイアウトはカップリング・発見可能性・将来のすべての変更の影響範囲を制御します。</li>
  <li><strong>レイヤー別</strong>はなじみ深く小さなアプリには問題ありませんが、10〜15の機能を超えるとがらくた引き出しになります。</li>
  <li><strong>フィーチャー別</strong>は変更を局所的に保ちます。新しい機能は一つのフォルダーを触ります。機能を削除することは一つのフォルダーを削除することを意味します。</li>
  <li><strong>Screaming Architecture</strong>は、トップレベルがフレームワークではなく<em>ビジネス</em>を説明すべきだと言います。非技術的な関係者がフォルダー名を読んでシステムが何をするかを理解できれば、成功です。</li>
  <li><strong>実用的なハイブリッド</strong>は多くの良いチームがたどり着く：トップにドメイン/フィーチャー名、各フォルダーの内部に小さな技術レイヤー。</li>
  <li><strong>スケールに合わせてバウンダリーを強制する。</strong>フォルダーの慣習は提案です；モジュールシステム・リントルール・モノレポパッケージがそれらを保証にします。</li>
  <li><strong>早めに再編成する</strong>――15の機能での一午後のフォルダーリファクタリングは、50の機能での何ヶ月もの混乱を避けます。</li>
</ul>
</div>
