14:T8dae,<p>ある時点ですべての開発者は同じ壁にぶつかります。コードの一部をテストしたいのに、そのコードが内部で <code>new Database()</code> を呼んでいます。何をしても、本物のデータベース接続が起動します。メールプロバイダーを交換したいのに、メールSDKが三層も深くに埋まっています。同じロジックをHTTPリクエストとスケジュールジョブの両方で動かしたいのに、フレームワークがあまりにもがっちりとボルトで締まっていて、実質的に不可能です。</p>
<p>これらはすべて、一つの根本的な原因の症状です：<strong>自分自身の依存を作るコード</strong>。解決策には正式な名前があります――<em>Dependency Injection</em>――ですが、その根底にあるアイデアは拍子抜けするほどシンプルです。コードが自分で協調者を見つけるのをやめて、<em>外から渡す</em>ようにします。それだけです。他のすべて――IoCコンテナ・DIフレームワーク・インジェクショントークン――はその一つのアイデアの上に構築された仕組みにすぎません。</p>
<p>このガイドでは初めからほぐしていきます：問題、原則、パターン、そしてテストにおける実践的な恩恵。TypeScript を全体で使用します。あらゆるWebまたはサーバーバックグラウンドを持つエンジニアにとって読みやすいからです。</p>

<h2 id="the-problem">ハードワイヤードな依存：コードが自分で <code>new</code> を呼ぶとき</h2>
<p>注文を処理するサービスがあります。一見すると完全に合理的に見えます：</p>
<pre><code><span class="cmt">// order-service.ts — 「修正前」バージョン</span>
<span class="kw">class</span> OrderService {
  <span class="kw">async</span> placeOrder(cart: Cart): Promise&lt;Order&gt; {
    <span class="kw">const</span> db     = <span class="kw">new</span> PostgresClient(process.env.DATABASE_URL!)  <span class="cmt">// ハードワイヤードなDB</span>
    <span class="kw">const</span> stripe = <span class="kw">new</span> Stripe(process.env.STRIPE_SECRET_KEY!)        <span class="cmt">// ハードワイヤードなベンダー</span>
    <span class="kw">const</span> mailer = <span class="kw">new</span> SendGridMailer(process.env.SENDGRID_KEY!)    <span class="cmt">// ハードワイヤードなベンダー</span>

    <span class="kw">const</span> order = Order.fromCart(cart)
    <span class="kw">await</span> stripe.charges.create({ amount: order.total, source: cart.token })
    <span class="kw">await</span> db.query(<span class="str">'INSERT INTO orders …'</span>, [order])
    <span class="kw">await</span> mailer.send({ to: cart.email, template: <span class="str">'order-confirmed'</span> })
    <span class="kw">return</span> order
  }
}</code></pre>
<p>これを使おうとした瞬間、三つの問題が明確になります：</p>
<ul>
  <li><strong>単独でテスト不可能。</strong>本物のクレデンシャル・ライブのデータベース・ライブのStripeサンドボックスなしに <code>placeOrder</code> をユニットテストで実行できません。デフォルトで、すべてのテストが遅くて壊れやすいインテグレーションテストになります。</li>
  <li><strong>特定の実装に硬くカップリングされている。</strong>ビジネスが Stripe から Adyen に移行するとき、このファイルを開いて手術をします。ステージングにダミーのメーラーが必要なとき、プラグインする明確な方法がありません。</li>
  <li><strong>隠れた依存。</strong>クラスのシグネチャを読む人は <code>placeOrder(cart)</code> を見て、このメソッドが秘密裡に三つの外部サービスを必要とすることを知りません。依存はすべての行を読むまで見えません。</li>
</ul>
<div class="blog-callout blog-callout--note"><span class="blog-callout__label">注目すべきパターン</span><p>メソッドやコンストラクター本体の<em>内部</em>で <code>new SomeExternalThing()</code> を見るたびに――コンポジションルートではなく――それはハードワイヤードな依存です。クラスは自分の仕事をすることと、その仕事に必要なツールを見つけて構築することの両方に責任を負っています。これは二つの仕事であり、互いに反対方向に引っ張ります。</p></div>

<h2 id="inversion-of-control">Inversion of Control：誰が作成を管理するか？</h2>
<p>コードで問題を解決する前に、関連する原則を名前で呼ぶことが役立ちます。<strong>Inversion of Control</strong>（IoC）は、コードが自分自身の依存を取得することに責任を負うべきではないというアイデアです。その代わりに、外部のもの――フレームワーク・テストハーネス・コンポジションルート――が物を作って渡すことを担います。</p>
<p>いわゆる<em>ハリウッド原則</em>を聞いたことがあるかもしれません：<strong>「こちらから電話しないで、向こうから電話するから。」</strong>古典的な命令型コードでは、ビジネスロジックが依存を直接呼び出します――データベースに手を伸ばして掴みます。IoCでは、依存があなたのところに来ます。その関係の制御が逆転します：いつどのように協調者を取得するかをあなたが決めるのではなく、外の世界が決めて届けます。</p>
<p>IoC は広い原則であり、特定の技術ではありません。フレームワークはさまざまな方法でそれを実装しています：</p>
<ul>
  <li><strong>Dependency Injection（DI）</strong>――協調者をコンストラクター引数またはメソッドパラメーターとして渡す。最も一般的で最も明示的な形式。</li>
  <li><strong>Service Locator</strong>――コードが実行時に名前で依存を取得するグローバルレジストリ。（後でこれがアンチパターンである理由について触れます。）</li>
  <li><strong>Template Method</strong>――基底クラスがサブクラスがオーバーライドするフックメソッドを呼び出すので、基底クラスがフローを制御し、サブクラスが空白を埋めます。</li>
  <li><strong>Event / Observer</strong>――コンポーネントがイベントを発行し、フレームワークがそれをサブスクライバーにルートします。どちらの側も相手の存在を知りません。</li>
</ul>
<p>これらの中で、<strong>Dependency Injection は最も透明でテストしやすいものです</strong>。またIoC自体と最もよく混同されるものでもあります。だから二つの用語をほぐす価値があります：IoCは<em>原則</em>であり、DIはその原則の一つの<em>実装</em>です。</p>

<h2 id="dependency-injection">Dependency Injection：依存を受け取る</h2>
<p>DIの仕組みは、一度見れば恥ずかしいほどシンプルです。クラスの内部で協調者を構築するのではなく、パラメーターとして宣言し、呼び出し側に提供させます。</p>
<p>三つの種類がありますが、ほとんどの状況で一つの明確な勝者があります：</p>
<h3 id="constructor-injection">コンストラクターインジェクション（推奨）</h3>
<p>依存はコンストラクターで宣言されます。これが標準的な選択です：すべての依存がクラスのシグネチャに見え、構築時に必要とされ、後で何かが再代入できないように <code>readonly</code> にできます。</p>
<pre><code><span class="cmt">// order-service.ts — コンストラクターインジェクションを使った「修正後」バージョン</span>
<span class="kw">interface</span> PaymentGateway { charge(amount: Money, token: string): Promise&lt;Receipt&gt; }
<span class="kw">interface</span> OrderRepository { save(order: Order): Promise&lt;void&gt; }
<span class="kw">interface</span> Mailer          { send(to: string, template: string): Promise&lt;void&gt; }

<span class="kw">class</span> OrderService {
  <span class="kw">constructor</span>(
    <span class="kw">private readonly</span> payments: PaymentGateway,
    <span class="kw">private readonly</span> orders:   OrderRepository,
    <span class="kw">private readonly</span> mailer:   Mailer,
  ) {}

  <span class="kw">async</span> placeOrder(cart: Cart): Promise&lt;Order&gt; {
    <span class="kw">const</span> order = Order.fromCart(cart)                   <span class="cmt">// 純粋なビジネスルール</span>
    <span class="kw">await</span> <span class="kw">this</span>.payments.charge(order.total, cart.token)
    <span class="kw">await</span> <span class="kw">this</span>.orders.save(order)
    <span class="kw">await</span> <span class="kw">this</span>.mailer.send(cart.email, <span class="str">'order-confirmed'</span>)
    <span class="kw">return</span> order
  }
}</code></pre>
<p>そのコンストラクターのシグネチャを声に出して読んでください：「<code>OrderService</code> を作るには、決済ゲートウェイ・注文リポジトリ・メーラーが必要です。」これは、メソッド本体を一行も読まずにプレーンなコードで書かれた見える契約です。</p>
<h3 id="setter-injection">セッター / メソッドインジェクション（時々使用）</h3>
<p>依存は構築後にメソッド経由で設定されます。オプションの協調者や、二段階でオブジェクトを構築するフレームワーク（例：一部のテストフレームワーク・一部のIoCコンテナ）に役立ちます。控えめに使いましょう――オブジェクトが部分的に設定された状態で存在することを許可し、これが微妙なバグの元になります。</p>
<pre><code><span class="cmt">// セッターインジェクション — 構築時の配線が不可能な場合のみ使用</span>
<span class="kw">class</span> ReportGenerator {
  <span class="kw">private</span> formatter: ReportFormatter = <span class="kw">new</span> DefaultFormatter()

  setFormatter(f: ReportFormatter) { <span class="kw">this</span>.formatter = f }  <span class="cmt">// オプションの上書き</span>

  generate(data: ReportData): string {
    <span class="kw">return this</span>.formatter.format(data)
  }
}</code></pre>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">シンプルな経験則</span><p>デフォルトで<strong>コンストラクターインジェクション</strong>を使いましょう。クラスが<em>常に</em>必要とするすべての依存はコンストラクターに属します。依存が本当にオプションの場合、またはフレームワークが強制する場合のみセッターインジェクションを使いましょう。依存グラフについて考えることを避けるための逃げ道としてセッターインジェクションを使わないでください。</p></div>

<h2 id="composition-root">コンポジションルート：すべてを一つの場所でつなぐ</h2>
<p>コンストラクターを通じて依存を注入すると、新しい質問が生まれます：<em>実際に誰が <code>new</code> を呼んでピースをつなぎ合わせるのか？</em>答えは<strong>コンポジションルート</strong>です：アプリケーションのエントリーポイントにある、すべての本物のオブジェクトが構築されてお互いに注入される単一の場所。</p>
<p>コードベース全体で <code>new ConcreteImplementation()</code> を自由に書いても良い唯一の場所だと考えてください。他のすべてはインターフェースの言葉で話します。コンポジションルートは「抽象システム」と「現実の世界」の間のシームです。</p>
<pre><code><span class="cmt">// composition-root.ts — 本物のものが配線される唯一の場所</span>
<span class="kw">import</span> { PostgresOrderRepository } <span class="kw">from</span> <span class="str">'./adapters/postgres-order-repository'</span>
<span class="kw">import</span> { StripePaymentGateway }    <span class="kw">from</span> <span class="str">'./adapters/stripe-payment-gateway'</span>
<span class="kw">import</span> { SendGridMailer }          <span class="kw">from</span> <span class="str">'./adapters/sendgrid-mailer'</span>
<span class="kw">import</span> { OrderService }            <span class="kw">from</span> <span class="str">'./domain/order-service'</span>
<span class="kw">import</span> { OrderController }         <span class="kw">from</span> <span class="str">'./http/order-controller'</span>

<span class="kw">export function</span> buildApp() {
  <span class="cmt">// 1. 最初に末端の依存を構築する（依存を持たないもの）</span>
  <span class="kw">const</span> db      = <span class="kw">new</span> PostgresOrderRepository(process.env.DATABASE_URL!)
  <span class="kw">const</span> stripe  = <span class="kw">new</span> StripePaymentGateway(process.env.STRIPE_SECRET_KEY!)
  <span class="kw">const</span> mailer  = <span class="kw">new</span> SendGridMailer(process.env.SENDGRID_KEY!)

  <span class="cmt">// 2. サービスを構築し、依存を注入する</span>
  <span class="kw">const</span> orderSvc = <span class="kw">new</span> OrderService(stripe, db, mailer)

  <span class="cmt">// 3. デリバリーレイヤーを構築し、サービスを注入する</span>
  <span class="kw">const</span> orderCtrl = <span class="kw">new</span> OrderController(orderSvc)

  <span class="kw">return</span> orderCtrl
}</code></pre>
<p>コンポジションルートがほぼレシピのように読めることに気づいてください：「これらの材料を作り、サービスに組み合わせ、サービスをコントローラーに入れます。」コードベース全体で<em>すべてがどうつながっているか</em>を理解するために行く場所が一つあります。これは新しいエンジニアをオンボーディングするとき、または材料を交換する必要があるときに非常に貴重な財産です。</p>
<p>テストでは、<em>テスト固有のコンポジションルート</em>を書きます――本物のアダプターの代わりにフェイクを配線する小さな同等物：</p>
<pre><code><span class="cmt">// テストファイルで — 同じサービス、フェイクの協調者</span>
<span class="kw">const</span> fakePayments = <span class="kw">new</span> InMemoryPaymentGateway()
<span class="kw">const</span> fakeOrders   = <span class="kw">new</span> InMemoryOrderRepository()
<span class="kw">const</span> fakeMailer   = <span class="kw">new</span> NoOpMailer()

<span class="kw">const</span> svc = <span class="kw">new</span> OrderService(fakePayments, fakeOrders, fakeMailer)
<span class="cmt">// ミリ秒で実行；ネットワークなし、データベースなし</span></code></pre>

<h2 id="by-hand-vs-container">手動DIとDIコンテナ――何が変わり、いつそれが重要か</h2>
<p>上記のすべては<em>手動DI</em>、またの名を<em>Pure DI</em>です。配線コードを自分で書きます。多くのプロジェクト――特に小さなもの――では、これが正しい選択です：プレーンなコードで、魔法がなく、すぐにデバッグ可能で、新しい開発者が一つのファイルを読むだけで配線全体を理解できます。</p>
<p>アプリケーションが成長するにつれ、手動アプローチは摩擦を生じさせます。DIコンテナ（IoCコンテナと呼ばれることもある）は依存の登録と解決を自動化し、その上にライフタイム管理を追加します。</p>

<p>コンテナが見返りを得るしきい値は様々ですが、大まかな目安として：コンポジションルート自体がメンテナンスの負担になり始めた感じ――常にスクロールしていて、新しいバインディングを追加し忘れたり、オブジェクトのライフタイムと格闘したりしている――それがコンテナに手を伸ばすシグナルです。それまでは、プレーンなバージョンの方が理解しやすくデバッグしやすいです。</p>

<h2 id="testing">テストの恩恵：フェイクを注入して高速テストを得る</h2>
<p>DI を学ぶ価値がある理由は、アーキテクチャの優雅さよりも、テストスイートにもたらすものです。すべての依存がコンストラクターを通じて届くとき、テストを書くことはフェイクを使った小さなコンポジションルートを書くことを意味します。データベースなし・ネットワークなし・ベンダーサンドボックスなし。テストはミリ秒で実行され、ステージング Stripe アカウントのレートリミットが理由で失敗することはありません。</p>
<pre><code><span class="cmt">// order-service.test.ts</span>
<span class="kw">class</span> InMemoryPaymentGateway <span class="kw">implements</span> PaymentGateway {
  receipts: Receipt[] = []
  <span class="kw">async</span> charge(amount: Money, token: string): Promise&lt;Receipt&gt; {
    <span class="kw">const</span> r = { id: <span class="str">'fake-receipt'</span>, amount }
    <span class="kw">this</span>.receipts.push(r)
    <span class="kw">return</span> r
  }
}

<span class="kw">class</span> InMemoryOrderRepository <span class="kw">implements</span> OrderRepository {
  store: Order[] = []
  <span class="kw">async</span> save(order: Order) { <span class="kw">this</span>.store.push(order) }
}

<span class="kw">class</span> SpyMailer <span class="kw">implements</span> Mailer {
  sent: string[] = []
  <span class="kw">async</span> send(to: string) { <span class="kw">this</span>.sent.push(to) }
}

<span class="cmt">// テスト自体は純粋で、高速で、決定論的：</span>
<span class="kw">it</span>(<span class="str">'注文を確認して顧客に通知する'</span>, <span class="kw">async</span> () =&gt; {
  <span class="kw">const</span> payments = <span class="kw">new</span> InMemoryPaymentGateway()
  <span class="kw">const</span> repo     = <span class="kw">new</span> InMemoryOrderRepository()
  <span class="kw">const</span> mailer   = <span class="kw">new</span> SpyMailer()

  <span class="kw">const</span> svc   = <span class="kw">new</span> OrderService(payments, repo, mailer)
  <span class="kw">const</span> order = <span class="kw">await</span> svc.placeOrder(testCart)

  expect(payments.receipts).toHaveLength(1)
  expect(repo.store).toContainEqual(order)
  expect(mailer.sent).toContain(testCart.email)
})</code></pre>
<p>これは <a href="/blog/architecture/ports-and-adapters">Ports &amp; Adapters</a> の中心にある洞察と同じです――インメモリのフェイクはテスト時のアダプターにすぎず、本物の Stripe または Postgres の実装は本番アダプターです。DIはスワップを可能にするメカニズムです；Ports &amp; Adapters はバウンダリーを意図的で明示的にするアーキテクチャパターンです。</p>
<p>ここで SOLID についても触れる価値があります。「D」は<strong>依存性逆転の原則</strong>を意味します：高レベルモジュールは低レベルモジュールに依存すべきではなく、両方が抽象に依存すべきです。上記のインターフェース（<code>PaymentGateway</code>・<code>OrderRepository</code>・<code>Mailer</code>）がその抽象です。<code>OrderService</code> が高レベルモジュールです。<code>StripePaymentGateway</code> と <code>PostgresOrderRepository</code> が低レベルモジュールです。どちらも相手のことを直接知りません――インターフェースで出会い、コンポジションルートがそれらを組み合わせます。</p>

<h2 id="pitfalls">アンチパターン：DIが間違った方向に行く方法</h2>
<p>アイデアはシンプルですが、それを損なうよく踏まれた方法がいくつかあります。</p>
<h3 id="service-locator">Service Locator</h3>
<p>Service Locator は、コードが実行時に依存を取得するために呼び出すグローバルレジストリです：</p>
<pre><code><span class="cmt">// アンチパターン：Service Locator — DIのように見えるがそうではない</span>
<span class="kw">class</span> OrderService {
  <span class="kw">async</span> placeOrder(cart: Cart) {
    <span class="kw">const</span> payments = ServiceLocator.get&lt;PaymentGateway&gt;(<span class="str">'PaymentGateway'</span>)
    <span class="kw">const</span> repo     = ServiceLocator.get&lt;OrderRepository&gt;(<span class="str">'OrderRepository'</span>)
    <span class="cmt">// … 続く</span>
  }
}</code></pre>
<p>これはインラインの <code>new Stripe()</code> よりクリーンに感じられますが、同じ問題を隠しています。依存はシグネチャに見えません。テストは実行前にグローバルレジストリを設定しなければなりません。クラスはいつでも何でも要求できます――DIを価値あるものにする明示性を失っています。</p>
<h3 id="over-injection">過剰なインジェクション（10引数コンストラクター）</h3>
<p>コンストラクターが七つ・八つ・または十のパラメーターを蓄積するとき、それは通常、DIが間違ったツールではなく、クラスがやりすぎているサインです。修正はクラスをより小さく焦点を絞った協調者に分解することで、それぞれが短い依存リストを持ちます。コンストラクターの長さは有用な健全性の指標です。</p>
<h3 id="hidden-globals">隠れたグローバルコンテナ</h3>
<p>サービス内から直接DIコンテナにアクセスすること――コンポジションルートでのみではなく――は、別の名前のService Locatorパターンです。ルールは厳格です：コンポジションルートだけがコンテナと話します。サービスは注入されたインターフェースとだけ話します。</p>
<h3 id="magic-you-cant-trace">追跡できない魔法</h3>
<p>一部のフレームワークはデコレーターとリフレクトメタデータを通じて依存を配線しますが、何かが壊れたときに本当に追いかけにくい方法でそれを行います。コンテナのドキュメントを読む時間が自分のビジネスロジックを読む時間より多ければ、それよりシンプルなアプローチを使うサインです――あるいは少なくとも主要なバインディングを明示的に書き出したよくドキュメント化されたコンポジションルートを追加すること。</p>

<h2 id="by-company-size">DIの適切なサイズ：ソロプロジェクトからエンタープライズまで</h2>
<p>ほとんどのアーキテクチャのアイデアと同様に、DIは問題に合わせてスケールします。目標は常に同じ――テストしやすく変更しやすいコード――ですが、そこに到達するための適切な仕組みは、チームとコードベースの成長に合わせて変わります。</p>

<p>フィンテックスタートアップは三つのサービスを配線する30行のコンポジションルートから始めるかもしれません。二年後、同じコードベースに60のサービスがあります：手動配線がそれ自体メンテナンスの作業になり、チームはグラフを自動的に処理するために NestJS か InversifyJS を使うようになります。<em>基盤となるパターン</em>――コンストラクターインジェクション・インターフェース・明確なコンポジションルート――が同じままであるため、スイッチは簡単です。コンテナは機械的なアシスタントであり、異なるアイデアではありません。</p>
<div class="blog-callout blog-callout--tip"><span class="blog-callout__label">実践的なアドバイス</span><p>手動DIとコンポジションルートから始めましょう。配線ファイルが負担になったらいつでもコンテナを導入でき、移行は機械的です：明示的な <code>new</code> 呼び出しを登録に置き換えるだけです。ドメイン・サービス・テストの何も変える必要はありません――コンポジションルートだけが変わります。</p></div>

<div class="blog-takeaways">
<h3 id="takeaways">まとめ</h3>
<ul>
  <li><strong>根本的な原因</strong>は、クラスが自分の依存に対して <code>new</code> を呼ぶことです。修正は外から渡すことです。</li>
  <li><strong>Inversion of Control</strong> は原則（依存を提供することに責任を持つのは他の誰か）。<strong>Dependency Injection</strong> はそれを実装する最も明示的でテストしやすい方法です。</li>
  <li><strong>コンストラクターインジェクション</strong>がデフォルトの選択です。依存はシグネチャに見え、構築時に必要とされ、テストで簡単に交換できます。</li>
  <li><strong>コンポジションルート</strong>は、コードベースで本物のオブジェクトが配線される唯一の場所です。他のすべてはインターフェースを通じて話します。</li>
  <li><strong>手動DI</strong>は小さなコードベースにとってよりシンプルでデバッグしやすいです。<strong>DIコンテナ</strong>はコンポジションルート自体がメンテナンスの負担になったときに見返りを得ます。</li>
  <li><strong>テストの恩恵</strong>は即座です：テストでフェイクを注入し、ネットワークとデータベースをスキップし、何千ものケースを秒単位で実行します。</li>
  <li>これはまた SOLIDの <strong>依存性逆転の原則</strong>（「D」）の実践でもあります：高レベルモジュールと低レベルモジュールは両方が抽象に依存し、互いに直接依存しません。</li>
  <li><strong>Service Locator</strong>（隠れたグローバルレジストリ）・過剰なインジェクション（10引数コンストラクター）・コンポジションルートの外でのコンテナアクセスを避けましょう。</li>
</ul>
</div>
